Mac 本地 AI Agent 週末專案｜Rapid-MLX、Gemma 4 E4B、Hermes、SearXNG 與 Firecrawl 的輕量私有代理架構

來源是 欒帥分享的 Threads 貼文。原文作者把一台 24GB Apple Silicon MacBook Air 當作本地 AI agent lab：用 Rapid-MLX 跑 MLX 量化模型，用 Hermes 作 agent 層，再用 OrbStack / Docker 部署 SearXNG 與 Firecrawl，讓 agent 有搜尋與網頁擷取能力。

這則貼文值得歸檔，因為它不是「本地跑模型」而已，而是一個很接近實務的私有代理最小架構：

```text

MacBook Air / Apple Silicon

├─ Miniforge：隔離 Python / Conda 環境

├─ Rapid-MLX：OpenAI-compatible local inference server

│ └─ mlx-community/gemma-4-E4B-it-qat-4bit：4-bit vision-text model

├─ Hermes Agent：對話、工具調度、skills、memory、gateway

└─ OrbStack / Docker

├─ SearXNG：自架 metasearch

└─ Firecrawl：網頁抓取、轉 Markdown / structured data

```

原文重點

| 層級 | 原文配置 | 判讀 |

|---|---|---|

| 硬體 | M5 MacBook Air 24GB | 以貼文描述為準；重點是 24GB unified memory 的輕量本地 agent lab |

| Web tools | Docker / OrbStack 安裝 SearXNG、Firecrawl | 讓本地 agent 可做搜尋與頁面擷取，降低對單一搜尋 API 或瀏覽器的依賴 |

| 實測 | 約 20 tokens/s | 以該硬體與模型組合為現場經驗值；不應直接外推到更大模型 |

技術架構判讀

1. Miniforge：不是模型加速器，而是環境隔離層

原文回覆中也補充：Miniforge 的角色是管理 Python 套件與獨立環境。這點很重要，因為本地 AI stack 常見衝突包括：

  • `mlx` / `mlx-vlm` / `transformers` / `torch` 版本不一致。
  • macOS 系統 Python 版本過舊或與 Homebrew Python 混用。
  • CLI 工具與 notebook / agent runtime 使用不同 Python。
  • 安裝 vision / audio extras 後依賴爆炸。

在 Apple Silicon 上，Miniforge 預設 conda-forge，通常比傳統 Anaconda / Intel wheel 混裝更穩。

2. Rapid-MLX：把 MLX 模型包成 agent 可用的 API

Rapid-MLX 的關鍵不是「又一個跑模型 CLI」，而是它把本地 MLX 模型整理成 OpenAI-compatible API。這代表上層 agent 不需要懂 MLX，只要把 base URL 指向本地 server，就能把 local model 當成一個 provider 使用。

對 agent 工作流的意義：

  • 可把低風險、隱私敏感、短上下文任務留在本機。
  • 可讓 coding / research agent 減少雲端 token 成本。
  • 可搭配工具呼叫與 OpenAI schema 類工作流。
  • 可當作「技能練習場」：先在本機模型上練 prompt、skills、tools，再把重要任務交給更強雲端模型。

需要注意：Rapid-MLX README 的 benchmark 多以特定機型、模型、batch / token 設定測試；原文的 20 tok/s 是該組合現場經驗，不等於所有 24GB Mac 都固定如此。

3. Gemma 4 E4B QAT 4-bit：適合 24GB Mac 的折衷模型

`mlx-community/gemma-4-E4B-it-qat-4bit` 的定位可理解為：

  • MLX 社群轉換模型，適合 Apple Silicon runtime。
  • 4-bit 量化，降低記憶體占用。
  • `image-text-to-text`，表示有多模態 / vision-text 方向用途。
  • E4B 等級比 20B / 70B / 100B 模型更適合 24GB unified memory 的筆電。

這類模型不應期待等同前沿雲端模型，但在以下任務很有價值：

  • 文章草稿、摘要、分類、標籤。
  • 本地資料清洗與格式轉換。
  • 低風險工具調度練習。
  • 小型 agent loop / skill 實驗。
  • 隱私敏感資料的初步處理。

為什麼要搭 SearXNG + Firecrawl？

本地模型若沒有工具，只是離線聊天機器人。加上搜尋與擷取後，才開始接近 agent。

| 工具 | 角色 | 對 agent 的價值 |

|---|---|---|

| SearXNG | 自架 metasearch | 聚合搜尋結果，降低被單一搜尋平台限制；也可避免直接把查詢丟給商業搜尋介面 |

| Firecrawl | 頁面擷取 / Markdown / structured data | 把網頁轉成 LLM 更容易消化的內容，尤其適合 JS-heavy 頁面、批次爬取與資料整理 |

| OrbStack | macOS 容器層 | 比傳統 Docker Desktop 更輕量，適合在 MacBook 上常駐開發服務 |

這個組合的本質是「把 agent 的外部感官自架化」：搜尋、擷取、整理都在本機或自控環境中完成，再交給本地 / 雲端模型推理。

可複製的最小實驗路線

若要把貼文精神轉成可落地練習，可拆成四步：

  • 建立 Python 環境：用 Miniforge / uv / venv 擇一，避免全域安裝互相污染。
  • 啟動 Rapid-MLX：先用預設小模型確認 OpenAI-compatible endpoint 可用，再換 Gemma / Qwen / vision model。
  • 接上 Hermes：把 local inference endpoint 當作模型 provider 或工具鏈的一部分，練習 skills、memory、tool calling。
  • 補 Web 工具：用 OrbStack 跑 SearXNG / Firecrawl，測試「搜尋 → 擷取 → 摘要 → 歸檔」的 agent loop。

風險與注意事項

1. 24GB MacBook Air 是學習機，不是萬能推理伺服器

24GB unified memory 可以跑不少 4-bit 小中型模型，但限制仍明顯：

  • 大模型或長上下文會吃掉記憶體與 KV cache。
  • 多模態模型通常更重。
  • 同時開 Docker、瀏覽器、IDE、Hermes、模型 server 會互搶資源。
  • Air 無風扇或散熱較弱，長時間滿載可能降頻。

因此它適合學架構、練技能、做低成本 agent lab；若要長時間高吞吐或大模型，仍需 Mac Studio / 多機 / 雲端 GPU。

2. 本地不等於完全安全

即使模型在本地，仍要注意：

  • SearXNG / Firecrawl 擷取外部網頁，內容可能有 prompt injection。
  • Docker 服務若對外開放，需設定網路與權限。
  • Agent 若能操作檔案 / shell，必須限制 destructive commands。
  • 把私密資料交給工具前，仍需確認是否會呼叫外部 API。

3. SearXNG 與 Firecrawl 授權 / 部署邊界要看清楚

SearXNG 採 AGPL-3.0；Firecrawl 是開源加 hosted service 的模式。若只是本機學習問題不大，但若放到公司或對外服務，需要檢查授權、資料保留、robots.txt、網站條款與爬取頻率。

給 欒帥的轉用建議

這則貼文可以當作「AI 學習雷達本地化實驗」的參考模板：

  • 用本地模型做初步分類：把分享連結先判斷是工具、教學、新聞、論文、工作流或生活資料。
  • 用 Firecrawl / browser fallback 抓正文：把網頁轉 Markdown，再交給 Hermes 整理。
  • 用 stronger cloud model 做最終校正：高價模型只用在來源校正、架構判讀與最後成稿。
  • 用 skills 固化流程：把「Threads → canonical source → KB article → build → deploy → verify」變成穩定 skill。
  • 用本地 lab 練 agent safety：特別是 prompt injection、工具權限、停止條件與 secret redaction。

結論

這是一個很好的「週末專案」範例：它把 Mac 本地推理、agent orchestration、自架搜尋、自架擷取與技能系統接在一起。真正的重點不是 20 tok/s，而是建立一套可控、低成本、可練習的 local-first AI agent stack。等未來換 Mac Studio 或更大記憶體機器時，這套架構可以直接往更大模型、更長上下文與更穩定的私有工作流升級。
