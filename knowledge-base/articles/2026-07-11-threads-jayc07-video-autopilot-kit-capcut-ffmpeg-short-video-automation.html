Jc Ming Threads｜video-autopilot-kit：CapCut JSON + ffmpeg 的短影音自動化開源框架

一句話摘要

這則 Threads 推薦的 `video-autopilot-kit` 是一套開源短影音/YouTube 製作框架：用 ffmpeg 與 Python 做可重現的自動剪輯，也保留 CapCut JSON 自動化路徑，適合把個人創作者的影片 SOP 轉成可維護的半自動系統。

兩條路徑：Programmatic 與 CapCut-assisted

README 把工具分成兩條同等地位的路徑：

Path 1 — Programmatic

適合 Mac、Linux、Windows，主打可重現、自動化、無 GUI 依賴：

  • `src/longform_maker/`：教學長片、premium motion、word captions、screen cleanup
  • `src/silent_vlog_maker/`：純 ffmpeg pipeline，直式 Shorts、靜音 vlog、素材整理
  • `src/capcut_helpers/` 中的 QA gates：交付前檢查頻閃、死空檔、字幕同步、b-roll 比例等
  • examples 可直接跑：`examples/01_vertical_short.py`、`examples/02_caption_broll_match.py`

這條路徑只需要 Python 3.9+ 與 ffmpeg/ffprobe，比較適合跨平台與批次化。

Path 2 — CapCut-assisted

適合真的需要 CapCut 花字、特效、雲端模板的創作者：

  • 透過 CapCut Desktop 草稿 JSON 直改來做剪輯自動化
  • 搭配 AI assistant + Computer Use 操作 CapCut 視窗套模板與匯出
  • Windows-first，且版本敏感
  • TROUBLESHOOTING 明確提醒剪映 CN 6.0+ 草稿加密不可直改；CapCut 國際版 6.x–9.x 通常仍是明文，但仍需先跑 `detect_draft_format()` 驗證

實務價值

這個 repo 的價值不在於「一鍵複製某位創作者的私有流程」，而在於把短影音製作拆成可問卷化、可配置、可測試的模組：

  • 讓創作者先填自己的品牌、niche、製作路徑、字體、BGM、b-roll 路徑
  • 用素材檔名或 keyword map 對齊 captions 與 b-roll
  • 用 ffmpeg pipeline 產出直式 Shorts
  • 用 QA gates 在交付前先抓死空檔、頻閃、螢幕錄影外框、字幕同步等常見問題
  • 有 CapCut 需求時，再把 CapCut 視為特效/模板層，而非唯一核心

對 AI 學習雷達的價值

這是一個值得追蹤的「創作者作業系統」型 repo：它把 AI 內容生成之外的後段工作，也就是剪輯、字幕、b-roll、QA、模板化設定，變成可程式化的工作流。

它特別適合用來觀察三個趨勢：

  • AI agent 不只寫文案，也開始操作與維護創作管線
  • 短影音製作從手工剪輯走向「素材庫 + 配置檔 + QA gate」
  • 開源工具如果能把私有數據抽掉，只保留方法論骨架，會更容易被不同創作者採用

採用前檢查表

導入前建議先確認：

  • 是否願意把自己的品牌、語氣、素材命名與工作流整理成 `profiles/` 與 `config.py`
  • 是否可接受 Path 1 的純程式 pipeline，而不是一開始就追求 CapCut 花字
  • 若走 Path 2，CapCut 版本是否仍可讀寫明文 JSON
  • 是否有 ffmpeg/ffprobe、Python 3.9+ 與可用字型
  • b-roll 檔名是否能描述內容，否則字幕對位會失準
  • 交付前是否願意跑 QA gates，而不是把「看起來能出片」當完成

限制與注意事項

  • Repo 是框架，不是可直接替你完成所有創作策略的 SaaS。
  • CapCut-assisted 路徑版本敏感，剪映/CapCut 更新可能破壞 JSON 自動化。
  • Mac 使用者較適合 Path 1；README 明確指出 Mac 上 CapCut GUI 自動化沒有等效機制。
  • 若要商業使用，仍需自行處理素材授權、音樂授權、字型授權與平台政策。
