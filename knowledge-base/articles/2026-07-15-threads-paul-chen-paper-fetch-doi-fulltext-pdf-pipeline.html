paper-fetch｜從 DOI 合法抓取全文 PDF 的 publisher-aware 論文流程補件

一句話摘要

`paper-fetch` 是一個「DOI in, PDF out」的論文全文抓取工具：先找開放取用版本，再試出版社官方文字探勘 API，仍失敗才走使用者自己的機構圖書館 proxy；它補上 AI 論文閱讀管線中最麻煩、也最容易被忽略的「合法拿到全文 PDF」這一段。

原文重點

陳柏威指出：要讓 AI 幫你讀論文，前提是先拿得到全文；很多人以為卡住的是 AI，其實卡住的是權限、出版社與圖書館。

這個工具的策略是給它一個 DOI，讓它按順序走一套階梯：

  • 先試 Open Access / 開放取用來源。
  • 再試出版社官方開放的文字探勘 API。
  • 若仍不行，再用你自己的圖書館帳號與機構 proxy，抓取原本就有訂閱權限的期刊。
  • 全部失敗時，印出 resolver link，讓使用者手動點選接手。

作者也明確說明：沒有 Sci-Hub，每一條路都是使用者本來就有資格走的路徑。圖書館登入那層每家醫院 / 機構都不一樣，所以公開版留成空白；repo 內有寫給 AI 看的說明，可交給 Claude / Codex 幫忙接上自己機構的圖書館。

Route ladder 的設計價值

這個工具最值得學習的是「合法路徑優先」的階梯設計：

  • Open Access：Unpaywall、Semantic Scholar openAccessPdf、PMC / Europe PMC、repository landing page 等。
  • Publisher TDM API：Elsevier、Wiley、Springer 等官方文字探勘端點。
  • Institutional proxy：使用者自己的遠端圖書館登入與 EZproxy / NetScaler 等 proxy。
  • Resolver link：讓使用者手動完成最後一步，而不是硬闖或繞過權限。

這套架構讓 AI 研究助理不只是「能讀 PDF」，而是有能力先負責尋找合法全文、處理出版社差異、保留失敗時的人類接手點。

對 AI 學習雷達的價值

這是 AI 研究工作流中很實用的補件。許多「AI 讀論文」demo 都假設使用者已經有 PDF，但真實研究流程往往卡在：

  • 只有 DOI 或 PubMed record，沒有 PDF。
  • 期刊有訂閱，但需要圖書館 proxy / 醫院 VPN / SSO。
  • 出版社頁面回傳 HTML、Cloudflare 或 paywall，不是真正 PDF。
  • 不同出版社 PDF URL 規則完全不同。

`paper-fetch` 把這些髒活拆成 route ladder，並留下可由 AI agent 協助客製化的接口。這比「把 PDF 丟給 AI」更接近完整的研究學習系統。

---

實務提醒

  • 不要把機構帳號、cookie、token 或 SSO 憑證提交到 GitHub 或聊天紀錄。
  • 每個機構圖書館 proxy 都不同，公開版不能直接保證適用。
  • 需要遵守出版社 TDM API 條款、圖書館授權與機構政策。
  • 自動抓 PDF 前應控制頻率，避免對出版社或圖書館 proxy 造成異常流量。
  • 失敗時保留 resolver link 是好設計：讓人類用合法方式接手，而不是引導到灰色來源。

可用於課程 / 工作坊的角度

這篇適合放在「AI 研究助理」或「學術工作流自動化」單元：

  • 從 DOI → PDF → appraisal → digest → learning cards 的完整管線拆解。
  • 比較 Open Access、TDM API、institutional proxy 的合法性與限制。
  • 設計一個練習：讓 Agent 閱讀 `AGENTS.md` / README，協助把 paper-fetch 接到特定學校或醫院圖書館 proxy。
  • 討論「AI 讀論文」真正困難點不只在摘要，而在資料取得、權限、品質驗證與合規。
