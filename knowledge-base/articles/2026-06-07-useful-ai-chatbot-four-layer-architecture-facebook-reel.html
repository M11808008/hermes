真正有用的 AI Chatbot 至少四層｜從 Facebook Reel 延伸出的企業聊天機器人架構檢核

來源：David Ooi 的 Facebook Reel。畫面與 Open Graph 描述可見重點為：「真正有用的 AI chatbot，至少有四层 / A useful AI chatbot needs at least 4 layers」，並標註 #AI #Chatbot #Automation。Reel 可見畫面沒有完整列出四層名稱，因此本筆記將它作為架構提醒，並用官方技術文件延伸成企業導入檢核。

一句話摘要

不要把企業 AI chatbot 做成「一個聊天框接一個模型」；真正能幫公司省錢的 chatbot，至少要同時設計介面層、知識/RAG 層、工具/工作流層，以及治理/評估層。

四層架構檢核

1. 介面與需求分流層

這一層回答：使用者從哪裡來？他要做的是問答、查資料、下指令，還是需要轉真人？

可檢核項目：

  • 是否支援員工已經在用的入口，例如 LINE、Slack、Teams、網站客服或內部系統。
  • 是否能先分類意圖：查詢、申請、報修、訂單、HR、IT、財務等。
  • 是否有清楚的 fallback：低信心、敏感問題、權限不足或多次失敗時轉人工。
  • 是否保留對話脈絡與必要的審計紀錄。

為什麼重要：如果入口與分流沒做好，模型回答再聰明，也只會變成一個難管理的萬用問答玩具。

2. 知識與 RAG 層

這一層回答：模型依據什麼資料回答？如何避免胡說？

Microsoft Azure AI Search 對 RAG 的說明是：Retrieval-Augmented Generation 用企業自己的內容來 grounding LLM 回答。企業 chatbot 若要回答 SOP、產品文件、課程資料、合約條款或內部知識，就需要可更新、可檢索、可追溯來源的知識層。

可檢核項目：

  • 文件是否有清楚來源、版本、更新時間與權限。
  • 是否有向量檢索、關鍵字檢索或 hybrid search。
  • 回答是否附引用來源，讓使用者能回到原文件確認。
  • 文件更新後是否能自動重新索引。
  • 是否能處理「找不到資料」而不是硬編答案。

3. 工具、系統整合與工作流層

這一層回答：chatbot 只會講，還是真的能幫公司做事？

OpenAI function calling、Anthropic tool use 與 MCP 都指向同一件事：模型需要能在受控條件下呼叫外部工具。真正有價值的企業 chatbot 不只是回答「請假流程是什麼」，而是能幫使用者查剩餘假期、建立請假單、送簽核、回報狀態。

可檢核項目：

  • 是否有明確工具清單與 JSON schema / API contract。
  • 是否能串 CRM、ERP、Google Workspace、Notion、Airtable、客服系統或內部資料庫。
  • 是否有權限邊界：誰可以查、誰可以改、誰需要二次確認。
  • 是否有 human-in-the-loop：付款、刪除、發送正式訊息等高風險動作需確認。
  • 是否有錯誤補償與重試策略。

4. 治理、安全、評估與營運層

這一層回答：系統上線後如何避免出事、如何知道真的省錢？

Microsoft Foundry 的 Responsible AI 文件強調生成式 AI 應納入風險管理、評估與責任式開發流程。企業 chatbot 更需要把安全與營運當作產品功能，而不是上線後才補救。

可檢核項目：

  • 是否有 PII / 個資 / 機密資料處理規則。
  • 是否有 prompt injection、越權查詢、錯誤工具呼叫的防護。
  • 是否記錄成功率、轉人工率、平均處理時間、節省工時、使用者滿意度。
  • 是否有測試集：常見問題、邊界問題、惡意輸入、權限測試。
  • 是否有回饋迴路：錯誤答案可回報、修正知識庫、更新評估集。

導入判斷表

| 階段 | 常見做法 | 風險 | 更成熟做法 |

|---|---|---|---|

| Demo | 聊天框 + LLM | 好看但不可控 | 加上明確場景與成功指標 |

| 知識問答 | 上傳 PDF 問答 | 來源不清、容易過期 | RAG + 引用 + 自動索引 |

| 自動化 | 讓模型直接操作系統 | 權限與錯誤風險高 | Tool schema + 權限 + 二次確認 |

| 上線營運 | 有問題再修 | 難以證明 ROI | 指標、評估集、回饋與治理流程 |

欒帥可延伸應用

  • 教育/課程客服：把課程 FAQ、報名資料、繳費與證書流程做成 RAG，再接 Google Sheets 或報名系統查詢狀態。
  • 企業內訓助教：讓 chatbot 回答教材問題、推薦單元、整理學員問題，並把高頻缺口回饋到課程內容。
  • AI 學習雷達：可用同樣四層思維，把「搜尋/歸檔/摘要/部署/回覆」拆成清楚工具鏈，而不是只靠模型即興整理。
  • 客服節流：先以「轉人工率下降」「平均回覆時間下降」「重複問題自動處理率」衡量是否真的省錢。

判讀限制

  • Facebook Reel 可見內容只確認「有用 chatbot 至少需要四層」與 #AI #Chatbot #Automation；四層名稱並未在可見畫面完整列出。

  • 若要作為正式企業導入方案，仍需依實際資料敏感度、系統權限、法遵要求與內部流程重新設計。
