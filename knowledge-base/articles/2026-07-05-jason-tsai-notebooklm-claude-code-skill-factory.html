傑森所長 Jason Tsai｜用 NotebookLM 打造 Claude 技能工廠：把知識資料轉成可重複執行的 skill.md

一句話摘要

這篇的核心是「知識蒸餾成技能」：先用 NotebookLM 從大量範例與來源中萃取規則，再把規則寫成 Claude Code 可讀的 `skill.md`，讓 Claude 每次處理同類任務時都有固定食譜，不必從零開始教。

Threads 貼文重點

傑森所長 Jason Tsai（@jasonxtsai）在 Threads 提到：

  • NotebookLM 幫 Claude 寫技能這件事，還沒有太多人使用。
  • 做法是把自己熟悉主題的資料丟進 NotebookLM，請它輸出 `skill.md`。
  • 再把 `skill.md` 放進 Claude Code 的技能資料夾。
  • 之後同樣類型的任務，Claude 可以每次跑出更一致的結果，不必反覆重新解釋規則、語氣與格式。
  • 作者在置頂留言補上完整教學頁：

https://evoke728.github.io/jason-pages/skill-factory

留言討論中，有人問「為什麼不直接叫 Claude Code 寫？」作者補充得很清楚：

  • 資料少的任務，直接叫 Claude Code 寫就好，NotebookLM 不必硬插一層。
  • 這套流程適合資料很多又很雜的情況。
  • NotebookLM 的角色是先整理來源、保留引用、確認規則。
  • 再交給 Claude Code 做成可重複執行的 skill。
  • 優勢不在省 token，而在降低漏資料與亂整理。

教學頁內容整理

作者的教學頁標題是「用 NotebookLM 打造你的 Claude 技能工廠｜一次設定，永久生效」。它把問題定義為：Claude 不是不強，而是每次都從零開始。

常見痛點包括：

  • 每次叫 Claude 寫 Email，都要重新解釋語氣與格式。
  • 每次叫 Claude 做 FAQ，輸出的品質忽好忽壞。
  • 明明 AI 可以省時間，但每次都要重新「調教」。

教學頁提出的解法是 Claude Code 的 Skill：放在技能資料夾裡的 `.md` 檔案，讓 Claude 在執行任務前先讀取任務標準，包括輸出格式、語氣、必守規則與範例。

5 步驟流程

Step 1：開啟 NotebookLM，上傳最佳資料來源

目標是讓 NotebookLM 有足夠的原料理解任務標準。教學頁建議依任務類型準備資料：

  • 電子郵件：過去寫過的優質 Email 範例 5–10 封。
  • 落地頁文案：覺得寫得好的落地頁截圖或文字稿。
  • FAQ：客戶常問問題清單與標準回覆。
  • 客服回覆：已脫敏的客服對話紀錄。
  • 社群貼文：高互動貼文 10–20 篇。
  • 課程講義：課程大綱、簡報、教學影片逐字稿。

關鍵原則是品質比數量重要：要上傳自己認為「這就是好輸出」的範例，若有固定格式也要一起放入。

Step 2：請 NotebookLM 撰寫 skill.md

NotebookLM 的價值在於根據使用者提供的來源整理，不是憑空發揮。它適合先從來源萃取：

  • 任務目標
  • 輸出格式
  • 語氣與風格
  • 禁止事項
  • 必備檢查清單
  • 正反範例
  • 常見錯誤

Step 3：把 skill.md 放進 Claude Code 技能資料夾

完成後，把技能文件放到 Claude Code 可讀取的技能位置。官方 Claude Code docs 對 Skills 的描述是：建立一個 `SKILL.md` 檔案放入技能資料夾，Claude 會在相關任務時把它加入工具箱，也可以用 `/skill-name` 直接呼叫。

Step 4：Claude 讀取技能，照著食譜執行

技能不是一次性 prompt，而是可重複使用的任務說明書。對於需要固定風格、固定格式、固定判斷規則的工作，它能讓輸出穩定度提高。

Step 5：每種任務建立一個技能

這個方法最適合把常做工作拆成多個可維護的技能，例如：

  • 品牌電子報
  • FAQ 回覆
  • 客服訊息
  • 社群貼文
  • 課程講義整理
  • 落地頁文案
  • 會議摘要
  • 銷售回覆

官方資料交叉確認

Brave Search 交叉查詢 Claude Skills 後，找到官方與社群資料佐證：

  • Claude Code Docs：Skills extend what Claude can do；建立 `SKILL.md` 後，Claude 會在相關時使用，也可直接用 `/skill-name` 呼叫。
  • Anthropic 官方 `anthropics/skills` GitHub repo：每個 skill 是自含資料夾，裡面有 `SKILL.md`，提供 Claude 使用的 instructions 與 metadata。
  • Anthropic engineering blog 提到，當 `SKILL.md` 太大時，可拆成多個檔案再引用，以降低 token 使用並提升結構可維護性。

這與 Jason Tsai 教學頁的方向一致：不要把所有背景都塞進一次性 prompt，而是把可重複任務整理成可維護的技能包。

實務價值

1. 把「會做」變成「可複製」

許多知識工作者不是缺 AI 工具，而是缺一套把個人經驗標準化的方法。NotebookLM 可以先把散落資料整理成規則，Skill 則把規則變成 Claude 可重複執行的操作手冊。

2. 降低反覆講 prompt 的成本

同一件事做三次以上、每次都要貼長指令或補充格式時，就值得做成 Skill。這能減少「每次重講第七次」的摩擦。

3. 適合資料多又雜的任務

作者特別提醒：資料少時直接用 Claude Code 即可。NotebookLM 這一層真正適合：

  • 來源很多
  • 需要保留引用
  • 有很多範例要抽象化
  • 規則容易漏
  • 輸出品質需要長期一致

4. 對自媒體與課程產品特別有用

這篇雖然以 Claude Skill 教學為主，但背後其實是自媒體工作流產品化：把個人語氣、內容結構、銷售話術與教學標準整理成技能，後續就能更快產出電子報、貼文、課程講義與客服回覆。

注意事項

  • 不要把所有任務都硬做成 Skill；太細會造成管理成本。
  • Skill 應保持 500–1500 字左右，太短不夠清楚，太長可能難以維護。
  • 應附上好範例與反例，比單純寫抽象規則更有效。
  • 若處理客服、報名名單或客戶資料，必須先脫敏再上傳 NotebookLM。
  • Skill 建好後要用真實任務測試，依結果持續修正。

結論

Jason Tsai 這篇的價值在於把 NotebookLM 與 Claude Code Skills 接成一條「知識蒸餾管線」：NotebookLM 負責從大量來源萃取規則，Claude Code Skill 負責讓規則在日常任務中穩定重用。對需要經常產出內容、客服、FAQ、講義或品牌文案的人來說，這是一個把個人工作方法系統化的低成本入口。
