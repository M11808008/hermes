DEIMv2-Wholebody49｜以單一物件偵測架構統一人體偵測、姿態估計與實例分割

來源：Instagram 貼文 `aicouncillor` 分享 DEIMv2-Wholebody49，並以 PINTO0309/PINTO_model_zoo 的 `488_DEIMv2-Wholebody49` README、classes 與下載資訊補充技術細節。

一句話摘要

DEIMv2-Wholebody49 把人體偵測、姿態/部位估計、實例分割與部分屬性分類壓進同一個物件偵測式輸出流程，嘗試用一次推論完成過去常由多個模型或多段 pipeline 處理的即時人體理解任務。

核心重點

  • 任務整合：Object Detection、Pose Estimation / body-part estimation、Instance Segmentation。
  • 類別數：共 49 類，涵蓋 body、adult/child、male/female、輪椅/拐杖使用者、head、face、眼鼻口耳、肩膀、手肘、手腕、手、腹部、髖、膝、踝、腳、bone 等。
  • 頭部方向：支援 front、right-front、right-side、right-back、back、left-back、left-side、left-front 等八方向分類。
  • 架構思想：作者強調沒有使用 Pose Estimation architecture，也沒有用人體關節 keypoint 資料訓練，而是以物件偵測架構估計肩、肘、膝等關節/部位。
  • 推論策略：以 one-shot inference 取得多種人體資訊，避免強行串接多個架構，以追求更高推論效能與更低訓練成本。
  • 實務取向：訓練資料加入 photometric noise、失真與作者重標註資料，目標是強化對 motion blur、Gaussian noise、contrast noise、backlighting、halation 等真實影像條件的耐受性。

為什麼值得觀察

1. 「偵測器即通用人體理解器」的路線

一般姿態估計會把 skeleton keypoints 視為獨立任務；DEIMv2-Wholebody49 的有趣之處，是把身體部位、關節位置、頭部方向與遮擋區域估計，都轉成類似物件偵測的任務。這代表在某些邊緣部署或即時視覺場景中，未必需要把 detection、pose、segmentation 拆成多模型 pipeline。

2. 對即時互動與邊緣 AI 友善

Instagram 貼文提到即時推論速度約 14ms/frame；PINTO0309 的 README 也明確把 real-world video、edge devices、single ancient GPU real-time inference 列為驗證方向。若穩定性足夠，這類模型可用在互動裝置、展演、監控輔助、機器人感知或低延遲創作工具。

3. 真實世界資料比 mAP 更重要

作者在 README 中寫下「Don't be ruled by the curse of mAP.」重點不是追逐標準資料集分數，而是透過失真、雜訊、遮擋、魚眼鏡頭、擁擠場景、縮放變異、左右交叉等情境測試實用性。這對產品化視覺模型尤其重要：實際部署時，影像品質往往比 benchmark 更混亂。

技術細節摘錄

  • 資料來源包含 COCO-Hand 與 CD-COCO，並加入作者自建/重標註資料；作者表示不計畫公開自有資料集。
  • README 提到曾在 ONNX 推論 demo 的 post-processing 發現並修正 sigmoid 執行時機 bug；模型本身不一定有問題。
  • 作者測試過 S 與 N 變體，但未達預期準確度，因此不推薦。
  • GitHub 目錄提供 `download.sh`，模型資源由 Wasabi S3 下載。
  • 相關 upstream / fork：Intellindust-AI-Lab/DEIMv2 與 PINTO0309/DEIMv2。

欒帥可延伸觀察

  • 若 AI 學習雷達未來要做即時互動 demo，可把這類 unified vision model 視為「一個低延遲感知層」，再把輸出接到生成影像、投影互動、教學互動或智慧空間應用。
  • 與 MediaPipe、YOLO-pose、SAM 類工具相比，DEIMv2-Wholebody49 的差異在於「統一輸出與重實務場景」，可作為邊緣部署候選，但仍需自己測試穩定性。
  • 這類模型也提醒：資料標註策略與真實世界失真設計，可能比單純換更大模型更影響產品可用性。

風險與限制

  • 作者明確指出訓練 pipeline 曾有 bug，最終權重並非保存於最佳 mAP 狀態。
  • 自有資料集不公開，外部難以完整重現訓練流程。
  • README 說標準資料集與作者資料集差異大，精準比較 accuracy 並不容易。
  • Instagram 展示與 demo 不能取代正式產品評測；若要採用，仍需在目標攝影機、解析度、光線與硬體上實測。
