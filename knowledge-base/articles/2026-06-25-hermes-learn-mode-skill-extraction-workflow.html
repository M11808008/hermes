Hermes /learn 模式｜從任意資料提煉可重用 Skill 的 Agent 工作流

draco 的 X 貼文指出：「Hermes 上線 /learn 模式」，可以餵給它 GitHub 倉庫、程式碼、PDF、API 文件、配置檔等資料，讓 Hermes 自行學習、提煉並封裝成合適的 Skill。

即使不同 Hermes 版本對 `/learn` 的介面名稱可能不同，這則貼文真正值得收錄的是工作流思想：**把一次性的資料閱讀，變成可重複載入、可維護、可驗證的 Skill。**

一句話摘要

Hermes Skill 的價值不是「記住一段聊天」，而是把成功的流程、工具命令、驗證步驟、常見陷阱與安全邊界封裝成下一次可直接使用的操作手冊。

可輸入的資料類型

| 類型 | 可提煉成什麼 Skill |

|---|---|

| GitHub repo | 安裝、測試、部署、PR workflow、專案慣例 |

| 程式碼 | Debug 流程、架構說明、重構規則、測試策略 |

| PDF / 文件 | SOP、課程教材、研究流程、文件處理 pipeline |

| API docs | 鑑權、endpoint、curl 範例、錯誤處理、rate limit |

| 配置檔 | 環境變數、部署設定、工具啟用方式、平台限制 |

| 過往對話 | 已驗證的工作流、反覆踩坑的修正方式 |

一個理想的 Skill 應包含什麼

| 區塊 | 內容 |

|---|---|

| Trigger | 什麼情境要載入這個 Skill |

| Preconditions | 需要哪些工具、環境變數、權限、檔案路徑 |

| Steps | 可重複執行的步驟，最好含具體命令 |

| Verification | 如何確認結果正確，例如測試、HTTP 200、檔案存在、diff 檢查 |

| Pitfalls | 常見錯誤、平台限制、反例與不要做的事 |

| Security | token、個資、私密資料、部署範圍與遮蔽規則 |

| References | 官方文件、repo、API 文件、範例輸出 |

與記憶的差異

Hermes 有 Memory，也有 Skills。兩者不應混用：

| 類型 | 適合放什麼 | 不適合放什麼 |

|---|---|---|

| Memory | 使用者偏好、長期環境事實、穩定慣例 | 一次性進度、PR 編號、短期任務結果 |

| Skill | 可重複流程、命令、驗證方式、坑點 | 使用者個資、短期狀態、未驗證猜測 |

例如：「欒帥偏好檔名用連字號」適合 Memory；「如何把社群連結歸檔到 AI 學習雷達並部署驗證」適合 Skill。

對 AI 學習雷達的應用

這則貼文很適合轉成欒帥知識庫的維護規則：

  • 收到 X / Threads / YouTube / GitHub 連結時，先擷取原始內容。
  • 查重：若已有 canonical article，更新既有文章；否則新增文章。
  • 擷取外部可驗證來源：GitHub API、官方 docs、YouTube transcript、Brave Search。
  • 寫入 Markdown 與 YAML front matter。
  • 執行 `build-knowledge-base.py` 重建索引。
  • Git commit / push。
  • 用 HTTP 200 驗證 GitHub Pages。
  • 若流程反覆使用，整理成 Skill，避免下次重新摸索。

實作注意事項

  • **不要把原始資料直接當 Skill**：Skill 應該是抽象後的流程，不是資料全文搬運。
  • **不要把 token 寫進 Skill**：API key、GitHub token、LINE token 一律只存在 `.env` 或安全憑證系統。
  • **要有驗證步驟**：沒有驗證的 Skill 只是 prompt，不是可靠流程。
  • **要保留限制**：例如 X article 可能需要登入、PDF 需 OCR、社群數據會變動。
  • **要定期維護**：工具命令、API、文件路徑變動後，Skill 需 patch，而不是另開一份重複 Skill。

判讀限制

  • 本文以 draco 公開 X 預覽內容與 Hermes Agent 文件整理；實際 `/learn` 指令、介面名稱與支援範圍需以當前 Hermes 版本與官方文件為準。
  • 社群貼文適合作為趨勢訊號；若要教學或導入，仍應用本機 Hermes CLI、skills 目錄與文件驗證。
