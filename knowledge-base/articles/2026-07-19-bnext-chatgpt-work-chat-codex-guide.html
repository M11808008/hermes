ChatGPT Work、Chat、Codex 三模式：從聊天到可交付工作的任務分流

來源：數位時代 BusinessNext 文章〈新版ChatGPT攻略｜Chat、Work、Codex 差在哪？如何選擇適合自己的模式？差異一次看〉。文章把新版 ChatGPT 拆成 Chat、Work、Codex 三種工作入口：Chat 負責想法與文字，Work 負責跨工具完成辦公流程，Codex 負責程式碼庫與軟體開發任務。本文以 OpenAI 官方公告、Help Center 與 ChatGPT Learn / Sites 文件交叉查證。

一句話摘要

這篇最值得收的是「任務分流」框架：不要再把所有需求都丟進同一個聊天框，而是先判斷要動的是文字、工作流程，還是程式碼庫；Chat 用來思考與產文，Work 用來把跨工具工作做成文件/簡報/試算表/Sites，Codex 用來讀 repo、改程式、跑測試與產生可審查 diff。

三種模式的核心差異

| 模式 | 主要處理對象 | 適合任務 | 交付物 | 使用提醒 |

|---|---|---|---|---|

| Chat | 想法、文字、知識問答 | 發想、摘要、翻譯、文案、會議記錄、資料理解 | 一段可再加工的內容 | 適合「還在想清楚」的階段；跨 App 執行與檔案交付不是它的強項。 |

| ChatGPT Work | 工作流程、檔案、外部工具 | 彙整 Drive/Gmail/Slack 等資料、建立報告、簡報、試算表、網站 | 可直接使用的成果 | 要明確寫交付格式、資料來源、限制條件，以及何時需要停下來確認。 |

| Codex | 程式碼庫、測試、PR / diff | 讀 repo、修 bug、寫功能、跑測試、code review | 程式碼變更、測試結果、可審查 diff | 仍需開發者審查後再 merge / deploy；適合有版本控制與測試的專案。 |

官方查證與修正

  • Brave Search 找到 OpenAI 官方公告〈ChatGPT is now a partner for your most ambitious work〉，摘要確認 OpenAI 將 Codex app 併入新版 ChatGPT desktop app，且 ChatGPT desktop app 中有 Chat、Work、Codex 入口。
  • OpenAI Help Center〈ChatGPT Work and Codex〉確認：Work 可研究、分析資訊並建立文件、試算表、簡報、報告與 Sites；Codex 可寫/除錯程式、執行任務、檢視變更或處理 repository。
  • OpenAI Help Center〈Moving to the new ChatGPT desktop app〉確認新版桌面 App 的遷移脈絡，以及 Chat、Work、Codex 的定位。
  • ChatGPT Learn / Sites 文件確認：Sites 讓 ChatGPT 建立、託管、調整與分享網站、web app 或遊戲；可從桌面 App 或 ChatGPT web 的 Sites 入口管理，並不等同於 Codex CLI 的獨立部署流程。
  • 方案、模型名稱與地區開放屬於快速變動資訊。數位時代文中提到 GPT-5.6、Sol/Terra/Luna、Free/Go/Plus/Pro/Business/Enterprise 等權限差異；實務採用時仍應以帳號介面、Help Center 與管理員設定為準。

欒帥可直接採用的任務判斷法

  • 這次要處理的是「想法與文字」嗎？如果是，用 Chat。
  • 這次需要跨工具讀資料、整理檔案、產出簡報/文件/試算表/網站嗎？如果是，用 Work。
  • 這次需要讀整個 repo、修改程式、跑測試、提交 diff 或 review PR 嗎？如果是，用 Codex。
  • 如果任務同時包含策略與執行，先用 Chat 釐清規格，再交給 Work 或 Codex 產出交付物。
  • 交給 Work / Codex 前，先寫清楚「資料來源、輸出格式、不能做的事、需要你批准的操作、驗收標準」。

對 AI 學習雷達的意義

這篇文章把 ChatGPT 的產品演進整理成一個很適合教學的心智模型：AI 工具正在從「單一聊天框」拆成多個工作表面。未來課程不應只教 prompt，而要教學生如何分辨任務類型、選對 agent surface、設定權限、建立驗收標準，並保留人工審查與資訊安全邊界。

實務注意事項

  • Work 的價值不是「接很多工具」，而是把工作流程跑到可交付結果；若需求模糊，它只會把模糊流程自動化。
  • 連接 Gmail、Drive、Slack、CRM 等工具前，要確認資料權限、公司政策、個資與合約資料處理規範。
  • Sites 適合快速原型、活動頁、內部展示與輕量 web app；正式對外服務仍需檢查資料外洩、品牌、法遵、維運與可攜性。
  • Codex 的最佳使用情境是有 repo、測試、版本控制與 review 流程；不要把它當成「不用看 code 的自動上線按鈕」。
  • 模型與方案權限變動很快；歸檔文章保存的是 2026-07 當下產品方向，使用前仍要看最新官方頁面。
