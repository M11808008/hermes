羅達 Rhoda｜用一句話啟動資料分析 Pipeline：Prompt、Skills 與多 Agent 的新手系統化路線

羅達 Rhoda 在 Facebook「AI 新手村」五篇連載大結局中，示範如何從一個好 prompt，逐步養成可以重複執行的資料分析系統。貼文的核心案例是股票研究日報：過去手動抓資料、清洗、算指標、畫圖、寫結論可能要一兩個小時；現在透過一句觸發指令「幫我跑 XXXX 的日報」，讓一串 skills 接力完成。

這則貼文的價值不在於單一工具教學，而在於把新手最容易卡住的「我會問 AI，但不知道怎麼變成系統」拆成五個可執行階段：Prompt → Skill → 可組合 Skill → Skill 自我迭代 → 多 Agent 並行。

一句話摘要

先不要急著做大系統；從一個好 prompt 開始，把重複工作拆成小 skill，再把 skill 養到穩定，最後才交給多 agent 並行。

貼文中的五步路線

| 階段 | 貼文說法 | 實務轉譯 |

|---|---|---|

| 1. Prompt | 每一步都從一個 prompt 開始 | 先把「怎麼抓資料」「怎麼清洗」「怎麼算」用自然語言講清楚 |

| 2. 三層 Skill | 描述、指令、工具 | 一個 skill 至少要有何時啟用、執行步驟、欄位規則/腳本/模板 |

| 3. 可組合 | 不做萬能巨獸 | 把流程拆成抓資料、清洗、算指標、畫圖、寫結論等小模組 |

| 4. 自我進化 | 用完寫回 skill | 遇到除權息、單位、空值等錯誤，要回寫規則而不是每次重新提醒 |

| 5. 多 Agent | 一句話並行 | 在技能穩定後，再讓不同 agent 平行處理資料、計算、視覺化與撰稿 |

貼文給出的 Pipeline 範本

```text

一句話（觸發）

→ 抓資料 skill

→ 清洗 skill

→ 算指標 skill

→ 畫圖 skill

→ 寫結論 skill

→ 一份日報

```

貼文提供的新手 prompt：

```text

我每天／每週都要做『XX 分析』。請幫我：

（1）把它拆成 3～5 個各司其職的小 skill；

（2）每個 skill 標出描述、指令、需要的工具；

（3）設計一句『觸發指令』，讓我打一句話就能依序跑完整條流程。

```

對欒帥的實作版本

如果要把這套思路用在 AI 學習雷達、課程設計或商業分析，不建議直接建立「超大型萬能 AI 助理」。更穩的作法是先選一個高頻任務，拆成明確 pipeline。

範例 A：AI 學習雷達文章歸檔

```text

觸發：歸檔這個 AI 連結

→ source-extraction skill：擷取原文、作者、日期、可見圖片與外部連結

→ authority-check skill：找官方文件、GitHub、論文或產品頁補強

→ practical-translation skill：轉成欒帥可用的教學、商業或工作流觀點

→ knowledge-base-writing skill：產出 Markdown 與 frontmatter

→ publish-verify skill：重建索引、部署、驗證公開 URL

```

這其實已經接近目前愛馬仕小小斌的工作流：不是只摘要，而是把連結轉成可累積的知識資產。

範例 B：課程設計 Pipeline

```text

觸發：幫我把這個 AI 主題做成 90 分鐘課

→ audience-scan skill：確認學員背景、先備知識、痛點

→ concept-map skill：拆核心概念與誤區

→ activity-design skill：設計練習、討論、產出物

→ slide-outline skill：產生投影片架構

→ assessment skill：設計檢核題與課後任務

```

範例 C：股票或產業日報 Pipeline

```text

觸發：幫我跑某公司／某產業日報

→ data-source skill：抓行情、新聞、財報、公告

→ cleaning skill：處理缺值、單位、時間區間、除權息

→ indicator skill：計算關鍵指標

→ visualization skill：畫圖與標註異常

→ narrative skill：整理成管理者可讀結論

```

Skill 的三層設計檢查表

| 層級 | 必填內容 | 常見錯誤 |

|---|---|---|

| 描述 | 這個 skill 何時該被啟用、何時不該用 | 只寫「做資料分析」，太模糊 |

| 指令 | 明確步驟、輸入、輸出、錯誤處理 | 只寫目標，沒有流程 |

| 工具 | 欄位規則、公式、腳本、模板、範例 | 沒有可重用素材，每次都靠模型猜 |

Claude Code 官方 skills 文件也採用類似思路：Skill 不只是 Markdown 說明，還可以包含模板、腳本、範例輸出等輔助檔案。OpenAI Codex Skills 文件也強調，skills 讓 agent 能在需要時選用特定任務知識，而不是把所有細節都塞進主提示詞。

多 Agent 要晚一點再上

貼文提到 Anthropic 內部多 agent 研究系統在內部 research eval 中，比單一 Claude Opus 4 agent 高出 90.2%。這個數字來自 Anthropic 工程文章，脈絡是複雜研究任務下的 lead agent + subagents 架構，不代表所有任務都會自動提升 90%。

對欒帥來說，比較務實的判斷是：

  • 任務能拆成彼此相對獨立的子問題，才值得並行。
  • 每個子任務的輸入、輸出、完成標準要清楚。
  • 合併結果需要另一個「總編 / reviewer」角色檢查矛盾與重複。
  • 如果 skill 還沒穩，多 agent 只會把錯誤放大。

從新手到系統的落地路線

  • 選一個每週至少做一次的重複任務。
  • 寫出第一版「一句話觸發指令」。
  • 把任務拆成 3–5 個小 skill。
  • 每個 skill 補上三層：描述、指令、工具。
  • 跑一次完整流程，記錄失敗點。
  • 把失敗規則寫回對應 skill。
  • 重複 3–5 次後，才考慮自動化或多 agent 並行。

不建議一開始做的事

  • 不要先做「萬能助理」。
  • 不要把所有規則塞進單一超長 prompt。
  • 不要在流程尚未穩定前就上排程自動化。
  • 不要讓多 agent 同時改同一份資料或同一個檔案。
  • 不要只保存漂亮輸出；要保存錯誤修正規則。

對 AI 學習雷達的關聯

這篇可以與既有的 Claude Skills、Codex Skills、社群監測排程與工作流產品化文章互相連結。它補上的是「新手怎麼從第一個 prompt 走到可複用系統」的教學階梯。

若未來要開 AI 實作課，這篇可作為一堂 60–90 分鐘工作坊：

  • 學員選自己的重複工作。
  • 寫一句觸發指令。
  • 拆 3–5 個 skill。
  • 每個 skill 寫描述、指令、工具。
  • 讓 AI 產出第一版 pipeline。
  • 討論如何把錯誤寫回 skill。
