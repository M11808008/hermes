Prompt Case｜Hermes Browser Extension：把 Chrome 變成本機 Hermes Agent 終端

一句話摘要

Hermes Browser Extension 的重點不是「瀏覽器裡多一個聊天機器人」，而是把 Chrome / Edge / Chromium 的網頁上下文接到真正的 Hermes Agent runtime，讓瀏覽器成為本機 agent 工作流的入口。

Threads 貼文重點

Prompt Case 以「當 Chrome 遇上 Hermes Agent，我的瀏覽器直接進化成本機 AI 終端」為主軸，描述自己在 Perplexity / Comet 使用體驗下降後回到 Chrome，發現 `Hermes Agent Chrome Extension` 推出，安裝綁定後認為它與市面上常見 AI 瀏覽器不同。

貼文的核心判斷是：一般 AI 瀏覽器多半是在網頁上加一層摘要、問答或側欄；Hermes Browser Extension 則是把全能本機 Hermes Agent 塞進瀏覽器，讓它除了看網頁、開分頁與一般瀏覽任務外，還能串接 Hermes 已有的本機能力：讀寫檔案、跑終端、寫程式、跑測試、使用記憶系統、skills、自動化任務等。

作者在續串補充 GitHub 連結：`https://github.com/abundantbeing/hermes-browser-extension`，並提醒安裝不算複雜，可以把專案連結交給 Hermes Agent，請它協助 clone、安裝、設定 API 並寫一份簡短教學；但也必須注意安全，包含提示詞注入、權限設定與隔離保護。

另一則續串提到，如果想安裝 Hermes Agent，本身已有電腦端應用程式與安裝文章可參考。

主要能力整理

根據 README 與 manifest，這個 extension 的能力可拆成幾層：

  • 瀏覽器上下文層

- 擷取活動分頁標題、URL、選取文字、可讀頁面文字、metadata、headings、forms、links、buttons。

- 可選擇 Chat only、跟隨 active tab、固定特定 tab 或選擇哪些開啟分頁進入 prompt。

  • Hermes runtime 層

- 不是在 extension 內跑一個小模型，而是把請求送到 Hermes Gateway / API server。

- 因此可沿用 Hermes 的模型 provider、sessions、skills、memory、MCP、工具與本機工作流。

  • 工作流層

- 提供 `/summarize`、`/explain`、`/rewrite`、`/tabs`、`/action-items` 等 quick commands。

- 顯示「What Hermes saw」收據，讓使用者知道本回合帶入了哪些上下文。

- 串流 Hermes 回覆，並在必要時 fallback 到非串流 chat。

  • 安全 / 相容層

- 頁面內容會以 untrusted context 包裝，再送進 Hermes。

- 目前 v0.1.9 沒有 browser control / Runs UI / debugger / nativeMessaging。

- 有相容性面板與 diagnostics，讓較舊 gateway 以 fallback / manual 模式降級，而不是直接壞掉。

安裝與連線方式

README 的 quick start 是：

  • `git clone https://github.com/abundantbeing/hermes-browser-extension.git`
  • `cd hermes-browser-extension`
  • `npm install`
  • `npm run build`
  • 在 Chrome / Edge 開 `chrome://extensions` 或 `edge://extensions`。
  • 啟用 Developer mode。
  • Load unpacked，選擇產生的 `dist/` 資料夾，不是 repo root，也不是 `extension/`。
  • 開啟 side panel，連到 Hermes。

本機 Hermes API server 建議設定在 `~/.hermes/.env`：

  • `API_SERVER_ENABLED=true`
  • `API_SERVER_HOST=127.0.0.1`
  • `API_SERVER_PORT=8642`
  • `API_SERVER_KEY=<your-api-server-key>`
  • `API_SERVER_CORS_ORIGINS=chrome-extension://<your-extension-id>`

README 也強調：遠端 API server 可以使用 LAN、Tailscale、VPN 或可信任 HTTPS reverse proxy，但不要把 Hermes API server 裸露到公網，因為它能存取真正的 Hermes runtime 與工具。

與一般 AI 瀏覽器的差異

Prompt Case 的觀察可以整理成一個判斷框架：

  • 一般 AI 瀏覽器：通常以「看目前網頁、摘要、問答、改寫」為主。
  • Hermes Browser Extension：把網頁上下文接入本機 agent runtime，讓後續動作可以落到檔案、終端、程式碼、測試、skills、記憶、cron、MCP 與其他工具。

因此它真正打破的是「網頁資料」與「本機工作流」之間的牆。看到一篇文件、issue、PR、dashboard、競品頁或課程頁時，使用者不必手動複製貼上到終端 agent；extension 可把可讀上下文送進 Hermes，再由 Hermes 做後續處理。

安全判讀與風險

這類工具的強大之處也是風險來源。建議使用時至少注意：

  • 不要截圖、列印、commit API key；安裝指令也應明確要求 agent 不得揭露 key。
  • CORS origins 應只允許自己的 extension ID，不要放寬成萬用來源。
  • 遠端 API server 只放在可信任 LAN / Tailscale / VPN / HTTPS reverse proxy 後面，不要裸奔公網。
  • 把網頁內容視為不可信輸入，尤其是文件頁、論壇、issue、email、CRM、金融或付款相關頁面可能含提示詞注入。
  • 目前 public alpha 且未上架 Chrome Web Store，應以開源 repo / load unpacked 的實驗工具看待，定期檢查更新與 permissions。
  • 涉及本機檔案、終端、程式碼修改與測試時，仍應保留權限邊界、review 與 rollback 機制。

對 AI 學習雷達的價值

這則貼文值得高優先度保存，因為它代表 Hermes 生態的一個重要方向：把 agent 從聊天平台與 CLI 擴展到瀏覽器現場。對欒帥的工作流來說，可延伸成幾個使用情境：

  • 看技術文章時，一鍵讓 Hermes 歸檔、驗證外部連結、產生知識庫筆記。
  • 看 GitHub repo 時，讓 Hermes 讀 README、檢查 install、建立本機 spike 或測試。
  • 看課程 / 活動 / landing page 時，直接整理成課程設計參考或市場案例。
  • 看錯誤訊息、文件或 SaaS dashboard 時，讓 Hermes 把網頁資訊和本機終端除錯串起來。
  • 在 Chrome 中把 Claude / Codex / Hermes 的分工變成一條可操作的 agentic workflow。

建議後續實作

若要實際採用，建議先走保守模式：

  • 只連本機 `127.0.0.1:8642`，不要先開遠端。
  • 產生獨立 API key，並限制 CORS 到 extension ID。
  • 先在低風險網頁測試 `/summarize`、`/explain`、`/action-items`。
  • 確認「What Hermes saw」內容合理，不把敏感資料送入 session。
  • 再逐步測試 GitHub repo、文件、知識庫歸檔等高價值工作流。
  • 若要接遠端 Hermes，優先用 Tailscale / VPN，不要裸露 API。
