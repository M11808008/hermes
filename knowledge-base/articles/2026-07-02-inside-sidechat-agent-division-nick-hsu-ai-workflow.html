核心觀點一：Agent 不應照傳統組織圖切

Nick 回答「Agent 的功能要怎麼切」時，反對把公司組織圖直接複製成一堆 agent。他認為傳統組織架構是人類 bandwidth 有限下的產物：一個主管無法同時聽 60 個人講話，所以需要中層主管幫忙拆任務、統整回報。

但如果中層與執行單元都能由 AI 協助，問題就變成：為什麼還要保留原本的三部六省？

較合理的切法是：

  • 按任務切：架構設計、實作、測試、review、部署。
  • 按領域切：UI、邏輯、資料、產品、設計、成長。
  • 按上下文切：不同任務需要不同初始設定、知識、步驟與記憶。
  • 按驗證責任切：不要讓同一個 agent 自己寫完又自己審，容易自我認可。

這與欒帥 2026 GAI 年會簡報中提到的「企業大腦不是資料倉庫，而是判斷系統」高度呼應：Agent 分工不是把人形職位換成 AI，而是重設判斷、執行與驗證的流動方式。

核心觀點二：真正會用 AI 的人，差距在 context 與驗證

影片多次強調，多數人覺得 AI 不好用，往往不是模型能力不足，而是少了兩件事：

  • 沒有把問題、場景、目標與限制講清楚。
  • 沒有設計驗證 AI 成果是否正確的方式。

Nick 在面試工程師時，不再只是考演算法或傳統 live coding，而是要求候選人做 live building：

  • 先讓候選人向需求方提問。
  • 看候選人是否能把高併發、流量、交易、matching 等 spec 問清楚。
  • 看候選人能否請 AI 反問「我還漏問了什麼」。
  • 看候選人如何把需求條列化給 AI。
  • 看候選人如何驗證 AI 做出來的東西是真的完成，而不是只是宣稱完成。
  • 優秀候選人會另開 agent 產生測試方案，再用測試去 review 實作。

這裡的重點不是 prompt 技巧，而是標準工程流程：需求澄清、設計、實作、測試、review、部署。AI 只是把這套流程加速並可並行化。

核心觀點三：Claude Code 之後，工程流程變成多 agent 閉環

Nick 提到，在交易所這種人才密度與工具預算都高的環境中，Claude Code 類工具已經不只是輔助 coding：

  • 工程師會派出 10、20、30 個 agent 跑任務。
  • Agent 會掃整個 codebase 與 system。
  • 發現指標或系統問題後，自行讀 code、改 code、開 PR。
  • 人類隔天早上看 PR，決定接受、要求修改或繼續對話。
  • 會議後自動產生紀錄與 ticket，甚至部署給 agent 開始做。
  • Agent 可自行找相關文件、理解上下文、跑一整晚。

這代表 AI Coding 的重點正在從「寫幾行 code」走向「自動化工程作業系統」。人類的角色變成定義問題、批准方案、看成果、驗證風險與做產品判斷。

核心觀點四：生產力提升不會自動變成營收提升

影片中提到一個重要問題：為什麼 AI 讓 productivity 提升 5 到 10 倍，但 revenue 沒有跟著提升 5 到 10 倍？

Nick 與主持人用蒸汽機到發電機的比喻回答：舊工廠是為蒸汽機設計的，把發電機塞進舊工廠，生產力不會立刻十倍。真正的效率提升，必須重構工廠配置、流程與組織想像。

套到 AI：

  • 把 AI 插進舊部門、舊 KPI、舊審批流程，只會得到局部效率。
  • 若 workflow、資料流、決策流、驗證流不改，AI 只是更快地產生舊流程裡的文件與摩擦。
  • 真正的變化發生在公司重新設計任務、角色與系統邊界之後。

這點可直接用於企業 AI 導入課程：不要問「哪些職位要變成 agent」，而要問「哪些流程可以改成可驗證、可循環、可交付的 agentic workflow」。

核心觀點五：未來組織可能更接近 Builder 與 Seller

影片引用 Cloudflare 老闆對組織的觀點：未來可能只剩 builder 與 seller 兩大角色。雖然這個說法仍有爭議，但它指出一個趨勢：中間層、行政層、彙報層與度量層會被 AI 壓縮。

過去需要中層，是因為人類無法管理大量資訊流。AI 讓資訊蒐集、整理、彙報、初步判斷、文件生成與流程追蹤的成本下降，中層若只負責轉譯、彙整與追進度，就會面臨壓力。

但這不代表人類只剩工程與銷售兩種工作，而是代表高價值工作會往兩端移動：

  • Builder：能把系統、產品、流程、內容或服務做出來的人。
  • Seller：能理解人、建立信任、成交、維繫社群與市場的人。
  • 介於兩者之間的協調、報告、追進度、排程與行政，會被工作流自動化大量吃掉。

Nick 的實體產業 AI 實驗：衝浪學校案例

影片後段最值得保留的案例，是 Nick 把 AI 導入衝浪學校與實體服務業：

  • 用一晚做出網站並串金流，取代人工轉帳後五碼與 LINE 對帳。
  • 支援分期付款、降低收款與行政摩擦。
  • 不一定做傳統後台，而是用 skill / API / database 直接讓人與系統對話。
  • 把過去 200 多支影片抓下來，做截圖、transcript 與資料庫。
  • 用 Notion / database 支援選題、排程與內容雛形產生。
  • AI 協助教練訓練：把過去案例、回饋、教學話術放進 AI，讓教練每天對話、寫作業，由 AI 批改。

這個案例很重要，因為它說明 AI 的機會不只在軟體公司，而是在實體產業中降低高摩擦流程：金流、報名、對帳、客服、行政、內容、訓練、回饋與社群經營。

對欒帥的應用建議

這集可作為「企業 AI 導入」與「AI Agent 工作流」課程中的核心案例，尤其適合接在 2026 GAI 年會簡報重點之後：

  • 用 Nick 的觀點補強「不是把每個職位換成 agent，而是重設判斷系統」。
  • 把 agent 分工教成任務／領域／上下文／驗證，而不是組織圖複製。
  • 將 live coding 改成 live building，訓練學員用 AI 做需求澄清與驗證。
  • 引入多 agent 流程：planner、builder、reviewer、tester、publisher。
  • 用衝浪學校案例說明實體產業如何靠 AI 降低高摩擦流程。
  • 強調 AI 導入不是 prompt 課，而是工作流、資料、記憶、驗證與交付設計。

可轉成課程練習

練習一：Agent 分工重設

請學員拿一個傳統流程，例如報名、客服、內容製作、銷售追蹤、課程助教，先畫出原始人工作業，再改成：

  • Planner agent
  • Builder / Executor agent
  • Reviewer agent
  • Tester / QA agent
  • Human approver

要求每個 agent 都寫清楚輸入、輸出、需要的 context、禁止事項與驗收標準。

練習二：Live Building 面試法

給學員一個模糊需求，例如「我要做一個高流量投票網站」或「我要做一個課程報名系統」。

評分不看第一個 prompt，而看：

  • 是否先問清楚需求。
  • 是否讓 AI 協助找漏問問題。
  • 是否整理 spec。
  • 是否規劃測試與驗證。
  • 是否能用第二個 agent review 第一個 agent 的成果。

練習三：實體產業高摩擦流程盤點

讓學員列出一個實體產業的高摩擦流程，例如：

  • 轉帳後五碼對帳。
  • LINE 報名與人工確認。
  • 學員回饋整理。
  • 教練訓練與作業批改。
  • 社群內容選題與排程。

再設計一個 AI workflow，目標不是炫技，而是降低行政摩擦、保留人類服務與社群價值。

風險與限制

  • 影片談的是前沿實務與個人觀察，不能直接視為所有企業都能立即複製的標準答案。
  • 多 agent 工作流需要權限、成本、資料安全、測試與人類審核設計，不能讓 agent 無限制改 code 或部署。
  • 「不到 1% 的人做得出成果」是強烈觀察語句，適合當提醒，不宜當統計事實引用。
  • Builder / Seller 的二分法有啟發性，但組織設計仍需依產業、法規、風險與品牌需求調整。
  • Crypto 與投資相關內容應視為觀點討論，不是投資建議。
