GitHub 奶爸級教學｜AI 編程者必備的註冊、2FA、Git 與協作工作流

老王霸 AI Lab 的 X Article 標題是：「一文超詳細講透 GitHub：註冊，雙重認證，搜尋，建倉庫，issue，本地 git，部署，fork，star，pull，push」。貼文定位很清楚：如果你是 AI 編程愛好者，GitHub 是必修基礎設施。

X Article 需要登入才能完整閱讀，本文依公開預覽與 GitHub 官方文件補強，整理成 AI 編程工作流角度的 GitHub 入門地圖。

一句話摘要

對 AI 編程者來說，GitHub 不只是放程式碼的網站，而是讓 Codex、Claude Code、Cursor、Hermes 這類 agent 能安全修改、審查、部署與回滾成果的工程協作平台。

為什麼 AI 編程者必須懂 GitHub

| 能力 | 對 AI 工作流的意義 |

|---|---|

| Repository | 讓 agent 有明確專案邊界與檔案上下文 |

| Git commit | 每一步改動可追蹤、可回滾 |

| Branch | 讓 AI 修改在隔離分支發生，不污染主線 |

| Pull Request | 將 AI 改動整理成可審查的 diff |

| Issues | 把需求、bug、任務拆成 agent 可執行的單元 |

| Actions / Pages | 自動測試、部署網站與產物 |

| Fork / Star | 學習開源專案與保存資源 |

| 2FA / Token | 保護帳號與自動化權限 |

建議學習順序

1. 註冊與雙重認證

先完成 GitHub 帳號、Email 驗證與 2FA。AI 編程者尤其要注意：

  • 不要把 GitHub token 貼進聊天視窗。
  • PAT 應只給必要 scope。
  • 若用 Codex / Claude Code / Hermes 操作 repo，先確認操作範圍與分支。
  • 重要 repo 建議開啟 branch protection。

2. 搜尋與看懂開源專案

GitHub 搜尋不是只找 repo 名稱，也能找：

  • README 關鍵字
  • specific file path
  • language / stars / updated filters
  • issues / pull requests
  • topics

AI 學習雷達在歸檔 GitHub 專案時，也應同步查看 README、license、stars、forks、issues、last updated，而不是只看社群貼文說法。

3. 建立 repository

AI 編程專案建 repo 時，最少應有：

  • README.md：專案目的、安裝、啟動、測試
  • .gitignore：排除 node_modules、venv、.env、build cache
  • LICENSE：若要開源需明確授權
  • AGENTS.md / CLAUDE.md：給 agent 的專案規則
  • tests / CI：讓 agent 的改動可以驗證

4. Issues：把需求拆成任務

Issue 是把模糊需求變成可交付任務的地方。好的 issue 應包含：

  • 背景與目標
  • 範圍：哪些檔案可動、哪些不能動
  • 驗收條件
  • 測試方式
  • 截圖或重現步驟
  • 風險與限制

這也是 AI agent 最容易成功的任務格式。

5. 本機 Git：clone、branch、commit、pull、push

AI 編程者至少要掌握：

```bash

git clone <repo-url>

git checkout -b feature/name

git status

git diff

git add .

git commit -m "feat: concise message"

git pull --rebase

git push -u origin feature/name

```

重點不是背命令，而是知道每一步代表什麼狀態：工作區、暫存區、commit history、remote branch。

6. Fork / Pull Request

Fork 適合學習、貢獻或改造別人的開源專案。PR 則是把改動送回原 repo 的審查單位。

AI agent 產生的 PR 建議附上：

  • 改了什麼
  • 為什麼改
  • 如何測試
  • 截圖或輸出
  • 已知限制
  • 是否影響相容性

7. 部署：GitHub Pages / Actions

對知識庫、課程頁、靜態網站而言，GitHub Pages 是低摩擦部署方式。AI 學習雷達目前也用 GitHub Pages 公開部署。

基本流程是：

  • 產生靜態 HTML。
  • commit 到 repo。
  • push 到 main 或指定分支。
  • 等 Pages 發布。
  • 用 HTTP 200 驗證頁面。

對欒帥的實作建議

| 需求 | 建議 |

|---|---|

| 教 AI 初學者 | 先教 GitHub repository / commit / PR，再教 Codex 或 Claude Code |

| 維護知識庫 | 所有公開文章都經 Git commit，可查歷史 |

| 保護私密資料 | 報名名單、個資、token 不進公開 repo |

| 做課程網站 | Pages + 靜態 HTML，低成本可持續 |

| 讓 agent 協作 | 每個 repo 放 AGENTS.md，定義測試、部署、命名與禁區 |

常見錯誤

  • 把 `.env`、API key、token commit 到 GitHub。
  • 讓 AI 直接在 main 分支大改，沒有 branch / PR。
  • 只看 AI 說「完成」，不看 git diff 與測試輸出。
  • 沒有 README，導致 agent 每次都要重新理解專案。
  • 沒有 issue / task 拆解，一次丟過大的需求給 agent。
  • Pages 部署後沒做 200 OK 驗證。
