Antigravity Workflows 社群技巧｜用 Markdown 流程、Slash 指令與多代理並行壓縮重複任務

來源是 欒帥分享的 Threads 貼文。Rock 描述一個 Antigravity 用法：把開發流程包成 `.antigravity/workflows/*.md`，再用 slash command 觸發，讓多個 AI agent 並行執行部署、PR、發信等工作。

這則貼文值得收錄，但要以「社群技巧 / 待實測」角度保存：概念很有價值，與 Claude Code slash commands、Hermes skills、Codex / Cursor project rules、Git worktree 多代理協作高度相通；但貼文中的特定數字與目錄語法，仍需以 Antigravity 官方文件與本機實測確認。

一句話摘要

重點不是「Antigravity 某個神秘功能」，而是 agentic development 的共通趨勢：把 30 步人工流程寫成可版本控管的 Markdown workflow，透過 slash command 觸發，再交給多個 agent / sub-agent 並行處理，最後由人類審核與合併。

可抽象成通用 Agent Workflow Pattern

這個模式不只適用於 Antigravity，也適用於 Claude Code、Codex、Cursor、Hermes 或其他 coding agent：

```text

workflow markdown

→ slash command / skill trigger

→ agent decomposes tasks

→ parallel worktrees / sub-agents

→ tests / checks / logs

→ human review

→ merge / deploy / notify

```

一個較安全的 deploy workflow 寫法

若把貼文精神轉成可落地模板，建議不要讓 agent 直接「自動合併、部署、發信」，而是分成安全階段：

```markdown

deploy

Goal: prepare a deployment candidate, but do not push to production without human approval.

Steps:

  • Read project README, deployment docs, and recent git diff.
  • Run lint and tests. If failing, stop and report root cause.
  • Build the project locally.
  • Create a release summary: changed files, risk areas, rollback plan.
  • Open or update PR with checklist.
  • Draft notification message, but do not send.
  • Ask for human approval before merge, deploy, or external messages.

```

這樣能保留 workflow 的效率，又避免 agent 在沒有批准時做外部副作用。

為什麼它重要

1. 從 Prompt Engineering 走向 Workflow Engineering

過去的問題是「怎麼問 AI」。現在更重要的是「怎麼把可重複流程寫成 AI 能執行、能驗證、能回報的工作規格」。Markdown workflow、slash command、skills、project rules，本質上都是把人類 SOP 變成 agent 可操作的介面。

2. 多代理的真正價值在隔離與驗證，不是炫技

同時開 5 個 AI 並行，只有在任務可切分、可驗證、可合併時才有價值。例如：

  • 一個 agent 修測試。
  • 一個 agent 更新文件。
  • 一個 agent 準備 PR 描述。
  • 一個 agent 檢查部署風險。
  • 一個 agent 做回歸測試或安全掃描。

若沒有 worktree / branch 隔離、測試與 merge policy，多代理只會放大混亂。

3. 對 欒帥 的知識庫管線很有參考價值

AI 學習雷達目前已經很接近 workflow 化：

  • 讀 Threads / YouTube / GitHub / 新聞來源。
  • 找 canonical source。
  • 校正官方資料。
  • 寫 Markdown front matter 與文章。
  • build 靜態網站。
  • commit / push。
  • 驗證 GitHub Pages。

這其實就能包成一個「archive-link」workflow。差別只在於是否把每一步拆成 sub-agent 並行，例如：一個查官方來源、一個讀既有知識庫避免重複、一個寫草稿、一個做驗證。

風險與導入注意

  • **不要自動執行高風險副作用**：merge、deploy、寄信、刪資料、付款、公開發布都應保留 human approval。
  • **workflow 要有停止條件**：例如最多 3 輪修復、測試仍失敗就停止回報。
  • **要有驗證訊號**：tests、build、HTTP status、diff review、preview URL，比「agent 說完成」可靠。
  • **多代理要用 worktree / branch 隔離**：否則容易互相覆蓋。
  • **社群數字要實測**：貼文提到的效率提升可作靈感，不應直接用於提案承諾。

給 欒帥的轉用建議

可以把這則貼文視為「AI 工作流技能化」案例，未來若要做課程或內部 SOP，可設計成：

  • 先讓學員把一個重複任務寫成 Markdown SOP。
  • 加上 done criteria、風險區、不得執行事項。
  • 用 Claude Code / Codex / Hermes skills 或 Antigravity workflows 其中一種工具觸發。
  • 用測試或 checklist 驗證。
  • 將成功 SOP 版本化，變成 team skill。
