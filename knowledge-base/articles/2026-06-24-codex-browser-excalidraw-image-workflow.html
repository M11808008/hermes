Codex 內建瀏覽器 × Excalidraw｜把手繪草稿變成可批量化的生圖工作流

來源是 ekcheungAI 在 Threads 分享的 Codex 玩法：用 Codex 內建瀏覽器直接開 Excalidraw，手畫一個草稿後，讓 Codex 依草稿生成圖片，並可結合 Codex 的程式能力做批量化、統一風格與自動後製。留言中也討論了 token / 額度消耗與高品質生成會燒額度的問題。

一句話摘要

這則 Threads 值得收錄的重點不是「Codex 也能生圖」而已，而是它把 Codex 從 coding agent 推向「可操作瀏覽器與視覺草稿的創意工作台」：人用 Excalidraw 快速表達構圖，Codex 負責理解、生成、批量化與後製，形成比單次生圖更可控的內容生產流程。

工作流拆解

1. 人先畫「低成本視覺意圖」

Excalidraw 的價值是快：不需要精準設計，也不需要先開 Figma。只要把畫面關係畫清楚，例如：

  • 主角位置；
  • 背景與前景層次；
  • 物件比例；
  • 鏡頭方向；
  • 文案區塊；
  • 留白與焦點。

這比純文字 prompt 更容易傳達構圖，也比精修設計稿更低成本。

2. Codex 用內建瀏覽器讀取草稿

OpenAI Codex app 官方文件有獨立的 In-app browser 與 Computer Use 文件。官方說明 Codex 可在 app 內使用瀏覽器處理本地開發伺服器、file-backed pages，也可透過 browser / computer-use 操作與檢查畫面。這使它不只適合寫程式，也可處理「畫面 → 程式 / 資產 / 報告」的任務。

在這個案例中，Excalidraw 是人機共同的中介：人畫草圖，Codex 讀畫面，接著產生圖片 prompt、HTML / SVG / PNG 或後製腳本。

3. 生成圖片不是終點，批量化才是重點

貼文最有價值的一句是「再配上 Codex 的程式能力，批量化、統一風格、自動後製都行」。這代表流程可以升級成：

  • 用 Excalidraw 畫 1 張構圖模板。
  • Codex 讀取模板，拆成構圖描述與設計規則。
  • 生成多組 prompt / 圖片版本。
  • 用程式統一尺寸、邊框、色調、浮水印、檔名與輸出格式。
  • 自動產出社群平台需要的比例，例如 1:1、4:5、9:16、16:9。
  • 建立人工審核清單，再批量輸出。

這是「創意 prompt」到「內容生產線」的差別。

和 Excalidraw 官方能力的關係

Excalidraw 本身不只是白板。官方 developer docs 提供匯出工具，例如 exportToBlob、exportToSvg、exportToClipboard，可輸出 PNG / SVG / clipboard 等格式。若要把這套流程做穩，可以讓 Codex 或腳本讀 `.excalidraw` JSON，直接轉成 SVG / PNG，再交給影像模型或後製工具。

對 Agent 工作流來說，推薦把 Excalidraw 當作「可機讀草稿」而不是單純截圖：

  • `.excalidraw`：保存可編輯結構。
  • SVG：保留向量構圖與文字。
  • PNG：提供給需要影像輸入的模型。
  • JSON：讓 Codex / Python 腳本解析物件位置、大小與標籤。

給欒帥的實作價值

1. 課程圖卡與知識庫封面

可以先用 Excalidraw 畫出課程模組、Agent 架構、AI 工具流程，再讓 Codex 轉成統一視覺風格的圖卡。這比每張圖重新 prompt 更穩定。

2. AI 學習雷達文章配圖

對知識庫文章，可把「文章重點 → Excalidraw 草圖 → Codex 生成多版封面 / 示意圖」做成流程。草圖保存後，下次同類文章能沿用構圖語言。

3. 批量社群內容

若要把一篇文章拆成多張 Threads / Facebook / LINE 圖卡，可以要求 Codex 讀同一個草稿，批量產出不同標題、比例與 CTA，但保持統一視覺系統。

建議 Prompt 骨架

你是視覺生產 Agent。請先閱讀目前瀏覽器中的 Excalidraw 草稿，理解構圖與標籤，不要直接自由發揮。

請輸出：

  • 草稿元素清單：主體、背景、文字區、動線、視覺焦點。
  • 生成圖片 prompt：保留構圖，但補足風格、材質、光線與色彩。
  • 版型規格：輸出比例、尺寸、留白、字級與安全邊界。
  • 批量變體：至少 3 個風格方向，但構圖一致。
  • 後製腳本需求：重新命名、裁切、壓縮、加 logo、輸出 WebP / PNG。
  • 人工審核清單：是否偏離草稿、文字是否可讀、品牌風格是否一致。

適合 / 不適合情境

適合

  • 需要大量同風格圖卡。
  • 需要先用草稿鎖定構圖。
  • 需要把 AI 生圖結果再用程式後製。
  • 需要保留可追溯的創作過程。
  • 教學、講義、社群圖卡、知識庫封面。

不適合

  • 只想快速玩一次單張圖。
  • 對圖片精準商業授權與素材可用性要求很高，但沒有人工審查。
  • 需要極高畫質商業修圖，仍可能需要 Photoshop / Figma / 專業影像流程。
  • 額度緊張時，不宜大量反覆高品質改圖。

額度與成本注意

Threads 作者提醒：會耗 token，高品質生成和改圖會更快消耗額度。這點很重要。這類流程應避免讓 Codex 不斷試錯，建議採用「先草圖 → 先小樣 → 再批量」：

  • 先只生成 1 張低成本樣張。
  • 確認構圖與風格後再批量。
  • 後製交給本機腳本，不要每次都要求模型重畫。
  • 對多平台尺寸，優先用裁切 / 版型重排，不要每個尺寸都重新生圖。

與既有知識庫的關聯

這篇可與既有「Codex 簡報 PPT Skill｜先大綱、再單張試作」互補：簡報那篇聚焦大綱與樣張審核；本文聚焦「草圖作為視覺規格」與「Codex + 瀏覽器 + 程式後製」的創意生產線。兩者都支持同一個原則：不要一次交給 Agent 全自動亂生成，要先用中間產物鎖定方向。
