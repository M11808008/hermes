cc-switch 與 CLIProxyAPI｜讓 Claude Code 跑 GPT 模型時，先理解 proxy 翻譯層

原文重點

王拐子回應近期社群熱議的 CLIProxyAPI 工作流：OpenAI Codex 團隊的 Tibo 曾推薦用 CLIProxyAPI，讓 Claude Code 介面直接跑 GPT-5.6 Sol 類模型。社群常把它說成三步驟：安裝、連上、設 alias。

但作者實際安裝後認為沒那麼簡單：

  • CLIProxyAPI 是自架伺服器，需要先跑起來。
  • 每個帳號都要做授權。
  • 要把 Claude Code 指到本機 proxy。
  • 還要設定假 token / 環境變數。
  • 官方文件有些細節寫得很省，對只想試模型的人來說偏麻煩。

作者建議先理解中間 proxy 到底做什麼，再決定要走 CLIProxyAPI 還是 cc-switch。

---

為什麼需要 proxy？

Claude Code 對外說的是 Anthropic API / Claude 風格的請求格式；GPT / Codex 模型端則使用另一套格式。兩邊不能直接相接。

proxy / router 的角色像翻譯員：

  • Claude Code 把請求送到本機一個「看起來像 Anthropic」的 endpoint。
  • proxy 把請求即時翻譯成 GPT / Codex provider 可理解的格式。
  • 模型回覆後，再翻譯回 Claude Code 能顯示的格式。
  • 對 Claude Code 來說，它以為自己仍在跟 Claude-compatible endpoint 溝通。

理解這件事後，就能看懂：真正有價值的是協定翻譯與 provider 管理，而不是某一串 alias 本身。

---

CLIProxyAPI vs cc-switch

作者的判斷是：

  • CLIProxyAPI：適合想自架 gateway、寫腳本、自動化、多帳號或深入客製的人。彈性高，但需要自己跑 server、設定環境變數與處理授權。
  • cc-switch：適合只想在 Claude Code / Codex / Gemini CLI 間快速切換 provider，降低設定摩擦的人。它以桌面 app 形式提供本地 router，透過 UI 點選完成切換。

GitHub / Brave Search 交叉驗證：

  • `farion1231/cc-switch`：描述為跨平台 desktop all-in-one assistant，支援 Claude Code、Codex、OpenCode、OpenClaw、Gemini CLI、Hermes Agent 等工具。
  • `router-for-me/CLIProxyAPI`：描述為把 Antigravity、ChatGPT Codex、Claude Code、Grok Build 等包成 OpenAI / Gemini / Claude / Codex compatible API service。

---

對 AI 學習雷達的價值

這篇值得保存，因為它把「模型混搭」從炫技拉回工程現實：

  • 一個 agent UI 不等於一個模型後端。
  • Claude Code 的互動體驗可以和 GPT / Codex 後端拆開。
  • 但拆開後一定需要協定翻譯、授權、token、環境變數與本機安全邊界。
  • 工具選型要看目標：快速試用、長期自動化、多人共享、多帳號管理或企業導入，答案不同。

這也呼應近期 AI coding stack 的拆層趨勢：harness / agent loop、model provider、proxy/router、skills、MCP、local credential store 會逐漸變成可組合元件。

---

實務提醒

  • 不要把 OAuth token、API key、cookie 或 proxy config 貼到公開 repo / 社群。
  • 本機 proxy 應綁定 loopback，例如 `127.0.0.1`，避免暴露到區網或公網。
  • alias / shell profile 修改前應備份，避免污染原本 Claude Code / Codex 設定。
  • 先確認帳號方案、模型名稱與額度池，避免以為能用 GPT-5.6 Sol 但實際帳號不可用。
  • 若只是試用，cc-switch 類 UI 工具可降低摩擦；若要自動化或整合多帳號，CLIProxyAPI 類 gateway 較有彈性。
