NVIDIA LocateAnything-3B 本機視覺定位測試｜OCR、物件偵測與 Promptable Grounding 觀察

來源：欒帥在本機測試 NVIDIA LocateAnything-3B 並補充兩張 LocateAnything Lab 截圖。官方模型卡顯示 LocateAnything-3B 是 NVIDIA 發布的 3B vision-language grounding model，核心能力是把自然語言 prompt、物件類別、OCR/GUI/pointing 等任務轉成 bounding boxes 或 points。

一句話摘要

LocateAnything-3B 比傳統 YOLO 更像「可用文字指定目標的通用視覺定位模型」：可以做 object detection、OCR text localization、GUI grounding、pointing 等任務；但本機實測權重約 7.5GB、載入後約吃 16.5GB VRAM，slow 模式在 RTX 5090 上約 2.4 秒，精準場景可用，即時高頻辨識仍需審慎評估。

| 項目 | 實測觀察 |

|---|---|

| 模型 | NVIDIA LocateAnything-3B local visual grounding |

| 權重大小 | 約 7.5GB |

| 載入後 VRAM | 約 16.5GB |

| 推估顯卡門檻 | 至少 24GB VRAM 顯卡較安全；16GB 卡可能很緊，需量化/降解析度/省記憶體策略 |

| 測試顯卡 | RTX 5090 |

| 任務介面 | Phrase、Objects、OCR、GUI Box、GUI Point、Point 六類 |

| 核心差異 | 可自由用 prompt 指定偵測目標，比固定類別 detector 彈性更高 |

官方模型重點補充

根據 NVIDIA Hugging Face model card 與專案頁：

  • LocateAnything 是 fast and high-quality visual grounding VLM。
  • 模型參數約 3B。
  • 架構基於 transformer-based vision-language model。
  • Vision encoder：MoonViT。
  • Language model：Qwen2.5-3B-Instruct。
  • Multimodal projector：MLP projector。
  • 核心創新是 Parallel Box Decoding（PBD）：以 block-wise multi-token prediction 方式一次預測完整 bounding boxes / points，而不是逐 token 自回歸輸出座標。
  • 訓練資料規模官方稱包含 12M images、138M+ queries、785M bounding boxes，涵蓋自然場景、機器人、駕駛、GUI、文件理解等域。
  • 支援輸入：RGB image + text prompt。
  • 支援高解析度：production image resolution up to 2.5K。
  • 輸出是文字序列，內含 `<ref>` 與 `<box>` 等結構化座標 token。
  • 官方定位為 research and development only。

六大任務怎麼看

| 任務 | 用途 | 適合場景 |

|---|---|---|

| Phrase | 根據描述定位目標 | 「找穿紅衣的人」、「找畫面左側按鈕」這類 referring expression grounding |

| Objects | 多物件類別偵測 | 輸入 dog, car, bicycle 等類別，回傳多個 boxes |

| OCR | 文字偵測與定位 | 路牌、文件、截圖中文字區塊定位 |

| GUI Box | 找 GUI 元件區域 | UI 自動化、RPA、agent 操作畫面理解 |

| GUI Point | 找 GUI 可點擊點 | 電腦使用 agent 的點擊目標定位 |

| Point | 以點回應目標位置 | 需要 point-based localization 而非完整 box 的場景 |

slow / hybrid / fast 模式差異

官方 README 對 generation modes 的說明如下：

| 模式 | 官方定位 | 欒帥實測解讀 |

|---|---|---|

| fast | MTP only，不 fallback 到 autoregressive | 最快，適合簡單場景與低延遲需求，但複雜框選穩定性可能較低 |

| slow | pure autoregressive decoding | 最慢但最穩，適合 OCR、複雜版面、需要高精準度的定位 |

| hybrid | MTP 先跑，不確定時 fallback 到 AR，再切回 box boundary | 平衡速度與穩定性，官方預設；物件偵測截圖約 0.63 秒 |

與 YOLO 的差異

| 面向 | YOLO 類 detector | LocateAnything-3B |

|---|---|---|

| 類別 | 通常依訓練類別或 fine-tune 類別 | 可用 prompt 自由指定物件/文字/GUI 目標 |

| 輸入 | 圖片為主 | 圖片 + 自然語言 prompt |

| 輸出 | boxes、classes、scores | 結構化文字 token + boxes / points |

| 優勢 | 速度快、部署成熟、即時偵測友善 | 彈性高、可跨物件/文字/GUI/指令式定位 |

| 限制 | 類別擴展需訓練或 open-vocab 模型 | VRAM 較高、延遲較高、仍需驗證穩定性 |

| 典型用途 | CCTV、即時偵測、邊緣部署 | 視覺 agent、文件/GUI grounding、互動式定位、promptable detection |

實作與部署提醒

  • 若要本機跑完整模型，24GB VRAM 以上較合理；欒帥實測載入後約 16.5GB VRAM，還要預留影像解析度、KV cache、框選輸出與系統餘裕。
  • 官方目前主要支援 Linux、Transformers、BF16 precision 與 KV cache；TensorRT、TensorRT-LLM、Triton 尚未支援。
  • 官方列出的硬體相容包含 NVIDIA Ampere、Hopper、Lovelace、Blackwell；Hopper/Blackwell 可額外考慮 Magi Attention 做長上下文加速。
  • 如果要做 near real-time 視覺 agent，應先測 fast/hybrid 在實際輸入解析度下的延遲與錯誤率；不能只看 demo 圖。
  • GUI 自動化應加上二次驗證，例如點擊前確認目標文字、位置與 UI state，避免誤點。

欒帥可延伸測試清單

  • 同一張 OCR 圖分別跑 fast / hybrid / slow，比較框選漏字與時間。
  • 用中文、英文、混合語言 prompt 測 GUI Box 與 GUI Point。
  • 測手機截圖、LINE 對話截圖、瀏覽器管理介面，評估是否能做本機視覺 agent。
  • 測小物件、多物件、遮擋場景，觀察 hybrid fallback 是否穩定。
  • 與 YOLOv8/YOLOv10、Grounding DINO、Florence-2、Qwen-VL 系列做同圖比較。
  • 嘗試 4-bit/8-bit 量化或降低解析度，確認是否可讓 16GB 顯卡可用。

判讀限制

  • 欒帥提供的是單機初步實測，不代表所有硬體、所有影像解析度與所有場景的平均效能。
  • 16.5GB VRAM 是本次環境觀察值；不同 precision、batch size、image size、Transformers 版本與 CUDA/PyTorch 設定會改變記憶體占用。
  • OCR slow 模式約 2.4 秒是在 RTX 5090 上的觀察；其他 GPU 可能差距很大。
  • 模型卡標示 research and development only；若要用於生產、安防、醫療、交通或自動點擊操作，需要額外安全與品質驗證。
