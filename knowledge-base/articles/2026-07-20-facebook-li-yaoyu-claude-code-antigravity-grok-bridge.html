Claude Code 串 Antigravity/Grok：用 agy -p 與 grok -p 當省額度 Subagent 的橋接思路

來源是 Claude Taiwan 社團李耀宇的公開貼文。貼文表示：開始使用 Claude 後幾乎不再用 Gemini，但發現可透過 `agy -p` 讓 Claude 驅動 Gemini 模型，甚至也能用少量 Sonnet/Opus 4.6 做 Subagent 類任務；因此做了 `cc-to-antigravity-cli-bridge`，讓 Google Antigravity 承接 Claude Code system prompt。貼文也提到 `cc-to-grok-bridge`，偏向讓 `grok -p` 處理搜尋引擎抓資料類任務。

一句話摘要

這則貼文的重點不是「換掉 Claude Code」，而是把 Claude Code 當主控，把 Gemini/Antigravity 或 Grok 當可呼叫的外部研究與協作 lane：用 `agy -p` / `grok -p` 承接搜尋、第二意見、低風險子任務與額度分流，同時透過橋接 repo 盡量共享規則、prompt 與安全邊界。

對欒帥的實務價值

  • 額度分流：Claude Code 繼續當主控，Gemini/Grok 處理研究、搜尋、第二意見、草稿或低風險批次任務。
  • 搜尋備援：當 Claude 內建 web 工具遇到反爬或資料不足，可用 `agy -p` / `grok -p` 走另一條搜尋與整理路徑。
  • Prompt / rules 共用：Antigravity 方向用 `AGENTS.md`/`CLAUDE.md` 共享系統提示；Grok 方向則進一步處理 hooks、memory 與 rules 對齊。
  • 教學亮點：很適合放進「多代理協作」單元，展示主代理、外部 CLI、子代理、額度與安全邊界的分工。

可採用的操作模式

  • 研究型任務：用 `scripts/ccagy.sh "問題"` 或 `scripts/ccgrok.sh "問題"`，讓 wrapper 加上 research-prefix，要求 dated sources、facts/speculation 分離與多角度搜尋。
  • 程式審查/第二意見：Claude Code 完成主要修改後，呼叫外部 CLI 針對 diff、架構或 edge cases 給第二輪檢查。
  • 低風險子任務：文件摘要、資料搜尋、測試想法、issue triage 可外包；涉及寫檔、secret、部署、付費 API 的動作仍應由主控與人工審核。
  • 課程示範：可用「Claude Code 是 conductor，Antigravity/Grok 是 section player」比喻，讓學生理解 agent orchestration 不是單一模型全包。

Caveats

  • 這不是官方 Claude Code subagent 標準功能，而是作者橋接 repo 實作與 CLI wrapper 工作流；實際可用性取決於本機是否已安裝、登入並授權相應 CLI。
  • Antigravity bridge README 明確說 `agy` 目前沒有 Grok bridge 那種完整 memory API 或 documented hook schema，因此安全與狀態共享能力較薄。
  • 若使用 Google 訂閱額度，README 提醒不要設定 `GEMINI_API_KEY`；否則可能改走 API 計費。
  • 社團貼文可見內容有限，本篇只歸檔公開頁可見文字與 GitHub/官方來源查核，不宣稱已驗證作者本機 workflow。

Why it matters

多模型、多 CLI 的新常態正在形成：Claude Code 不一定要獨自吃下所有研究、搜尋、審查與草稿任務。這類 bridge 的價值在於把「模型能力」轉成「可調度資源」：誰擅長搜尋、誰額度便宜、誰適合審查，都能被主代理按任務派工。但真正能進 production 的關鍵不是多叫幾個模型，而是 prompt/rules/security/memory 的邊界是否清楚、可審計且可回收。
