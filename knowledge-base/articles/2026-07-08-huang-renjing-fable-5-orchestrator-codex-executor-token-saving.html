黃仁靖｜Fable 5 省 Token 工作流：最強模型當 Orchestrator，Codex 寫 Code，Gemini 做 Reviewer

一句話摘要

這篇的重點不是「Fable 5 很會寫 code」，而是「Fable 5 最值得放在控制平面」：讓它切工單、定義 done、仲裁 reviewer、讀 diff 與決定是否 close，反而比親自寫大量 code 更省 token、更能收斂。

貼文脈絡

黃仁靖在 Claude Taiwan 分享一次 AI coding workflow 實驗：三個專案同時開 Fable 5 Ultracode，連續跑約 8 小時，據作者描述 Fable 5 額度只燒約 20%，卻把原本可能要一週、甚至 Codex 卡關兩天的工程問題推進完成。

作者強調，這次不是讓 Fable 5 直接大量寫 code，而是讓它當 tech lead / orchestrator：

  • Fable 5：最高層 orchestrator / arbitrator。
  • Codex 5.5：executor，跑 /goal 長模式。
  • Gemini 3.1 Pro / AGY：reviewer，只負責懷疑與挑毛病。
  • 測試與 smoke：客觀驗證。
  • 人類：最終產品驗收與方向判斷。

這篇與前一篇「Field Guide to Fable」可接在一起看：前者談 context、unknown matrix、stay in the loop；這篇則把「最強模型應放在哪」落到 AI coding 團隊分工。

Codex 單獨跑長任務的痛點

作者指出，Codex 不是不強，而是單獨跑大型工程任務時容易卡在最後 20%：

  • 測試是否真的通過？
  • 服務是否真的重啟？
  • curl smoke 是否成功？
  • 真實檔案是否跑過？
  • review finding 是真 bug 還是假陽性？
  • 改動是否該 commit？
  • 任務到底算不算 close？

問題不在前 80% 的 code，而在最後的收斂、驗收、邊界與完工定義。Codex 單獨跑時，容易一直做、一直加、一直修、一直報進展，但 diff 越滾越大，PLAN / REPORT 越來越多，任務仍 close 不起來。

Fable 5 的高槓桿位置：控制平面

作者認為 Fable 5 最值錢的地方不是當「廉價碼農」，而是做最貴、最關鍵的判斷：

  • 先診斷問題。
  • 把任務切成很窄的工單。
  • 定義驗收條件。
  • 指定禁區，哪些檔案不能碰。
  • 逐條查證 reviewer 的指控。
  • 否決假的 finding。
  • 把真的 finding 交給 Codex 修。
  • 讀 diff，不信完工報告。
  • 決定下一輪是否繼續。
  • 負責把任務 close 起來。

這其實是 AI coding 的控制平面 / 執行平面分離：

  • 控制平面：方向、邊界、驗收、仲裁、收斂。
  • 執行平面：改 code、跑測試、回報 diff 與風險。

Codex 5.5 的位置：低熵工單 executor

Codex 在這個配置中不是被丟一個大任務，而是接收 Fable 5 壓縮後的低熵 goal：

  • 只做這件事。
  • 只碰這幾個檔案。
  • 驗收條件是這些。
  • 不能放鬆 gate。
  • 不能順手重構。
  • 不能擴大 scope。
  • 做完回報 plan、diff、測試結果、剩下風險。

這比「幫我把 workflow 修好」更容易成功，因為 executor 不必同時猜根因、驗收標準、產品意圖與 reviewer 哪些說法可信。Fable 5 先把大任務壓成小、清楚、可驗收的任務，Codex 就能專注執行。

Gemini / AGY 的位置：異質 Reviewer，但沒有決策權

貼文中特別重要的一點是：reviewer 只負責懷疑，不負責拍板。

Gemini 3.1 Pro / AGY 可以 aggressive 地報問題、P1、contract risk、edge case；但 Fable 5 必須逐條查證：

  • 真的 finding → 交給 Codex 修。
  • 假陽性 → 直接否決。

作者提醒，很多 multi-agent workflow 失敗，是因為 reviewer 一講話，executor 就開始亂修。假陽性沒被擋掉，任務會越修越歪。

這個分工可濃縮成：

  • Gemini 負責懷疑。
  • Fable 負責仲裁。
  • Codex 負責修復。
  • 測試與 smoke 負責客觀驗證。
  • 人類負責最後產品驗收。

七個關鍵原則

作者整理這套流程的七個關鍵，可保存為 AI coding checklist：

1. Fable 5 不做苦工，只做控制平面

不要讓最強模型親自寫最多 code。讓它負責方向、切工單、驗收與仲裁，才能把 token 用在高槓桿判斷。

2. Codex 5.5 變成執行平面

Codex 適合做 executor，尤其是 /goal 長模式。只要上層 goal 寫得夠精確，就能發揮改 code、跑檢查、整理回報的能力。

3. Gemini 3.1 Pro / AGY 當異質 reviewer

同模型自審容易盲點重疊。異質 reviewer 的價值在於錯誤分布不同，可抓到不同類型問題。

4. Reviewer 沒有決策權

Reviewer 可以報問題，但不能直接指揮修復。所有 finding 需由 orchestrator 查證，避免假陽性帶偏 executor。

5. Done definition 放在模型外面

SPEC、PLAN、REPORT、HANDOFF、MEMORY、測試結果、commit 不是裝飾，而是讓每一輪不用只靠模型記憶，避免 context 變長後漂移。

6. 每一輪都要能收斂

不能只叫 agent「繼續努力」。每一輪都要有明確驗收：測試、smoke、真檔、diff review、commit 邊界。過了就收；不過就開下一個窄任務修。

7. 人類還是終審

AI 可以高速推進，但產品行為、商業判斷、風險接受，最後仍由人類決定。

為什麼 token 反而省

作者指出，舊流程最貴的不是 token，而是重工：

  • 卡死一小時沒產出。
  • 朝錯方向修一輪。
  • 把假 bug 當真 bug 修。
  • scope 越改越大。
  • 測試沒跑就說完成。
  • 下一輪重新理解上下文。
  • diff 爆炸，任務沒人敢收。

因此，多花一些 orchestrator / reviewer token，如果能減少重工、錯方向與假陽性，整體反而更省。Fable 5 看似昂貴，但放在最高槓桿位置時，是在放大 Codex，而不是取代 Codex。

作者的核心句：

Fable 5 不是拿來取代 Codex 5.5 的；Fable 5 是拿來放大 Codex 5.5 的。

分流表

貼文最後給出簡明分流：

  • 小而明確的單點修改：直接丟 Codex。
  • 需要跨檔案、跨服務、驗收、部署、close 的任務：走 Fable 5 orchestrator。
  • 需要找風險與盲點：丟 Gemini 3.1 Pro / AGY review。
  • 最後人類只看結果、產品行為、風險是否可接受。

這張表很適合轉成 AI coding 工作流 SOP。

與公開 multi-agent 架構的對照

使用 Brave Search 交叉驗證，這篇的實務經驗與多個公開 multi-agent pattern 呼應：

  • Anthropic 的 multi-agent research system：lead agent 規劃、派生 subagents、綜合結果並決定是否需要更多研究。
  • OpenAI Agents SDK：multi-agent orchestration 中，agents as tools 適合讓 specialist 幫忙 bounded subtask，但不接管整體對話。
  • OpenAI orchestration 文件：第一個設計選擇是決定每個 workflow branch 由誰擁有最終回答 / 決策權。
  • Microsoft Foundry connected agents：primary agent delegates to purpose-built subagents。

這些文件都支持一個方向：多 agent 不是越多越好，而是要有明確 owner、delegation boundary、specialist role 與決策權設計。

對 AI 學習雷達的價值

這篇值得高優先保存，因為它把「AI coding 團隊」從口號落到可操作分工：

  • 最強模型不一定該做所有事。
  • 高能力模型更適合放在 control plane。
  • coding 模型應接低熵、窄範圍、可驗收工單。
  • reviewer 必須與 arbitrator 分離。
  • done definition 要外部化，不靠模型記憶。
  • human-in-the-loop 不只是審美，而是最終風險與產品責任。

這也可延伸為欒帥 / Hermes 的工作流設計：歸檔、部署、PDF 處理、課程頁、私密名單等任務，都可以用「orchestrator → executor → reviewer → objective gates → human final acceptance」來設計。

風險與限制

  • 數字如「8 小時、Fable 20%、Codex 40%、三個專案」是作者個人實驗與自述，樣本數有限，不能視為通用 benchmark。
  • Fable 5、Codex 5.5、Gemini 3.1 Pro / AGY 的版本與帳號額度條件會影響結果。
  • 這套流程需要紀律：SPEC、PLAN、REPORT、HANDOFF、MEMORY、測試、smoke、commit 邊界缺一不可。
  • 多 agent workflow 若沒有清楚 owner 與仲裁者，仍可能變成噪音與重工。
  • 外部記憶檔案會累積，需定期摘要 / 精簡，避免每輪讀取造成 context 膨脹。
