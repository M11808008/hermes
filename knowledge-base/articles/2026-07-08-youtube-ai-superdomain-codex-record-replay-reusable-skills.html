AI超元域｜Codex Record & Replay：錄一次操作流程，自動生成可複用 Skill 的工作流實測

一句話摘要

Codex Record & Replay 的關鍵不是「錄屏回放」本身，而是把一段人工示範轉成可檢查、可編輯、可重複調用的 agent skill，讓不擅長寫長 prompt 的使用者也能把複雜工作流產品化。

影片主題

影片介紹 OpenAI Codex 的 Record & Replay（錄製與回放）插件。根據影片描述，其核心流程是：

  • 使用者簡單描述要建立的工作流。
  • 開始錄製一遍實際操作。
  • Codex 觀察流程，調用 skill creator。
  • 生成可複用的 skill。
  • 之後使用者可直接調用該 skill 重跑工作流。
  • 若 skill 跑偏，可要求 Codex 修改與優化，直到流程更穩定。

這個功能對不擅長寫提示詞、也不熟悉 skill 結構的人有吸引力，因為它把「示範」變成「可執行的文字流程與資源包」。

官方文件驗證：Record & Replay 是什麼

OpenAI Developers 的 Record & Replay 文件明確說明：

  • Record & Replay is a fast way to create a skill from a demonstrated workflow。
  • 使用者可在 Codex app 的 Plugins 中選擇 Record a skill。
  • 錄製結束後，Codex 會檢查 captured workflow 並草擬 skill。
  • Skill 會描述何時使用、需要哪些輸入、要遵循哪些步驟，以及如何驗證結果。
  • 之後 Codex 可將該 skill 作為 reusable context，並用當前環境可用工具完成工作流，例如 Computer Use、browser actions、installed plugins。

官方文件也列出適合的示範案例：報支出、訂停車位、建立 issue、發布影片、下載定期報告等。這些都符合「流程固定、輸入可變、結果可驗證」的特徵。

可用性與限制

OpenAI 官方文件指出：

  • Record & Replay available on macOS。
  • 初期可用性排除 EEA、United Kingdom、Switzerland。
  • Computer Use 必須可用且已啟用。
  • 若組織用 `requirements.toml` 管理 Codex，`[features].computer_use` 也會控制 Record & Replay；若 `computer_use = false`，兩者都不可用。

因此這支影片中的示範應被理解為：在符合環境、地區、權限與 Codex app 條件下可用；不代表所有 Codex CLI / Windows / Linux / 企業工作區都一定可直接使用。

Codex Skills 的定位

OpenAI Developers 的 Agent Skills 文件說明：

  • Skill 是 Codex 的 reusable workflow authoring format。
  • Skill 可包含 instructions、resources、optional scripts。
  • Skill 目錄通常包含 `SKILL.md`，可選 `scripts/`、`references/`、`assets/`、`agents/openai.yaml`。
  • Codex CLI、IDE extension、Codex app 都支援 skills。
  • Skills 使用 progressive disclosure：Codex 一開始只看到名稱、描述與路徑，只有決定使用某個 skill 時才讀完整 `SKILL.md`。
  • 若已知工作流且「示範比描述更容易」，官方建議使用 Record & Replay；若想用文字描述 skill，可使用 built-in `$skill-creator`。

這點很重要：Record & Replay 不是取代 skill，而是降低 skill authoring 的門檻。

三個實戰案例判讀

案例一：Claude Design 生成 5 頁 AI 趨勢 PPT

影片第一個案例錄製「調用 Claude Design 建立 PPT」的流程，接著讓 Codex 產生對應 skill，再測試自動生成與下載 5 頁 AI 趨勢報告 PPT。

這類流程適合 Record & Replay，因為它包含：

  • 固定入口：打開特定工具或 plugin。
  • 可變輸入：主題、頁數、風格、內容重點。
  • 固定產出：PPT / slides / downloadable file。
  • 可驗證結果：是否生成 5 頁、是否下載成功、檔名是否符合規則。

但實務上必須補上 hidden preferences：檔名規則、版面風格、是否要繁中、是否要引用來源、輸出放哪個資料夾。

案例二：Gemini 影片生成與發布

第二個案例把流程升級到 Gemini 影片生成、下載、上傳與發布。這比 PPT 更高風險，因為它涉及外部平台、檔案上傳與公開發布。

適合保存的判讀：

  • Record & Replay 可以記住 GUI / web app 操作流程。
  • 影片生成流程若固定，可被包裝成 skill。
  • 但「發布」屬於高影響操作，應保留人工確認。
  • Skill 應明確要求在公開發布、上傳或提交前停下等待使用者核准。

這類流程若沒有 approval gate，很容易把測試內容誤發到公開平台。

案例三：Claude Code 互動模式與 Deep Research

第三個案例嘗試突破 Claude Code 非互動模式限制，在互動模式中調出 deep research，並透過 skill 設定限定 subagents 數量與避免每次重複授權。

這個案例的價值在於：Record & Replay 不只適合「表單式 GUI 操作」，也能捕捉較難用一般非互動 CLI 表達的流程。當某些工具必須透過互動式操作才能觸發，錄製流程可能比手寫腳本更快。

但風險也更高：

  • 互動模式中的狀態與權限可能不穩定。
  • deep research 可能消耗大量額度或啟動多個 subagents。
  • Skill 需明確設定 subagents 上限、停止條件、輸出格式與驗收方式。

為什麼這件事重要

Record & Replay 代表 agent workflow 的一個轉折：

  • 以前：你要會寫 prompt、會寫 skill、會描述每個步驟。
  • 現在：你可以先做一遍，讓 Codex 把示範轉成 skill，再由人類審修。

這降低了工作流自動化門檻，尤其適合「我做得出來，但說不清楚每一步」的場景。

它也和近期 Fable / Unknowns 脈絡呼應：很多人其實有 tacit knowledge，知道怎麼做，但不知道怎麼寫成規格。Record & Replay 把 tacit workflow 先外化成可讀 skill，再讓人補 hidden preferences 與 verification gates。

最適合 Record & Replay 的任務

適合

  • 重複性高的瀏覽器或桌面工作流。
  • 需要多個工具串接，但步驟相對固定。
  • 產出可明確驗證：檔案、表單、報告、影片、issue、下載檔。
  • 使用者示範比寫 prompt 更快。
  • 需要把個人 workflow 分享給團隊或重複使用。

不適合直接全自動

  • 金流、付款、公開發布、寄信、刪除資料。
  • 權限敏感或涉及個資的流程。
  • 每次決策都高度依賴人類判斷的流程。
  • 沒有穩定 UI 或頻繁變動的網站。
  • 需要大量創意取捨但沒有明確驗收標準的任務。

Skill 生成後一定要補的內容

官方文件也提醒：錄製後應 refine skill，標註 hidden preferences，例如 naming conventions、field defaults、decision points。

建議每個 Record & Replay skill 都補上：

  • 何時觸發：哪些任務適用，哪些不適用。
  • 必要輸入：主題、檔名、輸出位置、帳號、平台。
  • 禁止事項：不可付款、不可公開發布、不可刪除、不可改權限。
  • 人工確認點：上傳前、發布前、寄出前、提交表單前。
  • 驗收條件：頁數、格式、是否下載成功、是否有來源、是否有截圖證據。
  • 失敗處理：登入失敗、UI 找不到、生成內容跑偏、外部平台錯誤時要停下回報。
  • 隱私邊界：不要記錄密碼、token、個資與不必要的截圖。

對欒帥 / Hermes 的實踐價值

這支影片很值得保存，因為它與 Hermes 的「可複用技能」概念高度一致。

可延伸到：

  • 錄製一次「AI 學習雷達歸檔 → 建置 → 部署 → 驗證」流程，轉成 skill checklist。
  • 錄製一次「課程頁內容整理 → 本機預覽 → GitHub Pages 部署」流程。
  • 錄製一次「YouTube 影片 metadata / transcript / 官方驗證 → KB 筆記」流程。
  • 錄製一次「私密名單處理」流程，但必須加上不可公開部署與個資保護 gate。
  • 錄製一次「簡報 / 圖卡 / 課程宣傳素材」流程，讓設計偏好外化成可檢查 skill。

尤其對欒帥偏好的「高自治、低摩擦、可重複、可驗證」工作流，Record & Replay 的概念可以作為未來 Hermes skill authoring 的參考。

風險與注意事項

  • 影片未提供 transcript，本筆記依 metadata / 章節 / 官方文件整理，非逐字稿摘要。
  • Record & Replay 初期可用性有限，官方文件標示 macOS 且排除部分地區；也需要 Computer Use 可用並啟用。
  • 錄製工作流可能捕捉敏感資訊，需避免密碼、token、個資與機密資料入鏡。
  • 自動發布、上傳、提交表單等高影響動作必須有人工 approval gate。
  • 自動生成的 skill 不應直接信任，需人工 review `SKILL.md`，補上 hidden preferences、禁止事項與驗收條件。
  • UI 變動會造成 skill 失效，需定期維護。
  • Skill 可以降低 prompt 門檻，但不能取代測試、差異檢查與人類最終驗收。
