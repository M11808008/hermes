Omnigent｜把 Claude Code、Codex、Pi 與自訂 Agent 串成同一層的開源 Meta-harness

來源為 Keith Rumjahn 在 Threads 分享 Omnigent：一個把多個 AI agent / coding harness 串在同一個管理層的開源專案。貼文重點是 Claude Code、Codex、Pi 與自訂 agent 可以在同一個 harness 裡協作，支援 multi-agent coding、live collaboration 與自訂 control policy。

一句話摘要

Omnigent 是 Databricks 開源的「meta-harness」：它不是又一個單一 coding agent，而是架在 Claude Code、Codex、Pi、OpenAI Agents、Claude Agents SDK 與自訂 YAML agents 之上的共同層，讓多個 agent 可以被組合、治理、沙盒化、協作與跨裝置共享同一個 live session。

為什麼重要

過去使用 coding agent 的痛點是每個工具都是孤島：Claude Code、Codex、Pi、Gemini search、Slack、文件與終端都各自有上下文、權限與工作資料夾。Omnigent 的核心觀點是：未來不是只選一個 agent，而是需要一層在所有 harness 之上的共同介面，讓不同模型與 agent 能一起工作。

Databricks 在官方文中稱這一層為 meta-harness，目標是處理單一 harness 停止處理的問題：composition、control、collaboration。

核心能力

| 能力 | 說明 |

|---|---|

| Composition | 在同一個 session 中組合 Claude Code、Codex、Pi 與自訂 agents，無需為每個 harness 重寫流程 |

| Multi-agent coding | 內建 Polly 範例：像 tech lead 一樣規劃、委派 coding sub-agents 到平行 git worktrees，再交給不同 vendor 的 reviewer 審查 |

| Model / harness switching | 可用 API key、Claude / ChatGPT subscription、OpenAI / Anthropic-compatible gateway、Databricks profile 等憑證切換模型 |

| Live collaboration | 分享 live session URL，隊友可觀看、留言、審查檔案，甚至共同下指令 |

| Multi-device access | 同一個 agent session 可在 terminal、browser、mobile、macOS desktop app 與 REST API 使用 |

| Cloud sandbox | 可在本機、Modal、Daytona 等 disposable sandbox 或 server-managed host 執行 |

| Contextual policies | 可依 session 狀態做動態管控，例如下載 npm package 後要人工批准才能 git push |

| Cost policies | 可追蹤 session LLM cost，超過指定花費時暫停並詢問是否繼續 |

| Secure OS Sandbox | 可限制檔案系統與網路存取、隱藏憑證，並由 egress proxy 在核准請求中注入 token |

| Custom agents | 用 YAML 定義自己的 agent，並能用一行變更在不同 harness 間移植 |

安裝與啟動

官方 quick start 提供一行安裝：

```bash

curl -fsSL https://raw.githubusercontent.com/omnigent-ai/omnigent/main/scripts/install_oss.sh | sh

```

也可用 uv、pip 或 Homebrew：

```bash

uv tool install omnigent

pip install "omnigent"

brew install omnigent-ai/tap/omnigent

```

基本啟動：

```bash

omnigent

```

指定 harness：

```bash

omnigent claude

omnigent codex

omnigent run path/to/agent.yaml

```

啟動 local server 與 web UI：

```bash

omnigent server start

omnigent host

```

預設 web UI 位於：

```text

http://localhost:6767

```

內建 agent 範例

| Agent | 說明 |

|---|---|

| Polly | 多 agent coding orchestrator；自己不寫 code，而是規劃、分派給 Claude Code / Codex / Pi 等 sub-agents，在平行 git worktrees 工作，再找不同 vendor reviewer 審查 |

| Debby | 雙頭腦 brainstorming partner；同一問題交給 Claude 與 GPT 兩邊回答，可用 `/debate` 讓雙方互相批判後收斂 |

對欒帥的實用價值

  • 非常適合觀察 2026 coding agent 的下一層抽象：從「單一 agent」走向「agent team orchestration」。
  • 可作為 AI 課程中的高階案例：Agent 工程不只 prompt，而是 harness、session、policy、sandbox、cost、collaboration 與 deployment。
  • 對欒帥既有 Codex / Claude Code 工作流有直接參考價值：Omnigent 提供一個把不同 agent 串到同一 session 的可能路徑。
  • Polly 模式可轉成課程 demo：「規劃 agent → coding agent → reviewer agent → 人類 merge」的工程流程。
  • Contextual policy 與 sandbox 特別值得研究，因為多 agent 協作真正困難的不是讓 agent 說話，而是誰有權執行高風險動作、誰可以覆蓋結果、誰能碰 token 與檔案。

風險與注意

  • 專案目前標示 alpha，開發速度快，CLI、API、policy 格式與部署方式可能變動。
  • Threads 說 MIT 授權，但官方 GitHub / README / Databricks 文章顯示 Apache 2.0，應以官方 repo 為準。
  • 多 agent coding 對成本、token、並行任務、git worktree 與 repo 權限要求較高，不宜一開始就對重要 repo 開 YOLO 模式。
  • 若使用 subscription 模式連 Claude / ChatGPT CLI，要注意各服務條款、帳號限制與額度。
  • 多人 live collaboration 會涉及本機檔案、終端、token、網路請求與工作目錄共享，正式使用前要先配置 sandbox、policy 與權限。

建議測試流程

  • 在非敏感 toy repo 安裝 Omnigent。
  • 先以 `omnigent codex` 或 `omnigent claude` 測試單一 agent session。
  • 啟動 `omnigent server start` 與 `omnigent host`，確認 web UI 與手機可接續同一 session。
  • 測試 Polly 範例，觀察多 agent 分工、git worktree 與 reviewer 流程。
  • 設定 policy：先限制 git push、網路、寫入範圍與 spend cap。
  • 再考慮接到真實 repo 或 team collaboration。
