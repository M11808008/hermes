Nicky Hsieh｜Claude Fable 5 短期開放：五個高槓桿用法與省額度策略

一句話摘要

Claude Fable 5 這類稀缺高階模型不該拿來做一般雜務，而應該用在「吃大量上下文、做深度判斷、產生可交給便宜模型執行的決策與計畫」上。

Threads 串文重點

Nicky Hsieh 轉述 Peter Yang 影片「Don’t Waste Claude Fable 5」的核心觀點：Fable 5 被重新開放後，訂閱用戶可短期使用，但額度受限；因此每次對話都要花在刀口上。官方說法也指出，Fable 5 在 2026-07-07 前可占部分付費方案每週用量上限的 50%，之後改以 usage credits 使用。

這篇值得收錄，不只是因為 Fable 5 的可用期很短，而是它清楚提出「昂貴模型該負責什麼」的工作分層：高階模型做判斷、診斷、規劃與審查；一般模型做準備、實作與大量產出。

五個值得用 Fable 5 的任務

1. 先問模型：哪些工作值得它出手

直接讓 Fable 檢視專案與記憶，回答：「列出五件最需要深度思考、最值得由你處理的事。」

這是很好的額度配置方法。與其人類先猜哪個任務最難，不如讓最強模型先掃描上下文，找出最需要跨資料整合、萃取洞見、做取捨的問題。

2. 人生與商業決策建議

Peter Yang 的作法是先準備 plan doc，包含年度目標、決策原則、定位、能量來源與財務狀況，再接上 MCP / API 資料源，例如銀行、電子報、YouTube、網站數據。最後才請 Fable 給建議。

這類任務的槓桿高，因為它不是問一個孤立問題，而是讓模型綜合長期資料，寫出「哪裡賺錢、哪裡漏水、未來三個月該做什麼」的診斷。

3. 出貨前抓重大 bug

串文提到 Peter 用 Fable 開多個 agent 掃描一個健身 App codebase，找到超過 12 個重大 bug，其中包含登出後可能把 A 用戶資料漏給 B 用戶的問題。

這代表高階模型可以用在出貨前的安全與品質 gate：不是取代一般測試，而是用更強的上下文理解能力找跨檔案、跨狀態、跨使用者流程的問題。

4. Fable 規劃，便宜模型執行

這是最實用的成本策略：讓 Fable 寫出足夠細的實作計畫，包含階段、決策點、風險、開放問題與驗收條件；再交給 Opus、GPT 或其他較便宜模型寫 code。

換句話說，昂貴模型負責「腦」，便宜模型負責「手」。這個模式適合大型功能、重構、資料管線、課程設計、知識庫整理等工作。

5. 重構大型 codebase 或 skill 系統

串文舉例 Stripe 用 Fable 遷移大量 Ruby code；Peter 也用它 audit 自己 40 多個 AI skills，找出 13 個該修的地方。

對 AI 學習雷達來說，這點尤其有價值：我們可以把 Fable 類模型用在 skill、知識庫分類、歸檔規則、課程素材與多代理 workflow 的整體審查，而不是只讓它回覆零散問題。

三個省額度原則

  • 便宜模型做準備，Fable 只做思考。
  • Fable 做規劃與審查，其他模型做執行。
  • effort 不要一律拉滿；先用 medium / lower effort，並觀察模型是否繞圈燒 token。

對 AI 學習雷達的價值

這篇的核心不是某個模型名稱，而是「高階模型治理」。當模型開始分層定價、使用額度受限，AI 工作流也要從單一聊天框變成任務路由系統：

  • 低階任務：整理資料、轉格式、初稿、搜尋、批次處理。
  • 中階任務：寫 code、生成內容、建立 MVP、跑測試。
  • 高階任務：定策略、做風險判斷、審查大型系統、產生多階段計畫。

這可延伸成課程案例：「如何把昂貴模型用在最值得的 20% 問題上」。

可直接套用的 prompt

  • 「請先閱讀我的專案目標、近期任務與既有記憶，列出 5 個最值得由你處理的高槓桿任務，並說明為什麼不適合交給一般模型。」
  • 「請把這個功能拆成可交給便宜模型執行的 implementation plan：包含階段、風險、驗收標準、測試策略與需要人工決策的地方。」
  • 「請以出貨前審查員角色掃描這個 codebase，優先找資料外洩、權限錯誤、狀態不同步、邊界條件與測試缺口。」
