原文重點

Yu-Sen Lai 分享的實驗流程可整理為：

  • 先用 Codex 裡的新 GPT‑5.6 Sol 強化一套給 ChatGPT 使用的自製 MCP。
  • 讓 ChatGPT 在一般 Chat 模式中，透過 MCP 操作本機開發環境。
  • 同時啟動多個 AGY CLI，由 Gemini 3.5 Flash 擔任子代理並行處理任務。
  • 由較強模型擔任總指揮，負責拆任務、分工、整合、測試與部署。
  • 最後產出兩款可玩的 3D 網頁遊戲：太空賽車與 3D 射擊。

貼文作者特別強調，這次體驗讓他意識到「子模型是否好用」高度取決於總指揮是否會拆解任務、安排流程與檢查結果。換句話說，Gemini 3.5 Flash 類模型在被放到合適位置、由更強模型協調後，能成為高性價比的平行工作者。

為什麼值得放進 AI 學習雷達？

這則貼文的價值不是「AI 做了兩款小遊戲」而已，而是它示範了 coding agent stack 的新分工：

  • ChatGPT / Codex：高階規劃、任務拆解、整合與品質把關。
  • MCP：把 ChatGPT 的一般對話模式接到本機工具與開發環境。
  • AGY CLI / Gemini 類子代理：負責可平行化的子任務，例如模組開發、素材整合、測試修復、部署檢查。
  • 預覽部署平台：讓每輪工作結果能立刻在瀏覽器驗證。

這其實很接近「一個總工 + 多位 junior engineer + CI / preview environment」的結構。真正的能力不只來自模型本身，而來自任務切分、上下文隔離、成果回收、測試與整合的 orchestration。

實務啟發

若要把這個案例轉成可重複的內部工作流，可抽成以下設計原則：

  • 總指揮模型只做高槓桿工作：拆任務、定介面、合併結果、跑測試、做最終判斷。
  • 子代理要給明確邊界：每個代理負責單一模組、檔案範圍或驗證任務，避免互相覆寫。
  • 每個子任務都要產出可驗證 artifact：檔案 diff、測試結果、preview URL、截圖或 log。
  • MCP / CLI bridge 必須有權限邊界：避免一般 Chat 模式取得過寬的本機操作權限。
  • 模型名稱與額度池不可假設永久有效：應在啟動前自動檢查目前可用模型與 quota / plan 限制。

---

風險與注意事項

  • 自製 MCP 若能操作本機開發環境，需小心檔案、shell、token 與瀏覽器憑證權限。
  • 多代理並行容易產生衝突修改，最好搭配 git worktree、明確檔案邊界或 patch review。
  • 「不消耗 Work / Codex agentic 額度池」是貼文作者的實測感受；不同帳號、方案與時點可能不同，不能當成永久保證。
  • 對外部署 preview 時，應確認沒有洩漏 API key、OAuth token、內部 endpoint 或私人資料。
  • 若用於課程示範，建議強調 orchestration、驗證與安全，而不是鼓勵學生盲目同時開很多代理。

對課程 / 工作坊的可用角度

這個案例很適合放進「AI agent 協作工程」單元：

  • 示範 MCP 如何把聊天介面變成工具操作入口。
  • 比較單代理 vibe coding 與多代理 orchestration 的差異。
  • 設計一個小型實作：一位總指揮 agent 拆出 UI、game logic、測試、部署四個子任務，再由子代理各自完成。
  • 讓學員觀察：好的任務拆解與驗證標準，往往比單純換更貴模型更重要。
