川野｜Codex 不只是寫文案：把它配置成可復用的工程工作系統

川野這篇 X Article 的重點不是「Codex 能不能寫程式」，而是：如果只把 Codex 當成一次性聊天框或自動補全，就只用了很淺的一層；真正拉開差距的是把它配置成一套能記住規則、復用流程、接外部工具、定期巡檢與分工協作的工程工作台。

一句話摘要

Codex 的高階用法不是更長的提示詞，而是把專案規則、常用工作流、外部工具、驗證機制與權限邊界沉澱成可重複運行的系統。

核心觀點

原文的關鍵句是：「Codex 的真正門檻，不是會不會寫提示詞，而是你有沒有把自己的工作方式沉澱給它。」這可視為從 prompt engineering 轉向 workflow engineering 的典型案例。

11 個操作層次

| 層次 | 做法 | 價值 |

|---|---|---|

| AGENTS.md | 寫最常用命令、最易踩坑約束、必守風格、驗證方式 | 讓 Codex 少猜專案隱性規則 |

| Skills | 把 release note、安全審查、PR 描述、週報等重複提示詞產品化 | 從一次性溝通變成可調用工作流 |

| MCP | 接 GitHub、Linear、Figma、資料庫、內部文件、知識庫 | 從代碼助手變成工作台 |

| Slash Commands | /compact、/diff、/review、/mention、/status、/permissions、/new、/init | 管理長任務與上下文風險 |

| 圖片輸入 | 丟 UI 截圖、設計稿、錯誤彈窗 | 前端問題不再靠文字猜測 |

| In-app Browser | 開頁面、看結果、截圖、修改、再驗證 | 建立前端視覺迴圈 |

| Review-first | 先讓 Codex review 風險，再允許動手 | 降低大改動改偏與刪掉隱性邏輯的風險 |

| Automations | 每日簡報、依賴風險、PR/CI 巡檢、部署狀態檢查 | 從即時工具變成背景巡檢助手 |

| Subagents | 呼叫鏈、測試、安全、歷史實作分給不同 agent 查 | 複雜任務避免單一上下文爆炸 |

| 權限與 Hooks | 控制網路、改檔、刪檔、重置 Git、測試前置檢查 | 讓 Codex 在可控邊界內做更多事 |

| IDE / CLI / App | 小任務在 IDE，本地工程在 CLI，跨頁面長任務在 App | 依任務現場選入口 |

對欒帥的可用意義

  • 這篇可作為「AI 工程工作流」課程素材：同一個模型，因為配置方式不同，會從聊天工具變成可協作系統。
  • 可對照 Hermes / Claude Code / Codex 的共同模式：專案規則檔、技能、工具接入、子代理、排程、權限護欄。
  • 它提醒使用者不要只收藏 prompt，而要把高頻工作流程沉澱成可驗證、可調用、可維護的資產。
  • 對團隊導入 AI coding agent，優先順序應是：先定義專案護欄，再建立 review 與測試流程，最後才擴大自動化。

實務導入清單

  • 為每個重要 repo 建立短而精準的 `AGENTS.md` 或等效規則檔。
  • 將重複三次以上的任務做成 Skill，而不是每次重新貼提示詞。
  • 大改動採用「先 review、再實作、最後再 review」的三段式。
  • 前端任務盡量加入截圖、瀏覽器觀察與視覺驗證。
  • 排程與自動化任務必須有成功標準、失敗通知與停止條件。
  • 高風險操作如刪檔、重置 Git、上 production、讀密鑰、處理個資，必須透過權限與 Hooks 限制。

判讀限制

本文根據 X Article 擷取整理；內容屬實務方法論與產品能力觀察，未逐一驗證不同 Codex App / CLI / IDE 入口在各區域與帳號方案中的功能差異。實際導入時仍需依當前 Codex 版本與團隊權限政策調整。
