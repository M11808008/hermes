Threads 觀察：Obsidian × Claude Code，把旅行足跡變成可互動地圖 Atlas

這則 Threads 分享的是一個很有代表性的「個人知識庫 + AI Coding Agent」工作流：使用者把多年旅行紀錄放在 Obsidian，接著交給 Claude Code 讀取、整理、抽取地點資訊，最後生成可瀏覽的 Travel Atlas 互動頁面。

它的重點不只是「做一張地圖」，而是示範了 AI 如何把原本散落在筆記中的生活資料，轉換成結構化、可視化、可再次探索的個人資料產品。

為什麼這個案例重要？

### 1. Obsidian 從筆記庫變成資料庫

Obsidian 的優勢在於 Markdown 檔案本地儲存、可版本化、可被其他工具讀取。當 Claude Code 能直接讀取資料夾後，筆記就不只是「人看的文字」，也成為 AI 可以加工的資料來源。

這種模式讓個人知識庫具備三層用途：

  • 記錄：把經驗、地點、時間、心得留下來。
  • 結構化：由 AI 抽取城市、國家、年份、分類。
  • 產品化：生成網頁、地圖、儀表板或搜尋介面。

### 2. Claude Code 適合做「本地資料轉網站」

Claude Code 的價值在這類任務特別明顯：它不只是聊天式回答，而是能在本地專案中讀檔、建檔、修改程式、生成頁面。對於已經有 Obsidian Vault 的使用者來說，它可以扮演「個人資料工程師」。

可延伸應用包括：

  • 閱讀筆記 → 生成主題索引。
  • 旅行筆記 → 生成互動地圖。
  • 課程筆記 → 生成學習路線圖。
  • 會議紀錄 → 生成決策追蹤儀表板。
  • 書摘筆記 → 生成概念網絡圖。

### 3. 個人資料產品化正在變得平民化

過去要做這種 Travel Atlas，通常需要會寫爬蟲、資料清洗、地圖 API、前端框架。現在流程可簡化為：

  • 平常照常寫筆記。
  • 給 AI Coding Agent 一個目標。
  • 讓它讀取資料、設計 schema、生成頁面。
  • 人類負責校對與美感調整。

這是「個人知識管理」轉向「個人資料產品」的典型案例。

可借鏡的 Prompt / 工作流

如果要複製這個模式，可以給 Claude Code 類似任務：

```text

請讀取我的 Obsidian Vault 中所有旅行相關筆記，抽取每趟旅行的年份、城市、國家、區域、日期、摘要與代表照片線索，整理成 JSON 檔，並建立一個可互動的 Travel Atlas 網頁。頁面需要包含世界地圖、年份篩選、區域統計、旅行卡片與每趟旅行的摘要。請不要修改原始筆記，只新增輸出資料與網站檔案。

```

進階版本可以要求：

  • 不修改原始筆記，只輸出衍生資料。
  • 所有抽取結果需要附來源檔案路徑。
  • 不確定的地點標記為 `needs-review`。
  • 地圖資料與展示頁分離，方便日後更新。
  • 提供一鍵重新產生腳本。

風險與注意事項

  • 地點辨識可能出錯：例如同名城市、城市與景點混淆。
  • 年份與日期要避免從照片檔名或筆記標題誤判。
  • 若包含私人行程、住址或同行者姓名，公開部署前要先脫敏。
  • AI 生成的地理座標、國家分類需人工抽查。
  • 不建議讓 Agent 直接改原始 Obsidian 筆記；衍生資料應另存。
