一句話摘要

這支 Reel 的核心不是單一工具，而是一條「AI 產品開發管線」：先讓 AI 把想法轉成 PRD 與設計/技術規格，再用 v0 / Lovable / Bolt.new 這類 AI builder 產生可運行網站，最後接 GitHub、code review、code scanning 與部署流程，讓創業原型或公司網站快速從概念走到上線。

AI 建站流程拆解

| 階段 | 產物 | AI 可以做什麼 | 檢查重點 |

|---|---|---|---|

| 1. 想法整理 | Problem statement / one-liner | 把粗略想法轉成目標用戶、痛點、價值主張 | 不要只寫功能；先定義誰會用、為什麼需要 |

| 2. PRD | Product Requirements Document | 產生功能列表、使用者故事、驗收標準、限制條件 | PRD 要有優先級、非功能需求、例外情境 |

| 3. 設計文件 | UX flow / wireframe / UI spec | 產生頁面架構、互動流程、文案、component list | 檢查手機版、空狀態、錯誤狀態、loading 狀態 |

| 4. 技術架構 | Tech spec / data model / API plan | 建議框架、資料庫、API、部署方式、權限模型 | 避免 AI 過度設計；先求可交付 MVP |

| 5. AI 開發 | 前端/後端程式碼 | 用 v0、Lovable、Bolt.new、Cursor、Claude Code 等生成與迭代 | 每次改動要小步 commit，可跑起來才算完成 |

| 6. 程式碼掃描 | Code review / CodeQL / secret scanning | 自動檢查安全、品質、lint、型別、依賴風險 | AI 生成碼很容易藏憑證、權限漏洞、未處理錯誤 |

| 7. Git 整合 | Repo / branch / PR | 推到 GitHub、產生 PR、用 Copilot code review 或 CI 檢查 | 保留 diff、review comments、rollback 方案 |

| 8. 部署 | Preview / production URL | Vercel/Netlify/Cloudflare Pages 等快速部署 | 設定環境變數、網域、監控與回復策略 |

可用工具定位

v0 by Vercel

v0 的官方定位是用 AI 建 full-stack web apps，搜尋結果與官方頁都強調可連接 GitHub、推送程式碼到 repository，並與 Vercel 的 preview/deploy 流程高度整合。

適合：

  • React / Next.js / Tailwind 類型網站
  • Landing page、dashboard、SaaS prototype
  • 想快速取得高質感 UI 並部署到 Vercel

Lovable

Lovable 是 AI app builder，官方文件明確提供 GitHub integration。連到 GitHub 後可備份程式碼、協作、使用 pull requests、branches 與 code reviews。

適合：

  • 快速產出完整 app 原型
  • 非工程背景產品人想先做 MVP
  • 需要把 AI 生成結果交給工程師接手維護

Bolt.new

Bolt.new 強調瀏覽器內 full-stack 開發，結合 AI 與 StackBlitz WebContainers，可 prompt、run、edit、deploy full-stack web applications。它適合快速跑起 Node/前端專案，而不是只有靜態 UI 圖。

適合：

  • 要立刻在瀏覽器內跑起前後端 prototype
  • 想用 prompt 迭代功能並即時測試
  • 要教學、demo、hackathon 快速交付

GitHub Copilot code review / code scanning

GitHub 官方文件指出 Copilot code review 可對 PR 中的程式碼變更提供回饋；code scanning / Copilot Autofix 則可協助處理安全警示，但官方也提醒 AI 建議需要人工審查與測試。

適合：

  • PR 自動 review
  • 找出明顯 bug、品質問題、安全風險
  • 搭配 CodeQL、secret scanning、CI 測試建立最低安全網

欒帥可用的高品質建站 SOP

Step 1：先寫一頁式產品簡報

```text

產品名稱：

目標使用者：

使用情境：

核心痛點：

三個最重要功能：

不做什麼：

成功指標：

```

Step 2：請 AI 產 PRD

Prompt 範例：

```text

請把以下產品想法整理成可交給 AI builder 和工程師使用的 PRD。

請包含：目標用戶、核心 user stories、MVP 範圍、功能優先級、資料模型、頁面列表、驗收標準、非功能需求、風險與不做項目。

```

Step 3：把 PRD 轉成設計與技術規格

輸出至少要有：

  • sitemap
  • user flow
  • component list
  • page-by-page UI spec
  • data schema
  • API routes
  • auth / permission rules
  • deployment target

Step 4：交給 AI builder 產第一版

根據需求選工具：

| 需求 | 優先工具 |

|---|---|

| 高質感前端與 Vercel 部署 | v0 |

| 快速完整 app / no-code MVP | Lovable |

| 需要瀏覽器內跑 full-stack | Bolt.new |

| 要精修既有 repo | Cursor / Claude Code / Codex |

Step 5：GitHub 化

不要只留在 AI builder 裡，至少要做到：

  • 建 GitHub repo。
  • 每個功能分 branch。
  • 透過 PR 合併。
  • 啟用基本 CI / lint / typecheck。
  • 啟用 secret scanning / dependency alerts / code scanning。
  • 部署 preview URL。

Step 6：上線前檢查

| 類別 | 檢查 |

|---|---|

| 功能 | 核心流程可完整跑完 |

| RWD | 手機、平板、桌面都能用 |

| 表單 | 必填、錯誤訊息、防重送 |

| 安全 | 沒有 hardcoded secrets、API key 不暴露前端 |

| SEO | title、description、OG image |

| 效能 | 首屏載入、圖片壓縮 |

| 分析 | GA / Plausible / Vercel Analytics |

| 回復 | 有 rollback / previous deployment |

關鍵判斷：AI 建站不是省掉流程，而是壓縮流程

這支 Reel 容易被誤解成「AI 直接幫你把網站做完」。更精準的理解是：AI 把原本跨產品、設計、工程、QA、DevOps 的流程壓縮到一個可快速迭代的工作台，但每一層仍需要驗證。

最常見的失敗點：

  • 沒有 PRD，直接 prompt 生成，導致功能散亂。
  • AI 生成漂亮 UI，但資料模型與權限設計不可用。
  • 沒接 GitHub，版本不可追蹤。
  • 沒有 code scanning，API key 或權限漏洞外洩。
  • 沒有測試，demo 可以跑，上線就壞。

欒帥可直接複用的工作流模板

```text

Idea → PRD → UX Flow → UI Spec → Tech Spec → AI Builder Draft → Local/Preview Test → GitHub PR → Code Review/Scanning → Vercel Preview → Production Deploy → Analytics/Feedback → Iteration

```

適合使用情境

| 情境 | 適合程度 | 理由 |

|---|---|---|

| 課程 landing page | 高 | 內容與 UI 可快速生成，部署簡單 |

| 活動報名頁 | 高 | 表單、名單、通知可快速串接，但注意個資與權限 |

| SaaS MVP | 中高 | 可先做 prototype，正式上線需補安全/測試/架構 |

| 公司官網 | 高 | PRD + design spec + v0/Vercel 很適合 |

| 複雜企業系統 | 中低 | AI 可輔助規格與局部開發，但需完整工程治理 |

| 金流/醫療/高風險系統 | 低 | 不能只靠 AI builder，需嚴格審計與合規 |

判讀限制

  • Reel 內容偏概念介紹，未指定單一工具或實際 demo 細節；本文以官方工具頁補充形成可操作流程。
  • v0、Lovable、Bolt.new、GitHub Copilot 等工具功能與價格會快速變動，實作前需看最新官方文件。
  • AI 生成程式碼仍需人工 review、測試與資安檢查，尤其是 auth、資料庫規則、API key、付款與個資。
  • 產品/公司網站若要正式公開，仍需檢查法律條款、隱私權政策、Cookie、SEO、無障礙與品牌一致性。
