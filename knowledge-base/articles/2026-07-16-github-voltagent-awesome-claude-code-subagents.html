  • Claude Code 官方 subagents 文件說明，自訂 subagents 是專門任務用的 AI assistants，可放在專案層 `.claude/agents/` 或使用者層 `~/.claude/agents/`。

對 AI 學習雷達的保存價值

這個 repo 可作為「子代理設計模式」資料庫，而不是只當現成安裝包：

  • 看分類：哪些任務值得拆成專門 agent，哪些應留給主 agent。
  • 看命名：agent name 與 description 如何影響自動委派。
  • 看工具配置：不同角色應該開哪些工具、禁哪些工具。
  • 看模型路由：README 提到用 `opus` / `sonnet` / `haiku` 等模型欄位平衡品質與成本，這對大型 agent workflow 的資源治理有參考價值。
  • 看編排代理：Meta & Orchestration 類別可用來研究多代理協作、安裝代理與 catalog 搜尋的工作流。

對欒帥的 Hermes / Claude Code / Codex 工作流來說，這份清單可當作「角色拆分靈感庫」：把通用助理拆成 code reviewer、security auditor、data analyst、documentation engineer、devops engineer 等專門角色，再依專案安全邊界給工具與上下文。

---

導入注意事項

  • 不要一次安裝全部 subagents；先挑 3–5 個高頻角色試用，觀察是否真的降低主 agent 負擔。
  • 社群 agent 可能包含過度授權、過時工具、產品導流或不適合本機環境的指令，安裝前應逐一讀 Markdown frontmatter 與 system prompt。
  • Project subagents 通常比 global subagents 更安全，因為可把權限限制在特定 repo。
  • 若 agent 會執行 shell、讀寫檔案或連外，應先檢查工具清單與資料外洩風險。
  • Claude Code 的 subagents / plugins 行為會隨版本變動；正式導入前需用 `claude` CLI 與官方 docs 驗證目前語法。

---

可行的下一步

若要把這份 repo 轉成實際可用工作流，建議先做小規模評估：

  • 選 1 個專案 repo。
  • 只安裝 Core Development、Quality & Security、Developer Experience 中最常用的少數代理。
  • 設計同一組任務，用「無 subagent」與「有 subagent」各跑一次。
  • 比較輸出品質、上下文污染、耗時、工具權限與人工覆核成本。
  • 把有效代理改寫成自家 `.claude/agents/` 或 Hermes skill / subagent workflow，而不是盲目沿用全部模板。
