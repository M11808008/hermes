一句話摘要

這則長文把 Codex 的新版定位解讀為「從 coding agent 走向跨職能工作台」：Work 負責研究、分析與內容交付；Codex 負責程式碼、可執行任務與驗證；Sites、Plugins、Browser、Visualize 與 Ultra Mode 則把它往「資料到成果、成果到互動介面、多代理並行」推進。但其中部分說法仍是作者實測與產品觀察，需以官方文件、方案與桌面版功能可用性為準。

X Article 目錄與主張

作者列出六個重點：

  • Work 到底是什麼？和 Codex 有什麼區別？
  • 幾個新增 plugins：Sites、Template Creator、Visualize、Default templates。
  • 內建 Browser 更新：把 Chrome / browser 工作流裝進 Codex。
  • 前端設計能力加強。
  • Ultra mode：多 agent 並行執行任務。
  • 將 GPT 網頁端聊天接入 Codex，適合做腦力激盪。

欒帥可用場景

  • 課程示範：用 Work / Codex 分工說明「研究交付物」與「可驗證程式變更」的差異。
  • 企業導入：先盤點職能插件與資料權限，再讓 Codex 接上 CRM、文件、簡報或 dashboard，不要一開始就全開。
  • 教學工作流：ChatGPT Work 先做教材架構與腦力激盪；Codex / Sites 把教材轉成互動頁、demo 或工具。
  • Hermes 對照：這其實和 Hermes 的 skills、browser、cron、subagents、deployment verification 是同一類 harness engineering 思維。

風險與注意事項

  • 「Work」「Codex」「Plugins」「Sites」的可用性受 ChatGPT plan、桌面版版本、workspace policy、地區與 rollout 影響。
  • Sites 是 beta / preview 類能力時，適合 demo、內部工具與輕量 app；正式 production 仍需安全、域名、監控、資料保存與備援設計。
  • Browser / Chrome / cookie / password 匯入屬於高敏感權限，教學中應使用測試帳號或 sandbox，不要接真實個資與金流。
  • Ultra / 多代理工作流成本高，適合複雜任務；普通工作先用 high 或單 agent，避免 token 快速消耗。
  • 社群長文中的「實測很夯」「直接替代」屬作者語氣，歸檔時應保留方向但加上官方可用性與安全 caveat。

Why it matters

這則 X Article 和前一篇 BusinessNext loop engineering 可以連成一條線：AI agent 的下一階段不是更會聊天，而是能在 Work / Codex / Plugins / Sites / Browser / Multi-Agent 之間形成可驗證的工作台。對 AI 學習雷達而言，它是「Codex 從程式工具走向跨職能執行平台」的重要觀察點。
