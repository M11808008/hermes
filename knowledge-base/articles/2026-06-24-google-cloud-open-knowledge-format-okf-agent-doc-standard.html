Google Cloud OKF｜用 Markdown + YAML 統一 Agent 知識交換格式

李宗倫在 Generative AI 技術交流中心分享 Google Cloud 的 Open Knowledge Format（OKF），指出它試圖統一 Agent 知識交換與文檔格式。短連結 `goo.gle/4uGvAEe` 指向 GoogleCloudPlatform `knowledge-catalog` repository 的 `okf` 目錄，核心規格文件是 `okf/SPEC.md`。

一句話摘要

OKF 的重點不是創造複雜新格式，而是把「Markdown + YAML frontmatter + 目錄索引 + 引用慣例」正式規格化，讓人類、agent、資料目錄、RAG 管線與靜態網站可以共用同一份知識 bundle。

OKF 是什麼

官方規格定義 OKF 是一種 open、human-friendly、agent-friendly 的知識表示格式，用來描述圍繞資料與系統的 metadata、context 與 curated insight。它可以由人手寫、由 agent 生成、跨組織交換，也可以由人類與 agent 共同消費。

格式刻意保持極簡：

  • 一個 knowledge bundle 是一個目錄樹。
  • 每個 concept 是一個 UTF-8 Markdown 檔案。
  • 每個非保留 Markdown 檔案都應有 YAML frontmatter。
  • 唯一必填欄位是 `type`。
  • `index.md` 與 `log.md` 是保留檔名，用於漸進式導覽與更新記錄。
  • Markdown link 用來表示概念之間的關係。
  • `# Citations` 可列出外部引用來源。

最小結構

OKF bundle 的典型結構如下：

```text

path/to/bundle/

├── index.md

├── log.md

├── <concept>.md

└── <subdirectory>/

├── index.md

└── <concept>.md

```

每個 concept document 的 frontmatter 範例如下：

```yaml

---

type: BigQuery Table

title: Customers

description: Customer master table for retail analytics.

resource: https://console.cloud.google.com/bigquery?p=...

tags: [customer, analytics]

timestamp: 2026-05-28T00:00:00Z

---

```

其中只有 `type` 是必填；`title`、`description`、`resource`、`tags`、`timestamp` 是建議欄位。生產者可加入額外 key，消費者應容忍未知欄位。

OKF 解決的問題

| 問題 | OKF 的回答 |

|---|---|

| Agent 讀不到企業脈絡 | 把資料、API、指標、runbook、業務規則整理成 Markdown concept |

| 不同工具輸出格式不一致 | 用少量共同規則建立可交換 bundle |

| 知識被鎖在平台裡 | bundle 是目錄，可放 git、tarball、repo 或靜態檔案伺服器 |

| LLM ingestion 需要轉換 | Markdown 可直接放入 context；frontmatter 供檢索與篩選 |

| 人類與 agent 維護兩套文件 | 同一份文件同時給人看、給 agent 讀 |

| 版本與審查困難 | git diff、PR、blame、review workflow 直接可用 |

官方設計原則

規格明確主張 OKF 應具備以下特性：

  • Readable：人類不靠工具即可閱讀。
  • Parseable：agent 不需要特定 SDK 即可解析。
  • Diffable：版本控制中可逐行比較。
  • Portable：可跨工具、組織與時間保存。
  • Minimally opinionated：只標準化必要結構，不規定領域分類法。

這也是社群貼文中提到的三個亮點：零工具依賴、LLM 友好、雙向可讀。

與現有格式的關係

OKF 規格指出，它刻意接近幾個既有模式：

  • LLM wiki repositories：Markdown + frontmatter 作為 agent-readable knowledge base。
  • Obsidian / Notion 類個人知識工具：階層式文件與交叉連結。
  • Metadata as code：把資料目錄與系統脈絡放進版本控制，而不是只存在某個 SaaS registry。

差異在於 OKF 嘗試把這些鬆散慣例寫成 specification，讓不同 producer / consumer 可以最低限度互通。

對 AI Agent 工作流的意義

OKF 可以把 agent workflow 分為兩段：

1. Production：Agent 產生知識 bundle

Agent 可從 Query、API、資料庫 metadata、文件、log、schema 或企業資料目錄抓取資訊，轉換為 OKF concept：

  • table / dataset 說明
  • API endpoint 文件
  • metric 定義
  • incident runbook
  • business process
  • domain glossary
  • schema 與 example queries
  • citations 與來源追蹤

2. Consumption：Agent 或人類消費知識 bundle

消費端不需要特殊 SDK：

  • 人類可直接 `cat`、`grep`、瀏覽 Markdown。
  • LLM 可把 relevant concept 直接載入 context。
  • 搜尋引擎可索引 frontmatter 與 body。
  • 靜態網站可渲染成可視化知識圖。
  • agent 可沿著 `index.md` 與 Markdown links 逐步探索，而不是一次塞入整包資料。

對欒帥 AI 學習雷達的實用判讀

1. 欒帥的知識庫其實已經接近 OKF 思路

目前 AI 學習雷達文章已採用 Markdown + YAML frontmatter，並有 metadata index、tag、週報與靜態網站輸出。這和 OKF 的基本精神高度一致：同一份內容同時可給人閱讀、給 agent 搜尋、給網站渲染。

可考慮的對齊方向：

  • 在特定子目錄試做 OKF-compatible bundle。
  • 為文章 frontmatter 補入 `type` 欄位，例如 `type: AI Tool Note`、`type: Workflow`、`type: Research Brief`。
  • 將週報、工具、prompt、課程、案例拆成更明確的 concept 類型。
  • 將引用來源集中到 `# Citations`，提升可驗證性。
  • 產生 `index.md` 給 agent 做 progressive disclosure。

2. OKF 可作為課程中的「Agent 記憶格式」教學案例

很多人談 agent memory 時會直接跳到向量資料庫，但 OKF 提醒另一條路：先用最穩定、最可審查的 file-based knowledge。這很適合課程示範：

  • 用 Markdown 寫一份 domain concept。
  • 用 YAML frontmatter 標示 type、tags、resource。
  • 讓 agent 讀取 bundle 並回答問題。
  • 修改 concept 後用 git diff 追蹤知識演化。
  • 再視需求加入 search index、RAG 或 graph visualizer。

3. 需注意 OKF 仍是 v0.1 draft

OKF 目前是 v0.1 draft，應視為值得追蹤的開放規格，而不是已成熟的企業標準。導入時建議採取「兼容而不綁死」策略：

  • 可以採用 `type`、`index.md`、`log.md`、`# Citations` 等低風險慣例。
  • 不要為了完全符合 OKF 而破壞既有知識庫欄位。
  • 保持 frontmatter extensible，讓本地欄位與 OKF 欄位並存。
  • 等 Google Cloud / 社群後續版本穩定後，再決定是否完整遷移。

收錄判斷

OKF 是高優先級追蹤項目，因為它把 agent-readable knowledge、metadata as code、Markdown knowledge base 與企業資料目錄接在一起。若未來被更多 agent framework、資料目錄或 RAG 工具採納，它可能成為「agent 知識包」的低摩擦交換格式。
