一句話摘要

Fable 這類更強模型的提示策略，不只是把規格寫更死，而是用更多脈絡、更少限制、主動探索未知，並用報告與測驗讓人類始終保有判斷權。

貼文主軸：

  • Fable 代表更強、更開放的模型能力，應該重新理解提示與協作方式。
  • 不要只給一堆 constraints，而要提供更多 context。
  • 用 Unknown Matrix 找出需求、設計、架構與盲點中的未知。
  • AI 可以協助探索人類尚未想像到的方向，但人類必須 stay in the loop。
  • 讓 AI 在大改動後產生 HTML report、implementation notes 與 quiz，確保人類理解變更。

Thariq 的四段框架

YouTube transcript 中，Thariq 把 Field Guide to Fable 分成四段：

  • Unhobbling Claude：解除模型被工具與提示框架限制的狀態。
  • Finding your unknowns：找出自己沒有寫進 prompt、甚至沒有意識到的未知。
  • Dealing with the grief：面對模型能力上升後，過去做事方式被改寫的落差。
  • Being unreasonable：重新想像 good / fast / cheap 的權衡，讓自己更有企圖心。

他用「Pokémon 名稱結尾 aw」例子說明 capability overhang：聊天模型可能答不出來，但 Claude Code 若能抓資料、寫 script 過濾，就能解出。重點不是模型「記得更多」，而是工具與工作方式讓能力被釋放。

Context > Constraints

貼文與 transcript 都指向同一個觀念：新模型不一定需要更大的限制網，反而需要更好的脈絡。

Thariq 提到：

  • models are grown, not designed。
  • prompt / harness 其實反映了人類對模型的理解。
  • 新一代模型更有想像力，過多 examples 或「不要做 X」可能反而限制它。
  • 與其把模型綁死，不如給它更完整的 context，讓它探索可行路徑。

這並不代表不設邊界，而是把「限制」轉成「任務脈絡、目標、評估標準、已知資源與人類偏好」。

Unknown Matrix：四種未知與對應提示

貼文整理了 known / unknown 四象限，可保存為 prompt 設計框架：

Known Knowns：已知已知

你已經清楚知道的需求，可以直接寫進 prompt。

例：用 Python 寫一個排序演算法。

Unknown Knowns：未知已知

你其實有偏好，但無法用文字描述；常見於視覺、美感、互動設計。

建議方式：brainstorms and prototypes。

貼文範例：

  • “I want a dashboard for this data but I have no visual taste and don't know what's possible. Make me an HTML page with 4 wildly different design directions so I can react to them.”

重點：不是要求 AI 猜中你的美感，而是先產生多個極端原型，讓人類透過反應把隱性偏好顯性化。

Known Unknowns：已知未知

你知道自己有些地方不清楚，例如資料庫架構、需求邊界、使用情境。

建議方式：interviews / references。

貼文範例：

  • “Interview me one question at a time about anything ambiguous — prioritize questions where my answer would change the architecture.”
  • “This Rust crate in vendor/rate-limiter implements the exact backoff behavior I want. Read it and reimplement the same semantics in our TypeScript API client.”

重點：讓 AI 優先問會改變架構的問題；或把既有程式 / 文件當作「舊地圖」，讓模型畫出新地圖。

Unknown Unknowns：未知未知

你不知道自己不知道的盲點，是最危險區域。

建議方式：blindspot pass。

貼文範例：

  • “I'm working on adding a new auth provider but I know nothing about the auth modules in this codebase. Can you do a blindspot pass to help me figure out my relevant unknown unknowns and help me prompt you better?”

重點：不要只叫 AI 找 bug，而是請它找出你還沒意識到的架構盲點、風險與接下來應該如何提問。

Stay in the Loop：人類仍是決策核心

Fable 能探索未知，不代表可以放手讓模型決定一切。貼文中特別保存了「保持人類在迴圈中」的操作方法：

  • 大改動後請模型產生 HTML report。
  • report 應包含 context、intuition、what was done、影響範圍。
  • 底部加入 quiz，讓人類確認自己真的理解。

貼文範例：

  • “I want to make sure I understand everything that's happened in this change. Give me a HTML report on the changes for me to read and understand with context, intuition, what was done, etc. and a quiz at the bottom on the changes that I must pass.”

Transcript 也提到 implementation notes：當 Fable 遇到未知或偏離原計畫時，要求它記錄下來，讓人知道 deviation 發生在哪裡、為什麼發生。

「規格派」不會消失，但要升級

貼文提出「規格派會式微」的觀點：模型越強，越不應只把它關在非常細的指令裡。更準確地說，規格派不是消失，而是升級成：

  • 用 context 描述世界，而不是只列規則。
  • 用 prototype 暴露隱性需求。
  • 用 interview 釐清架構關鍵問題。
  • 用 references 讓模型讀既有範例。
  • 用 blindspot pass 找未知風險。
  • 用 report / quiz 保留人類理解與決策權。

這比傳統「把每一步都規定死」更適合高能力 agentic model。

Pick Three：重新想像 good / fast / cheap

Thariq 在 transcript 中提到，Fable 改變了對 trade-off 的想像：過去常說 good / fast / cheap 只能選二，現在應該更常思考「pick three」。

但貼文也提醒：建立東西變簡單，不等於創造價值變簡單。真正困難的仍是：

  • 找到值得做的問題。
  • 讓產出對人或組織產生價值。
  • 讓人類保有理解、判斷與責任。

對 AI 學習雷達的價值

這篇可作為「高能力模型提示策略」課程單元：

  • 從 constraints prompt 轉向 context-rich collaboration。
  • 用 Unknown Matrix 設計不同類型的提示。
  • 把 prompt 從一次性指令升級成互動式需求發現流程。
  • 讓 AI agent 在大改動後自動輸出 report、implementation notes、quiz。
  • 把人類在迴圈中具體化，而不是只喊口號。

  • 多方向原型

  • 架構訪談
