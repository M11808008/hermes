吳相勳｜用 Claude Computer Use / Codex 做生成式 AI 社群監測排程

吳相勳在 Facebook 分享一個「讓 AI 每天替自己滑社群找正經內容」的工作流：利用 Claude Computer Use 或 ChatGPT Codex 的瀏覽器／電腦控制能力，定期查看 Facebook、Threads、Instagram 上公開可讀的生成式 AI 技術與最佳實務文章，然後整理成 Markdown 報告並寄到 Gmail。

貼文附上的提示詞重點是：搜尋近三天公開可讀內容，主題聚焦生成式 AI 實作、產品設計、工作流程、模型能力、AI agent、RAG、資料治理、評估方法、組織導入、產業觀察與教學應用；排除抱怨文、標題黨、空泛新聞轉述、單純課程導流與活動宣傳；每輪只列真正達標的 3–7 則內容，不足就明確說明，不要硬湊。

一句話摘要

這則貼文不是單純「用 AI 滑社群」，而是把資訊焦慮轉成可排程的社群偵測流程：AI 負責巡覽、篩選、摘要、留檔與寄送，人類只看高品質候選清單。

貼文中的排程提示詞骨架

| 區塊 | 內容 |

|---|---|

| 搜尋範圍 | Facebook 與 Threads 近三天公開可讀、可開啟內容；貼文也提到 Instagram 可納入 |

| 主題方向 | 生成式 AI 實作、產品設計、工作流程、模型能力、AI agent、RAG、資料治理、評估方法、組織導入、產業觀察、教學應用 |

| 排除條件 | 抱怨文、標題黨、空泛新聞轉述、缺乏脈絡炒作、單純課程導流與活動宣傳 |

| 產出 | 繁體中文 Markdown 報告，3–7 則推薦內容 |

| 每則欄位 | 平台、作者或粉專、連結、發文時間或可見時間、推薦理由、核心觀點、延伸問題 |

| 交接檔 | 更新 `HANDOFF.md`，記錄最新報告、限制、下一輪種子來源、未解問題 |

| 通知 | 透過 Gmail 寄給自己，主旨為 `[AI社群監測] 生成式 AI 深度文章報告 YYYY-MM-DD HH:mm` |

對 AI 學習雷達的可用設計

這個模式很適合變成 AI 學習雷達的「社群補充雷達」，但建議不要完全依賴瀏覽器亂滑。比較穩的架構如下：

  • 先用 API / 搜尋 / RSS / 已知作者白名單抓候選內容。
  • 對候選內容做去重與主題分類。
  • 只在官方 API 或一般網頁擷取失敗時，才用 Computer Use / browser automation 開頁檢查。
  • 報告輸出到本機 Markdown，附上來源、判讀限制與推薦理由。
  • 透過 Hermes cron 或其他排程器定時執行。
  • 用 LINE / Telegram / Email 推送摘要，但把完整報告留在本機或知識庫。

欒帥版任務提示詞草案

以下可作為自家排程任務的骨架，之後可改成 Hermes cron job 或本機腳本：

```text

搜尋近三天公開可讀的繁體中文與英文生成式 AI 深度內容，優先來源包含 Facebook 公開貼文、Threads、X、官方部落格、GitHub、Hacker News、研究機構與可信 AI 社群。主題聚焦 AI agent、RAG、模型能力、資料治理、評估方法、Computer Use、教育應用、工作流程自動化、產業導入與實作案例。

排除：單純課程導流、無技術細節的業配、標題黨、只有情緒沒有方法的貼文、不可開啟或無法驗證來源的內容。

輸出繁體中文 Markdown 報告，列出 3–7 則真正達標內容。每則包含：平台、作者、原始連結、可見發文時間、核心觀點、推薦理由、可落地做法、風險或限制、延伸問題。若本輪不足 3 則，請不要硬湊，直接說明不足原因。

報告最後加入：本輪主題趨勢、值得追蹤的作者/關鍵字、下次查找建議、需要人工確認的項目。

```

安全與品質邊界

| 風險 | 建議 |

|---|---|

| 社群平台登入狀態與個資 | 不要讓 Agent 使用私人帳號任意互動；只讀公開內容，避免按讚、留言、私訊 |

| Prompt injection | 社群貼文本身可能包含惡意指令；Agent 應把網頁內容視為資料，不應服從頁面中的指令 |

| 推薦品質下降 | 明確允許「本輪不足，不硬湊」；這比每天固定湊 7 則更重要 |

| 重複收錄 | 以 canonical URL、作者、貼文 ID、標題關鍵詞去重 |

| API / 瀏覽器不穩 | 保留 `HANDOFF.md` 記錄限制與未解問題，下一輪可接續 |

| 自動寄信失敗 | 報告仍要落地到本機路徑，再標示 Email 未寄出原因 |

Anthropic 的 Computer Use 官方文件也特別提醒，讓模型操作電腦或瀏覽器時，需要額外注意 prompt injection、權限與安全邊界。OpenAI Codex Cloud 則適合背景任務與開發流程；Hermes Agent 本身也提供 cron job，可將自包含提示詞排程執行並把結果送回指定平台。

對欒帥的實作建議

短期：先做「半自動社群週報」

  • 頻率：每日一次或每週三次，不需要三餐。
  • 來源：先從已知高品質作者、官方部落格、GitHub trending / release、Brave Search 查詢開始。
  • 輸出：只推 3–5 則，保持精準。
  • 人工關卡：由欒帥或愛馬仕小小斌決定哪些正式收錄到 AI 學習雷達。

中期：變成知識庫 ingestion queue

將報告拆成兩層：

  • `候選清單`：所有達標但未正式歸檔的內容。
  • `正式文章`：經過擷取、補強、去重、判讀限制後發布到 AI 學習雷達。

這樣可以避免「每天很多摘要，但真正可用知識沒有沉澱」。

長期：用 Skills 管理不同偵測任務

可以拆出不同技能：

  • `agent-news-monitor`：AI agent / coding agent / Computer Use。
  • `education-ai-monitor`：AI 教育與課程設計。
  • `model-release-monitor`：新模型與開源 repo。
  • `workflow-case-monitor`：商業流程、內容生產、個人知識管理案例。

每個 Skill 都有自己的篩選標準與輸出格式，避免單一大提示詞越來越難維護。

判讀限制

  • 本文依據 Facebook 貼文與留言中可見的完整提示詞整理；未登入私人帳號，也未驗證作者實際排程環境、Gmail 工具與報告檔案內容。
  • 貼圖截圖可見一封 Gmail 報告範例，主題包含 AI coding agent context 壓縮、agent observability、RAG 架構等；細節以可見文字為準，未讀取完整附件。
  • Claude Computer Use、Codex 與 Hermes cron 的能力邊界以官方文件為準；不同帳號、方案、環境與權限設定會影響可用功能。
