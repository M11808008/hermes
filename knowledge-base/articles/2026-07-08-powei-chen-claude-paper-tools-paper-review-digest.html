一句話摘要

這是一個把臨床 journal club 的讀論文流程封裝成 Claude Code skills 的案例：讓模型負責讀懂語意，讓小程式負責不可妥協的邏輯、引用與證據等級檢查。

  • paper-review：回答「這篇可不可信、做得好不好？」
  • paper-digest：回答「這篇講了什麼、怎麼最快吸收？」

貼文中特別強調，這比較像個人系統的一個案例，不是下載後就能無腦跑的產品；但可作為把臨床評讀、EBM 與 AI agent 工作流結合的參考實作。

paper-review：可信度與評讀

paper-review 不是單純叫 AI 寫一篇心得，而是把 journal club 會問的問題拆成一套流程：

  • 期刊可信度：impact metrics、indexing、peer review、predatory / retraction red flags。
  • 作者與背景：作者 track record、COI、相關研究脈絡。
  • 引用查核：先用 CrossRef 檢查引用文獻是否真的存在，再做語意層面的引用是否被正確表述。
  • 研究設計路由：依研究型態選擇適合工具，而不是一張萬用 checklist。
  • 論證與 spin 稽核：檢查結論是否超出資料能支持的範圍。
  • GRADE：把模型的證據等級判斷交給程式重算。

研究設計路由：不是萬用 checklist

README 與 skill 文件列出不同研究型態對應工具：

  • RCT：Cochrane RoB 2
  • 世代 / 病例對照 / 橫斷：ROBINS-I / Newcastle-Ottawa
  • 系統性回顧 / 統合分析：AMSTAR-2 + per-outcome GRADE，另檢查 PRISMA 2020 報告完整性
  • 診斷準確度：QUADAS-2
  • 預測模型：PROBAST
  • 臨床指引：AGREE II
  • 敘事型回顧：SANRA + cherry-picking check
  • 病例系列 / case report：JBI checklist

這對 AI 學習雷達很重要：AI 不是只要「看起來有分析」，而是應該先判斷任務類型，再套用對應工具。

確定性 gate：避免 AI 憑印象唬爛

最值得保存的架構是「模型與程式分工」：

  • LLM 做語意判斷，例如辨識研究結論、前提類型、GRADE 五個 domain 的嚴重程度。
  • Python 程式做確定性計算與規則檢查。

兩支代表性程式：

  • grade_judge.py

- 模型只評 risk of bias、inconsistency、indirectness、imprecision、publication bias。

- 程式根據起始等級與升降級規則重算最終 GRADE。

- 模型自報等級只當參考，與程式不一致時要回頭檢查。

- 可擋掉模型容易記錯的規則，例如觀察性研究只要有降級就不能再升級。

  • argdown_lint.py

- 模型標註每個 premise 類型，例如 direct RCT、association、surrogate outcome、single study、subgroup、secondary outcome。

- 程式檢查不合法推論跳躍，例如：

- 替代指標改善 → 宣稱 hard endpoint / causal benefit

- 相關性 → 宣稱因果

- 單一研究 → 宣稱「一致地顯示」

- 次族群或次要 outcome → 過度宣稱療效

這種設計可直接遷移到其他 AI 工作流：任何「看似主觀但其實有規則」的任務，都可以讓 AI 做標註、讓程式做 gate。

CrossRef gate：先確認引用真的存在

paper-review 的引用查核有一個便宜但高價值的第一關：

  • 對 DOI 呼叫 CrossRef API。
  • 404 或無 message 就標示為 fabricated / unresolvable DOI。
  • 若 DOI 可解析，再用標題 token overlap 檢查是否 DOI 對到不同論文。
  • CrossRef 通過只代表「存在」，不代表引用語意正確；仍需進入語意查核。

這個設計特別適合防止 AI 產生假 DOI、張冠李戴引用，或把存在的文獻講成它沒有說過的內容。

paper-digest：內容吸收與主動回憶

paper-digest 完全不評好壞，只做「怎麼最快把全文內容吸收進腦袋」。它採三層漸進揭露：

  • 30 秒層：一句話重點 + 3–5 個最高價值事實。
  • 5 分鐘層：每一節開頭都有要點摘要。
  • 完整層：依 paper type 重新組織內容與細節。

它還要求生成自我測驗卡，目標是把整理導向主動回憶，而不是只產生漂亮筆記。

重要硬規則：拿不到全文就停，不准只靠 abstract 生成一份看起來完整的內容整理。這點非常值得保留，因為「摘要型幻覺」最容易讓人誤以為自己已經讀懂全文。

對 AI 學習雷達的價值

這篇與近期 Agent Skills / Codex / Claude Code 主題形成連續脈絡：

  • Skills 的價值不是把 prompt 寫長，而是把可重複流程、檢查點、失敗處理與工具調用固定下來。
  • 高風險任務不應完全交給 LLM「憑整體感覺」評分。
  • 好的 AI 工作流會區分語意判斷與確定性驗證。
  • 任何需要可信度的學習內容，都應設計「引用存在性、邏輯跳躍、來源品質」的 gate。
  • 對課程或企業導入而言，這可做為「從 prompt 到 skill，再到可驗證流程」的案例。

可借用到 Hermes / 欒帥工作流的模式

  • 文章歸檔：模型擷取與摘要，程式驗證 URL、metadata、部署結果。
  • 課程頁生成：模型設計內容，程式檢查連結、表單、SEO metadata、無障礙結構。
  • AI 工具評測：模型整理功能，程式查 pricing、docs、API availability、版本日期。
  • 文件審查：模型判斷語意，程式做格式、欄位、隱私與發布範圍 gate。

風險與限制

  • 作者明確說這是 reference implementation，不是長期維護產品。
  • 部分流程深度綁定作者自己的 Obsidian、Zotero、複習網站與臨床 persona；使用者需要改 config.yaml。
  • 醫療與論文評讀不可視為醫療建議，也不能取代自己閱讀全文。
  • citation / author / PubPeer 等 fan-out 查核若失敗，應明確標為缺口，不可當作「沒找到問題」。
