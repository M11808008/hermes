Claude Code 官方 30 分鐘教學｜Agent 級提示工程與團隊工作流

Threads 貼文把這支 Anthropic 官方影片定位成「不用到處亂學提示詞，直接看官方示範」。實際重點確實不只是 Claude Code 操作，而是 coding agent 的完整工作法：先問 codebase、再規劃、再改碼、用測試或截圖迭代，最後把團隊工具與記憶配置成可重複的工作流。

一句話摘要

Anthropic 的 30 分鐘教學可視為 Agent 級提示工程範本：不要只丟一句「幫我寫功能」，而是讓 Claude Code 先理解 repo、調用 git / GitHub / CLI / MCP 等工具、提出計畫、取得驗證訊號，並用 CLAUDE.md、slash commands、permissions 與 MCP 把團隊脈絡制度化。

影片核心脈絡

主講者先把 Claude Code 定義成「fully agentic」的 coding assistant：它不是補幾行 code 的 autocomplete，而是用於做 feature、寫 function / file、修 bug、跑工具與整理工作。它刻意放在 terminal，因為能跨 VS Code、Xcode、JetBrains、Vim、SSH、tmux 等環境，減少被單一 IDE 綁住。

可直接採用的 Agent 工作流

| 階段 | 做法 | 對欒帥的應用 |

|---|---|---|

| 1. 先做 codebase Q&A | 新手不要一開始就叫它改碼，先問「這段怎麼用？」「這個 class 怎麼 instantiate？」「為什麼這個 function 有 15 個參數？」 | 用於接手 Hermes / KB / 課程頁 repo 時，先讓 agent 讀架構與 git history |

| 2. 要求先規劃 | 明確說「before you write code, make a plan / brainstorm / ask for approval」 | 大改版、部署、自動化前先產生計畫，避免 agent 直接亂改 |

| 3. 給它驗證工具 | unit tests、integration tests、Puppeteer 截圖、iOS simulator、web screenshot 都可當 feedback loop | KB 首頁、課程 landing page、管理介面都應讓 agent 截圖/測試後自行迭代 |

| 4. 教它用團隊工具 | CLI、MCP server、GitHub issue、Sentry、GCP logs、內部工具都可透過說明與設定交給 Claude Code 使用 | 把部署、OCR、Google Sheets、LINE 附件處理整理成工具與技能，而非每次重講 |

| 5. 把 context 制度化 | 用 CLAUDE.md、nested CLAUDE.md、slash commands、settings、permissions、MCP JSON | 對 Hermes skills / AGENTS.md / project memory 都是同一個 context engineering 思路 |

重要操作觀念

  • `terminal-setup`：讓終端輸入多行 prompt 更順；影片提到 Shift+Enter 類操作可改善長 prompt 體驗。
  • `theme`：設定 light / dark / daltonized theme。
  • `install github app`：可在 GitHub issue / PR 中 `@claude` 觸發工作。
  • allowed tools / permissions：常用安全命令可允許，危險命令或 URL 可封鎖。
  • 語音輸入 prompt：主講者提到在 macOS 用 dictation 直接「跟 Claude Code 說話」，適合描述複雜任務。
  • `commit push` 這類短指令：Claude Code 會根據 git log 與 repo 慣例自行整理 commit、branch、PR。
  • `#` 記憶：當工具用法或團隊偏好需要保存時，可把它寫入 memory。
  • `!` bash mode：直接跑本機命令，輸出會進入下一輪 context。
  • `@` mention files / folders：把關鍵檔案拉進 context。
  • Escape：停止目前操作並補充修正；Escape twice 可回到歷史。
  • `resume` / `continue`：接回先前 session。
  • Ctrl+R：查看更完整工具輸出。

Context Engineering 重點

影片中最值得保存的是「讓 agent 變聰明不是靠更長 prompt，而是靠正確脈絡配置」。可分成四層：

  • Project CLAUDE.md：repo 根目錄，放共享 bash commands、style guide、架構決策、核心檔案。
  • Local CLAUDE.md：個人偏好，不一定進版控。
  • Nested CLAUDE.md：特定子資料夾需要的脈絡，在 agent 進入該區域時再載入。
  • Enterprise / global policies：全公司共用的 permissions、blocked commands、MCP servers 與安全規則。

對 Hermes / Codex / Claude Code 類工作流來說，這與技能資料夾、AGENTS.md、專案記憶其實是同一件事：把「常識、工具、限制、驗證」變成 repo 內可維護的文字與設定。

Claude Code SDK / `-p` 的意義

影片後段提到 Claude Code SDK，可把 Claude Code 當成「超聰明 Unix utility」：

  • 將 prompt、allowed tools、output format 給 CLI。
  • 讓輸出成為 JSON 或 streaming JSON，接到 CI、incident response、log 分析、Sentry / GCP / GitHub pipeline。
  • 可以 pipe in / pipe out，讓 agent 進入現有 shell workflow，而不是另開一套系統。

這對欒帥的價值是：未來 AI 學習雷達、LINE 文件處理、報名名單整理、GitHub Pages 驗證都可逐步變成 CLI/cron/agent pipeline，而不只是聊天式操作。

團隊導入建議

  • 第一天不要要求同事直接「讓 agent 幫我開發功能」，先練 codebase Q&A。
  • 每個 repo 先建立短版 CLAUDE.md，只放高訊號規則：測試命令、部署命令、檔名規則、禁止事項、常見坑。
  • 對 UI / 文件 / 靜態站任務，要求 agent 必須截圖或 fetch public URL 驗證。
  • 對會動到 production、token、資料庫、個資的命令，預設要權限護欄。
  • 如果工作流重複出現三次，就把它升級成 slash command、skill 或腳本。
