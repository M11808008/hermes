Anthropic Agent Loop 工作坊｜Plan、Build、Judge 三代理建立可自我驗證的應用開發流程

Easycompany、Jokker 與 Hamza Khalid 轉述同一組 Claude Code / Anthropic 相關工作坊內容：工程師示範如何用 agent loop 從零建立完整 app。重點不是單一提示詞，而是讓多個代理分工：一個負責規劃，一個負責建造，一個負責評審，循環到成果真的可用。

一句話摘要

Agent loop 的價值不在「模型更聰明」，而在「流程能自我驗證」：Plan → Build → Judge 持續循環，直到符合明確成功標準或達到停止條件。

Loop 的基本模型

Easycompany 引用的長文將 Loop 拆成五步：

  • Discover：找出需要做什麼。
  • Plan：決定怎麼做。
  • Execute：執行任務。
  • Verify：對照目標檢查結果。
  • Iterate：未達標就把結果餵回去繼續。

這和傳統 prompt 最大差異是：Prompt 給 AI 一條單次指令；Loop 給 AI 一個目標、完成標準、驗證方式與放棄規則。

三個關鍵護欄

| 護欄 | 為什麼重要 |

|---|---|

| Verifier | 沒有真正的測試、規則或評分，Loop 只是 agent 反覆同意自己。 |

| State | 必須記錄完成、失敗、下一步，否則每輪都從零開始並重複犯錯。 |

| Stop condition | 沒有成功條件與硬性上限，Loop 可能整晚空轉、燒 token、仍無結果。 |

什麼任務適合 Loop？

原文建議先檢查四個條件：

  • 任務至少每週重複一次。
  • 壞結果可以被自動拒絕，例如測試、lint、type check、構建或硬規則失敗。
  • Agent 能從頭到尾完成，不會中途大量丟回給人類。
  • 「完成」必須客觀；若高度主觀，仍需人類在關鍵節點把關。

缺任何一條，就先不要搭重型 Loop；用好的 prompt 或半自動流程更穩。

為什麼先在軟體工程爆發？

程式碼有天然 verifier：測試通過、lint 乾淨、型別無錯、build 成功。這讓 agent 知道自己是否真的完成，而不只是產出看似合理的答案。

典型 coding loop 可寫成：

  • Goal：讓指定測試全綠、lint 乾淨、無 type error。
  • Each iteration：讀失敗訊息、選影響最大的一個失敗點、做最小修改、重新跑測試。
  • Verify：測試全綠 + 0 warning + 0 type error。
  • Stop：成功或達到迭代上限。
  • On stop：摘要已改內容與仍失敗項目。

組成元件

| 元件 | 角色 |

|---|---|

| Automation | 定時或事件觸發，讓任務不用每次手動啟動。 |

| Skill | 保存可復用任務說明，避免定時任務裡塞一大段難維護提示詞。 |

| Subagents | 將執行者與檢查者分開；writer 可以快，reviewer 要嚴格。 |

| Connectors | 讓 agent 能建立 PR、關聯 ticket、通知頻道，而不只給建議。 |

| Verifier | 真正的關卡：測試、型別、建構、規則驗證、rubric 評分。 |

成本與風險

Loop 不是免費魔法。每輪都可能重讀目標、上下文、上一輪結果與失敗原因；若再加入 maker + checker 結構，品質提升但成本也增加。成熟 Loop 需要：

  • 迭代上限。
  • token / 花費預算。
  • 便宜模型處理低價值步驟。
  • 接受率監控。
  • 失敗告警。
  • 硬性 verifier。

對欒帥的可用意義

  • 這是 AI Agent 課程中非常適合講「從 prompt 到 workflow」的案例。
  • 可用 Plan / Build / Judge 解釋多代理協作：不是叫三個 AI 同時聊天，而是給每個代理不同責任與驗證標準。
  • 可對照 Hermes 的 cronjob、skills、delegate_task、verification discipline，說明成熟 agent 系統需要觸發器、狀態、工具、驗證與停止條件。
  • 對企業導入 AI，可提醒「先手動跑通，再固化成 Skill，再加 Loop，最後才排程自動化」。

最佳導入順序

  • 先讓一次手動流程穩定成功。
  • 把流程沉澱成 Skill / SOP。
  • 加入 verifier、state 與 stop condition。
  • 拆出 maker / reviewer 或 planner / builder / judge。
  • 最後再放入 cron、GitHub Actions、hooks 或平台自動化。
