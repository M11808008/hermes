Runway Recipes API｜把產品廣告、商品替換與 UGC 影音工作流封裝成 API

來源是 Kufu 酷服在 Threads 的整理，並追到 Runway 官方 X 貼文與 Runway API 官方文件。Threads 原文的核心判斷是：Runway 不是只推出新模型，而是把官方設計好的影音生成流程封裝成 API endpoint，讓企業與開發者用一次 API 呼叫即可產出產品廣告、商品替換、UGC 影片或行銷素材。

一句話摘要

Runway Recipes API 把「模型選擇、prompt engineering、工作流設計、後處理」包成可呼叫的官方 recipe；對電商、MarTech、SaaS 與內容平台來說，這讓 AI 影音從「自己串模型」變成「把成熟影音能力嵌進產品」。

Runway Recipes 目前涵蓋的工作流

| Recipe | 官方用途 | 欒帥可怎麼理解 |

|---|---|---|

| Product Ad | 從產品參考圖、風格參考與創意方向生成 cinematic 產品廣告影片 | 商品照 → 社群短廣告，適合電商、課程教材商品化示範 |

| Product Swap | 把既有影片中的商品替換成新產品，保留鏡頭運動、燈光與場景構圖 | 一支成功廣告模板 → 套不同商品 SKU，適合 catalog / A/B creative |

| Product UGC | 用人物參考圖與產品圖生成 UGC 風格促銷影片 | 模擬 creator / KOC 短影音，但需特別注意肖像與授權 |

| Multi-Shot Video | 依 story prompt 或 3–5 個 shot list 生成多鏡頭影片 | 比單鏡頭更像完整短廣告，可用於品牌故事、課程預告 |

| Marketing Stock Image | 依文字 brief 與可選 Logo 生成品牌行銷圖 | 替代 stock photo 搜尋或拍攝前期草稿 |

| Product Campaign Image | 以單張產品圖與風格 brief 生成四張 campaign hero visuals | landing page、廣告素材、商品視覺方向探索 |

這件事為什麼重要

過去企業導入 AI 影片，技術成本常不是「能不能呼叫模型」，而是：

  • 如何把產品照、品牌規範、情境、鏡頭語言轉成穩定 prompt。
  • 如何在不同模型之間選擇合適能力。
  • 如何處理素材規格、長度、解析度、失敗重試與後處理。
  • 如何把一次性的創作流程變成可大量生成的 API 產品能力。

Runway Recipes 的意義在於把這些「工作流 know-how」商品化。它不是只賣模型，而是賣一組經官方調校過、可直接嵌入商業產品的 media generation workflow。

對欒帥的實用判讀

1. AI 影音平台會從「模型 API」進入「工作流 API」競爭

Threads 原文判斷很準：這不是單純模型競賽，而是 workflow competition。未來教學與產品設計可以把 AI 能力分成三層：

| 層級 | 例子 | 價值 |

|---|---|---|

| Base model | text-to-video、image-to-video、image generation | 高自由度，但要自己設計 prompt 與流程 |

| Workflow recipe | Runway Product Ad / Product Swap / UGC | 用較少參數完成常見商業任務 |

| Productized feature | 電商後台一鍵生成商品廣告、MarTech 平台自動生成素材 | 對終端使用者最直覺，可大規模部署 |

2. 很適合做「AI Agent × MarTech」課程案例

可以把 Runway Recipes 設計成一個課堂案例：

  • Hermes / ChatGPT 先整理品牌、商品、受眾與文案角度。
  • 用 spreadsheet 或 Airtable 管理商品圖片與 campaign brief。
  • 後端批次呼叫 Runway Recipes API 生成多組素材。
  • 人工審核品牌一致性、錯誤畫面、肖像與授權。
  • 合格素材進入廣告投放、社群排程或 landing page。
  • 成效數據回流，決定下一輪 creative prompt 與 reference media。

這比單純示範「AI 生成一支影片」更有教學價值，因為它把 AI 影音放進真實商業 pipeline。

3. 對 AI 學習雷達／課程宣傳的可能用法

| 場景 | 可用 Recipe | 實作方式 |

|---|---|---|

| 課程預告短片 | Multi-Shot Video / Product Ad | 將課程定位、學習成果、講師形象轉成 5–15 秒預告 |

| AI 工具介紹影片 | Multi-Shot Video | 用 3–5 個 shot list 呈現工具痛點、流程、結果 |

| 知識庫文章封面 | Marketing Stock Image / Product Campaign Image | 為重要專題產生一致風格的封面與 social card |

| 商品化教材素材 | Product Ad | 把課程、工作坊、教材包視為「產品」做廣告視覺 |

| 電商／品牌案例教學 | Product Swap / Product UGC | 示範同一廣告模板如何換商品、換受眾、換風格 |

建議實驗流程

| 步驟 | 做法 | 驗證點 |

|---|---|---|

| 1. 選一個明確 use case | 例如「AI 課程 10 秒短影音」或「單一產品 4 秒廣告」 | 任務是否屬於官方 recipe 覆蓋範圍 |

| 2. 準備 reference media | 單一主體、清楚角度、乾淨背景、足夠解析度 | 不要讓多個主體干擾產品識別 |

| 3. 寫 brief | 受眾、平台、語氣、風格、長度、禁忌元素 | brief 是否比單句 prompt 更接近商業需求 |

| 4. 呼叫 recipe endpoint | 使用官方 SDK 或 REST API；取得 task id | 任務是否穩定完成、是否需重試 |

| 5. 輪詢與保存輸出 | `GET /v1/tasks/{id}` 取得影片或圖片 URL | 檔案保存、命名、版本管理是否可追蹤 |

| 6. 人工審核 | 品牌、肖像、文字、Logo、商品形變、合規 | 是否可直接上線，或只適合作為草稿 |

| 7. 成本紀錄 | 記錄 credits、長度、解析度、成功率 | 是否適合批量生成 |

成本與規格提醒

Runway 官方文件列出 recipe pricing，且多數影片類 recipe 依解析度與秒數計費。例如 Product Ad、Product Swap、Product UGC 皆有 720p / 1080p 與 4–15 秒長度範圍；Multi-Shot Video 則以每秒 credits 計。這表示它適合「短影音 creative」與「廣告素材變體」，不應直接視為低成本長片生成方案。

實際部署前應建立：

  • 每支影片預估 credits。
  • 每批生成的最大張數／支數。
  • 失敗重試上限。
  • 人工審核與淘汰率。
  • 素材授權與肖像權檢查。

和既有 Runway MCP 的差異

知識庫中已有 Runway MCP 文章，重點是「在 ChatGPT、Claude、Cursor 等代理環境裡直接使用 Runway」。本篇的角度不同：Recipes API 是給開發者與企業產品嵌入使用的 workflow endpoint。

| 方向 | Runway MCP | Runway Recipes API |

|---|---|---|

| 使用者 | 創作者、agent 使用者、ChatGPT/Claude 使用者 | 開發者、SaaS、MarTech、電商平台 |

| 入口 | MCP server / 對話式代理 | REST API / SDK / endpoint |

| 核心價值 | 少切工具，在同一上下文生成素材 | 把成熟工作流嵌入產品，大規模批次生成 |

| 最適合 | 個人創作、互動式修改、agent demo | 產品化、自動化、catalog creative、行銷素材 pipeline |

判讀限制

  • Threads 原文是中文社群摘要，本文以 Runway 官方 X 貼文與官方 API docs 補強；沒有實際呼叫 API 測試輸出品質。
  • 官方 Recipes 清單、定價、長度限制與支援輸入格式可能持續變動，部署前應以 Runway API 文件與帳號方案為準。
  • Product UGC 牽涉人物參考圖與擬真促銷影片，實務上必須處理肖像權、揭露、廣告法規與品牌安全。
  • Product Swap 可降低 creative production 成本，但不代表所有產品外觀、材質、Logo 與使用情境都能穩定保真；重要商用素材仍需人工 QA。
