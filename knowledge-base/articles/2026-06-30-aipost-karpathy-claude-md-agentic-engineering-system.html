AI郵報｜放下神奇 Prompt：從 Karpathy 的 CLAUDE.md 看 AI Coding 系統化

AI郵報這則 Threads 串文與站內文章〈放下你的神奇 Prompt！從 Karpathy 的 CLAUDE.md 看 AI Coding 的殘酷真相〉，核心觀點是：AI Coding 的重點正在從「怎麼問 AI 比較聰明」轉向「怎麼設計一套讓 AI 不亂搞的工作系統」。

貼文用 Karpathy 的 CLAUDE.md 爆紅作為切入點，提醒開發者：收藏 100 份 prompt 模板無法解決 AI Coding Agent 的核心問題。AI 很強，但它常常不知道自己搞錯了；因此真正重要的是規則、上下文、驗證流程、錯誤記憶與交接格式。

一句話摘要

Karpathy 的 CLAUDE.md 之所以重要，不是因為它是一份神奇 prompt，而是它示範了 AI Coding 正在從 prompt 技巧走向 Agentic Engineering：用專案規則、任務邊界、驗證流程與安全審查，讓 AI Agent 在可控環境中穩定工作。

CLAUDE.md 是什麼

串文將 CLAUDE.md 解釋成：放在專案根目錄的一份 Markdown 檔。Claude Code 啟動時會讀取它，把它當成這個專案的工作守則。

簡單說，就是先告訴 AI：

  • 在這個專案裡，你該怎麼工作。
  • 什麼不能亂碰。
  • 遇到不確定時要怎麼處理。
  • 做完後要怎麼驗證。
  • 如何回報修改內容與待確認事項。

這讓 AI 不只是收到一次性的指令，而是在整個專案上下文中遵守一套持續存在的工程規範。

Karpathy 版 CLAUDE.md 的核心精神

貼文歸納出幾條看似樸素、但對 AI Agent 特別關鍵的規則：

  • 不要沉默假設。
  • 不要過度工程化。
  • 不要亂改任務外的東西。
  • 做完一定要驗證。

這些聽起來像菜鳥工程師的入職守則，但尷尬的是，這正好也是 AI Coding Agent 最常犯的錯。

AI Coding Agent 常見失控模式

AI郵報列出的問題非常貼近實務：

  • 需求沒問清楚時，AI 會直接猜。
  • 看到旁邊程式碼不順眼，AI 可能順手重構。
  • 明明只是簡單 bug，AI 可能蓋出一座抽象工廠。
  • 修改完後，AI 會自信地說「應該可以」。

這些問題不是靠更華麗的 prompt 就能解決，而是要靠可檢查的系統設計。

從 Prompt Engineering 到 Agentic Engineering

貼文認為 AI Coding 的關鍵已逐漸從一句漂亮 prompt，轉向一整套工程系統：

| 元件 | 作用 |

|---|---|

| Rules | 定義 AI 可以做什麼、不能做什麼 |

| Context | 提供專案架構、技術棧、命名慣例與限制 |

| Verification | 要求跑測試、lint、build 或其他驗證命令 |

| Memory | 記錄曾經踩過的錯誤與專案慣例 |

| Format | 規範交接、回報、diff 摘要與待確認事項 |

這些東西加起來，才是讓 Agent 穩定工作的環境。

Loop Engineering 的關鍵

貼文引用 Claude Code 創造者 Boris Cherny 的觀點：「我不再下 Prompt 給 Claude 了。我寫 Loops，Loops 來做工作。」

這就是 Loop Engineering：

  • 設計一個循環。
  • 讓 AI 定期檢查、修復、開 PR、處理 review、整理回饋。
  • 人類不再只是逐句下指令，而是設計 AI 該怎麼跑。

這與 AI 學習雷達既有條目「Agent Loops / Loop Engineering 入門」可串成同一條趨勢：AI Agent 的價值不在單次回答，而在可重複、可驗證、可停止的工作循環。

Vibe Coding vs Agentic Engineering

貼文也清楚區分兩種工作模式：

| 模式 | 適合場景 | 特徵 |

|---|---|---|

| Vibe Coding | 週末小專案、快速 demo、探索想法 | 用自然語言講感覺，AI 快速做出原型，很爽但不一定穩 |

| Agentic Engineering | 正式產品、團隊協作、可上線系統 | 有規格、階段、測試、驗證、邊界與審查流程 |

這是非常適合放進課程的觀念：Vibe Coding 可以降低入門門檻，但如果要上線產品或交付客戶，就必須進入 Agentic Engineering。

未來工程師能力的轉移

AI郵報認為，未來工程師的核心能力會發生轉移。單純「會寫程式碼」的價值會被壓縮，更重要的是：

  • 能不能判斷 AI 寫得對不對。
  • 能不能設計驗證流程。
  • 能不能定義任務邊界。
  • 能不能讓 Agent 在安全範圍內自己工作。
  • 能不能把工具變成一套穩定工作流。

也就是說，未來厲害的工程師會更像「AI 工作環境的架構師」。工具會變，但工作流會留下來。

CLAUDE.md 也可能是攻擊入口

貼文特別提醒：CLAUDE.md 不是無腦裝上去就好。它本身也可能變成攻擊入口。

如果有人在 CLAUDE.md 裡塞入惡意指令，AI Agent 可能每次啟動都照著壞規則做事。因此應把 CLAUDE.md 當成類似 CI/CD 設定或 `.env` 一樣敏感的設定檔：

  • 要審查。
  • 要看內容。
  • 不要亂抄網路模板。
  • 不要把來路不明的專案規則直接套進工作區。

對欒帥的課程應用

1. 把「神奇 Prompt」轉成「工作系統」

這則非常適合用在 AI Coding 課程的觀念轉折：

  • 初學者會先追求 prompt 模板。
  • 進階使用者會開始建立專案規則。
  • 真正能交付的人，會設計驗證、review、邊界與回報流程。

課堂可以用一句話概括：不要只問「怎麼問 AI」，要問「怎麼讓 AI 在一套安全系統裡工作」。

2. 可設計一份課堂版 CLAUDE.md / AGENTS.md

可讓學員練習建立自己的專案工作守則：

  • 專案目標與技術棧。
  • 可修改與不可修改範圍。
  • 命名規則與檔案結構。
  • 測試與驗證命令。
  • 遇到不確定時如何提問。
  • 完成後的回報格式。
  • 安全限制與禁止事項。

這可直接延伸到 Claude Code、Codex、Hermes Agent 或任何支援專案指令檔的 AI 工具。

3. 對 AI 學習雷達的啟發

AI 學習雷達本身也可以視為 Agentic workflow：

  • 規則：如何判斷是否歸檔、公開或私密。
  • Context：既有知識庫、標籤、課程脈絡。
  • Verification：建置、部署、檢查連結。
  • Memory：已知偏好與穩定規範。
  • Format：固定文章格式、摘要、來源與驗證回報。

這正是「工作流會留下來」的實際案例。
