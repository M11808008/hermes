NotebookLM 辦公室五大工作流｜多來源整合、報告、會議、引用問答與團隊知識庫

來源是 Threads@janrobert1722 的長文，主軸是「NotebookLM 在辦公室環境最常被用的 5 大功能」。原文偏實用教學與推廣語氣；本文保留可落地的工作流，但用 Google 官方 Help 與欒帥既有 NotebookLM 文章校正功能邊界，避免把尚未普及或方案限定的功能當成所有帳號都可用。

一句話摘要

NotebookLM 最適合放在「可信來源內的知識整理與交付」位置：先把公司文件、會議紀錄、報告、合約、SOP、課程教材整理成分層 notebook，再用明確 prompt 要它做跨來源比較、報告草稿、會議待辦、引用問答與新人/團隊知識庫；高風險輸出仍要人工複核來源、權限、數字與責任歸屬。

Threads 原文整理出的五大功能

| 功能 | 原文主張 | 轉成可執行工作流 |

|---|---|---|

| 多來源知識庫整合 | 匯入 Docs、Sheets、Slides、PDF、Word、網頁、YouTube、資料夾等來源，讓 AI 建成專屬知識大腦 | 依主題建立 notebook：公司規章、產品資料、客戶合約、行銷企劃、課程教材；不要把所有資料混在同一本 |

| 深度分析與專業報告 | 從資料中產出摘要、分析、建議、行動方案、簡報大綱 | 用「目的 + 範圍 + 格式 + 語氣 + 長度」下 prompt，並要求每個結論附來源 |

| 會議轉錄與整理 | 上傳錄音後整理逐字稿、決議、待辦、負責人與截止日 | 會前先放議程與背景資料，會後上傳錄音/逐字稿，再生成 action items；人名、日期、數字必須人工核對 |

| 知識問答與精準引用 | 像內部 Google，只從上傳來源回答並標註引用 | 適合查合約條款、SOP、過去案例、課程教材；問句要明確到文件範圍與輸出格式 |

| 協作共用與團隊知識庫 | 建立團隊空間、設定權限、共用知識資產 | 以 Google Workspace 權限治理為前提：公司級、部門級、專案級分層，指定知識守門員維護 |

欒帥可直接套用的 Prompt 模板

1. 多來源專案復盤

```text

請只根據我這本 Notebook 的來源，整理過去 3 年專案資料。

輸出表格欄位：專案名稱、目標、預算區間、成功關鍵、失敗原因、風險點、可沿用做法、來源引用。

請最後給出今年最適合沿用的 5 條 SOP，並標註每條來自哪份文件。

```

2. 給主管看的月報

```text

根據本月會議紀錄、任務追蹤與數據表，寫一份給主管看的月報。

請包含：成果摘要、進度、延遲/風險、下月計畫、需要主管決策的事項。

格式：先 200 字摘要，再用表格列重點；語氣正式、務實、不要過度行銷。

每個重要結論請附來源引用。

```

3. 會議待辦與爭議點

```text

請根據會議逐字稿與議程，整理：

  • 已決議事項
  • 待辦事項：任務、負責人、截止日、優先級
  • 尚未有共識的爭議點
  • 下次會議應先討論的 5 個問題

若逐字稿中沒有明確負責人或日期，請標註「待確認」，不要自行猜測。

```

4. 合約 / 規章問答

```text

請只根據我提供的合約與公司規章回答：

問題：如果對方延遲交付超過 14 天，我方可採取哪些處理方式？

請列出相關條款、條款原文摘要、可採取行動、風險提醒、需要法務確認的事項。

每一點都要標示來源文件與條款位置。

```

5. 新人入門知識庫

```text

請把這本 Notebook 的資料整理成新人入門指南。

輸出：第一週學習路線、必讀文件清單、常見問題、容易踩雷的流程、可以問誰、30 天內應完成的任務。

請區分「全公司通用」、「部門限定」、「專案限定」三層。

```

使用架構建議

| 層級 | Notebook 類型 | 適合資料 | 權限建議 |

|---|---|---|---|

| 公司級 | 公司規章 / 品牌 / 制度 | 員工手冊、SOP、政策、表單 | 多數人可檢視，少數知識守門員可維護 |

| 部門級 | 行銷 / 業務 / 行政 / 教學 | 部門流程、案例、話術、教材 | 部門內可編輯，跨部門可視需要檢視 |

| 專案級 | 單一客戶或活動 | 合約、會議、時程、報價、風險紀錄 | 僅專案成員可存取；含個資/商業機密不得公開分享 |

| 個人級 | 個人研究 / 備課 | 閱讀資料、靈感、草稿 | 個人維護，成熟後再升級到團隊 notebook |

重要校正與限制

  • **來源類型以官方 Help 與帳號介面為準**：Google Help 顯示 NotebookLM 支援加入或探索來源，且支援多種文件與音訊來源；但實際檔案大小、數量、格式、Drive 匯入限制與方案差異會變動。
  • **「即時會議轉錄」不應過度解讀**：原文把 NotebookLM 描述成開會時即時錄音整理工具；導入時應先確認目前帳號是否支援直接錄音或音訊上傳。更穩妥做法是用 Google Meet / 錄音工具產生音訊或逐字稿，再匯入 NotebookLM 整理。
  • **團隊空間與權限治理要小心**：NotebookLM 的分享、Workspace、公開 notebook、featured notebook 等能力會依帳號與政策不同；含客戶資料、合約、報名名單、個資的 notebook 不可任意公開。
  • **AI 報告不是最後答案**：合約、法規、財務、醫療、人資等場景，一定要人工確認引用來源、日期、數字、人名與決策責任。
  • **先分類再上傳**：原文最重要的一點是「不要把雜亂資料全丟進同一本」。NotebookLM 的品質很大程度取決於資料庫設計，而不是 prompt 魔法。

對欒帥的應用價值

  • **AI 課程教材**：可把 NotebookLM 當成「企業知識管理 / 辦公自動化」案例，讓學員練習建來源庫、問問題、產生報告、驗證引用。
  • **課程行政與報名資料**：適合整理去識別化後的 FAQ、SOP、回饋摘要；含個資的原始名單仍應保存在私密本機或受控 Workspace，不公開部署。
  • **AI 學習雷達互補**：雷達負責外部趨勢歸檔；NotebookLM 適合針對單一主題建立可信資料庫，再輸出教案、簡報、報告草稿。
  • **企業導入顧問**：可用「先建庫、再問答、再報告、再治理」四步驟說明，而不是只賣「AI 幫你整理」。
