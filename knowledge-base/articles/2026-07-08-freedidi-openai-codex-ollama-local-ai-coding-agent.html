文章主張：Codex 不一定要走雲端 API

零度博客這篇文章的主張可整理為：

  • AI 編程 Agent 過去多半綁定雲端模型與 API 額度。
  • Codex / Claude Code / Cursor Agent 等工具之所以強，是因為背後有高能力模型與工具執行環境。
  • Ollama 新增與 Codex 的整合後，本地開源模型可以被 Codex CLI / Codex App 調用。
  • 因此，本地模型也能讀取專案、建立檔案、修改 HTML / CSS / JavaScript、執行任務，變成「可自動幹活」的 coding agent。

這對 AI 學習雷達來說值得保存，因為它代表 AI coding agent 正在從「雲端訂閱 / API token」擴展到「本地模型 + agent shell」的混合架構。

官方驗證：Ollama 確實支援 Codex CLI

Ollama 官方文件〈Codex CLI - Ollama〉給出明確用法：

```bash

npm install -g @openai/codex

```

快速啟動：

```bash

ollama launch codex

```

手動用 Codex CLI 走 Ollama / open-source provider：

```bash

codex --oss

```

指定模型：

```bash

codex --oss -m gpt-oss:120b

```

也可建立 profile，例如 `~/.codex/ollama-launch.config.toml`：

```toml

model = "gpt-oss:120b"

model_provider = "ollama-launch"

model_catalog_json = "/Users/you/.codex/model.json"

[model_providers.ollama-launch]

name = "Ollama"

base_url = "http://localhost:11434/v1/"

wire_api = "responses"

```

然後執行：

```bash

codex --profile ollama-launch

```

Ollama Blog 也說明：Open models can be used with OpenAI’s Codex CLI through Ollama；Codex 可以 read、modify、execute code in your working directory。預設使用 local `gpt-oss:20b`，也可切換 `gpt-oss:120b` 或其他 open-weight alternatives。

OpenAI 官方文件：`--oss` 是 Codex CLI 支援的模式

OpenAI Developers 的 Codex CLI reference 中，`--oss` 被描述為：

> Use the local open source model provider，等同於 `-c model_provider="oss"`，並會驗證 Ollama 是否正在執行。

OpenAI Codex advanced configuration 也提供 model provider、profile、`model_provider`、`base_url`、`wire_api` 等進階設定方式。這確認了：Codex CLI 不只是一個 OpenAI 雲端模型前端，也能接入本地或自訂 provider。

正確理解「免費」

原文標題使用「OpenAI Codex 徹底免費」很吸睛，但知識庫應做更精準判讀。

真的免費的部分

  • 不一定需要 OpenAI API token 才能跑 Codex CLI agentic coding。
  • 使用 Ollama 本地模型時，推理不按 OpenAI API 計費。
  • 可使用 open-weight 模型，例如 `gpt-oss:20b`、`gpt-oss:120b` 或其他 Ollama 支援模型。
  • 程式碼與資料可留在本機，對隱私與離線工作流有吸引力。

仍然有成本的部分

  • 本機硬體：大模型需要 CPU / GPU / RAM / VRAM。
  • 上下文長度：Ollama Blog 建議 Codex 至少需要 32K context window。
  • 速度：本地模型速度取決於硬體，未必能接近雲端 frontier model。
  • 模型能力：本地 open-weight coding model 不一定能達到 GPT-5.5 / Claude Opus / Fable 等高階雲端模型的推理與收斂能力。
  • 維護成本：要管理 Ollama、模型版本、Codex CLI config、profiles、工作目錄權限。
  • 風險成本：agent 能修改與執行程式碼，仍需要 sandbox、approval、git diff、測試與人工 review。

因此，最準確的說法是：「Codex 的 agent shell 可以接本地模型，降低 API 成本並提升資料控制；但不是所有成本都消失。」

適合使用 Codex + Ollama 的情境

1. 學習與實驗

對想練習 AI coding agent 工作流的人，Codex + Ollama 很適合：

  • 不怕 API token 消耗。
  • 可反覆測試 prompt、AGENTS.md、profile、sandbox 設定。
  • 可在小型 repo 觀察 agent 如何讀檔、改檔、跑測試。

2. 隱私敏感但低風險的本地專案

若資料不適合丟雲端，但任務又不需要 frontier model，則可用本地模型處理：

  • 內部工具。
  • 個人筆記 / 自動化腳本。
  • 小型前端 demo。
  • 文件整理與格式轉換。

3. 大量機械式改動

本地模型可負責低風險、可驗收、重複性強的工作：

  • 批次改命名。
  • 簡單重構。
  • 補測試樣板。
  • 轉換格式。
  • 建立小型 HTML / CSS / JS demo。

4. 混合架構：本地 executor + 雲端 orchestrator

更進階的用法是：

  • 高階雲端模型負責規劃、拆工單、仲裁與 review。
  • 本地 Codex + Ollama 負責低階實作、跑測試、批次修改。

這與先前歸檔的「Fable 5 orchestrator + Codex executor」工作流一致，只是 executor 可以替換成更低成本的本地模型。

不適合完全依賴本地 Codex 的情境

  • 複雜跨服務架構與高風險 migration。
  • 安全敏感、金融、身份驗證、權限模型修改。
  • 需要強產品判斷、設計品味或長程規劃的任務。
  • 巨大 codebase 且需要超長 context 的任務。
  • 沒有測試、沒有 git、沒有 sandbox 的專案。

本地模型可幫忙，但不應被視為「免費高階工程師」。

建議的安全操作方式

若要嘗試 Codex + Ollama，建議採用以下守則：

  • 在乾淨 git branch 中操作。
  • 先用小型 toy repo 測試，不要直接丟 production repo。
  • 明確限制可修改檔案範圍。
  • 開啟 human approval，不要一開始就使用 yolo / bypass sandbox。
  • 每輪都看 git diff。
  • 每輪要求 agent 回報 plan、modified files、tests、remaining risks。
  • 任何敏感設定、金鑰、部署腳本、資料庫 migration 都要人工確認。
  • 本地模型若 hallucinate 指令或亂改檔，立即停止並重設工作樹。

這篇補上的角度是：當 agent shell 可以接本地模型後，未來可能形成三層架構：

  • Frontier model：做高階判斷、未知管理、任務切分與最終仲裁。
  • Local coding model：做低成本 executor，處理可驗收的小工單。
  • Objective gates：git diff、tests、smoke、lint、human review，防止低成本執行變成高成本重工。

對欒帥 / Hermes 的實踐價值

這篇值得保存，因為它可能影響未來 Hermes 本機工作流：

  • 用本地模型處理低敏感、可驗收的程式或文件任務。
  • 用 Codex CLI profile 分離雲端模型與本地 Ollama 模型。
  • 將本地 agent 作為批次 executor，減少雲端 token 消耗。
  • 對私密資料或課程文件，可考慮本地模型做初步整理，再由高階模型做摘要與驗收。

但要注意，Hermes 現有工作流重視穩定部署、私密資料邊界與高質感輸出；本地模型導入前，需要先用小型任務測試品質與安全性。

風險與限制

  • 原文網站被 Cloudflare 擋住，本文未完整引用原文，只保存經搜尋與官方文件驗證的主題脈絡。
  • 「Codex 徹底免費」不應理解為沒有任何成本；只是可改用本地 / open-weight 模型避開 OpenAI API token。
  • 本地模型能力差異很大，尤其在大型 codebase、長上下文、複雜 debugging、架構判斷上可能明顯弱於 frontier models。
  • Agent 能改檔與執行命令，仍需 sandbox、approval、git diff、測試與人工驗收。
  • 大模型本地推理可能需要大量 RAM / VRAM，對一般筆電未必流暢。
