HyperFrames 完整參考｜用 HTML、Agent Skills 與 CLI 建立可重現影片生產線

一句話摘要

HyperFrames 把影片製作改寫成「HTML + CSS + JS + 可 seek 動畫 + CLI render」的工程流程，讓 Claude Code、Codex、Cursor、Gemini CLI 等 AI coding agent 可以像維護程式碼一樣規劃、產生、檢查、預覽與輸出 MP4。

來源定位

本篇由 LINE 上傳文件《hyperframes-參考文件.pdf》轉製為 AI 學習雷達知識庫文章。原 PDF 標示為莊哲昀（Andy Juang）於 2026-06-01 彙整，資料來源為 HyperFrames 產品首頁、官方文件與 GitHub repo，並註明為非官方彙整參考。本文將其內容整理成可公開閱讀的「完成版」知識庫文章，並以 Brave Search、GitHub README、官方首頁與 HeyGen Help Center 交叉驗證核心定位。

為什麼這份文件值得收錄

HyperFrames 不是又一個「文字生成影片」工具，而是一套更接近 agent-native video engineering 的框架。它的核心價值在於：

  • 影片內容可以被版本控管：composition 是 HTML，不是封閉剪輯軟體專案檔。
  • Agent 比較容易操作：LLM 本來就擅長生成 HTML、CSS、JS 與結構化文件。
  • 輸出可重現：render 採逐影格 seek，不依賴即時播放時鐘。
  • 可納入自動化：CLI 支援 init、lint、preview、render、snapshot、transcribe、capture 等步驟。
  • 可延伸成內容工廠：同一份模板可套變數、資料、字幕、旁白與不同素材，批次輸出個人化影片。

對 AI 學習雷達來說，HyperFrames 可以把「知識庫文章」往下游延伸成短影音、課程宣傳、產品 demo、教學動畫與社群摘要影片。

HyperFrames 是什麼

官方口號是：Write HTML. Render video. Built for agents.

更具體地說，HyperFrames 讓開發者用標準 HTML、CSS、JavaScript 描述影片畫面，再透過 headless Chrome 與 FFmpeg 逐影格輸出 MP4。它不要求 React，也不建立專屬 DSL；composition 本身就是帶有 data 屬性的 HTML 檔。

最小心智模型如下：

  • HTML：定義畫面、clip、文字、媒體、圖層與時間。
  • CSS / Tailwind：定義視覺風格、排版、品牌色、字體與響應式尺度。
  • JS / GSAP / Lottie / Three.js：定義可 seek 的動畫 runtime。
  • HyperFrames runtime：在特定時間點 seek 到正確狀態。
  • Chrome：擷取每一影格。
  • FFmpeg：把影格與音訊編碼成影片。

因此，HyperFrames 的關鍵不是「播放動畫」，而是「在任何時間點都能 deterministic 地重建該影格應有的 DOM 狀態」。

三個主要受眾

給開發者

開發者可以使用熟悉的 web stack 製作影片：HTML、CSS、JS、GSAP、Lottie、Three.js、Anime.js、WAAPI、FFmpeg。Composition 不需要進剪輯時間軸，也不需要學一套專屬元件系統。

給 AI 代理

AI coding agent 天然擅長產生 HTML 結構與修改程式碼。HyperFrames 的 CLI 預設可以非互動執行，輸入用 flags，輸出是純文字，錯誤直接停下，這正適合 agent 自動化操作。

常見 agent 入口包含：

  • Claude Code skills
  • Codex skills / plugin metadata
  • Cursor rules / skills
  • Gemini CLI skills
  • GitHub Copilot CLI 類似工作流

給自動化流程

HyperFrames 的 render 管線是 seek-driven，而不是 clock-driven。概念上可理解為：frame = floor(time * fps)。每一影格都由 renderer 主動 seek，並等 composition 宣告 ready 後才截圖，所以預覽卡頓不代表輸出會掉格。

Composition Contract：HTML 不是隨便寫，而是影片規格

HyperFrames composition 的 HTML 需要遵守一套 contract。核心元素通常包含：

  • data-composition-id：composition 唯一識別。
  • data-width / data-height：輸出解析度，例如 1920 × 1080。
  • data-duration：影片或片段時長。
  • data-fps：每秒影格數。
  • class="clip"：標示可被 HyperFrames runtime 控制時間顯示的元素。
  • data-start：clip 起始秒數。
  • data-duration：clip 顯示時長。
  • data-track-index：軌道與 z-order 控制，數字越高通常越前景。
  • data-media-start：媒體來源裁切起點。
  • data-volume：音訊音量。

實作時最重要的觀念是：元素如果需要被時間控制，就要成為 clip；如果只是一般 DOM 元素，可能會一直存在於畫面中，造成時間軸錯亂。

時間與軌道模型

HyperFrames 的時間線不是傳統剪輯軟體 UI，而是 HTML 屬性與解析規則。

常見時間規則：

  • data-start="0"：絕對秒數。
  • data-start="intro + 0.5"：相對某個 clip 起點或結束點的偏移。
  • data-duration="3"：顯示三秒。
  • 同一 track 上 clip 不應重疊。
  • track-index 可用來安排畫面層級與時間列分組。
  • 連鎖參照可以使用，但應控制在 3–4 層內，否則可讀性下降。

實務上，agent 最容易犯的錯是把畫面寫得漂亮，卻沒有把 timing contract 寫完整。這會讓 lint、preview 或 render 出現錯誤，或讓某些元素全程顯示。

動畫 runtime：重點是可 seek

HyperFrames 可以接多種動畫方式，但共同要求是「能 seek 到任意時間點」。PDF 特別強調 GSAP 的使用方式：

  • timeline 應以 paused: true 建立。
  • timeline 要註冊到 window.__timelines 或 HyperFrames 可存取的位置。
  • renderer 會控制 timeline.totalTime 或 seek，而不是讓動畫自己播放。
  • 如果 GSAP timeline 太短，即使 video data-duration 很長，畫面也可能提早被截斷或停住。

最常見修法是用空 tween 延長 timeline，例如在最後時間點加一個不影響元素的 set / tween，讓 timeline duration 對齊影片總長。

Render Pipeline：為什麼可重現

HyperFrames 的輸出流程大致是：

  • 解析 composition HTML。
  • 啟動 Chrome / Puppeteer。
  • 載入頁面與所有 assets。
  • 等待 __playerReady 與 __renderReady 之類的 ready gate。
  • 依 fps 計算每個影格時間。
  • 對每個時間點 seek runtime 狀態。
  • 擷取 PNG frame。
  • 交給 FFmpeg 編碼。
  • 混合音訊、字幕或其他 media track。
  • 輸出 MP4 / WebM 等格式。

這解釋了兩個重要現象：

  • 瀏覽器 preview 可能卡，但 final render 不一定掉格。
  • local render 在不同機器可能因字體、Chrome、GPU 或系統 media codec 有細微差異；若要 CI/CD 完全一致，需要固定容器或雲端 render 環境。

CLI 地圖：從 init 到 render

PDF 整理了大量 CLI 能力。對入門者最重要的是這條 loop：

```bash

npx hyperframes init my-video

cd my-video

npx hyperframes lint

npx hyperframes preview

npx hyperframes render --output output.mp4

```

常用命令可整理如下：

  • init：建立新專案，可選 blank、範例、Tailwind、帶入 video 等。
  • lint：檢查 composition 結構、JSON、變數、clip、時間與 media 問題。
  • preview：啟動本機預覽環境，快速檢查節奏與畫面。
  • render：輸出 MP4 / WebM，可指定解析度、fps、變數、輸出檔。
  • compositions：列出專案內 composition ID、時長、解析度與元素數。
  • transcribe：把音訊或影片轉逐字時間戳，可用於字幕與旁白對齊。
  • capture：擷取網站截圖、設計 token、字型、素材與動畫線索。
  • snapshot：在指定秒數或影格輸出 PNG，便於 QA 與視覺審查。
  • skills：安裝或同步 HyperFrames skills 到 Claude、Gemini、Codex、Cursor 等目標。
  • doctor：檢查 Node、FFmpeg、Chrome、字體、相依套件等環境問題。

Variables：讓影片模板變成可批次生產的系統

HyperFrames composition 可以宣告變數，讓同一支影片模板套入不同文案、品牌、產品、CTA、角色或資料。

典型變數欄位包含：

  • id：變數名稱。
  • type：string、number、boolean、color、select 等。
  • label：給人或 agent 看的說明。
  • default：預設值。
  • options：select 類型的選項。

變數覆寫來源通常有優先序：CLI --variables 最高，其次 variables file、composition default，最後落回宣告的 default。

這讓 HyperFrames 很適合：

  • 每週知識庫摘要影片。
  • 多客戶版本的產品 demo。
  • 課程單元宣傳短片。
  • 活動倒數與講者介紹。
  • 資料驅動的圖表動畫影片。

Agent Workflow：不要直接叫 AI render，先叫它規劃

PDF 對 agent 操作的提醒非常實用：永遠先載入 HyperFrames skill，再讓 agent 進入影片製作流程。不要只說「幫我做一支影片」，而要要求它先產出 storyboard、frame plan 與 composition contract。

建議提示骨架：

```text

Use /hyperframes to create a 20-second 16:9 explainer video.

First produce a storyboard, scene timing table, visual direction, and frame.md plan.

Use premium dark editorial style, strong Traditional Chinese typography, high contrast, and mobile-safe captions.

Do not render yet. After the plan is checked, write valid HyperFrames HTML, run lint, preview, take snapshots, then render.

```

適合欒帥的繁中版本：

```text

請使用 /hyperframes，把這篇 AI 學習雷達文章改成 20 秒 16:9 解說短片。

先不要渲染。請先輸出 storyboard、每幕秒數、畫面構圖、字幕文案、動效方向與 frame.md。

風格：高質感深色 editorial、繁中字體清楚、手機觀看安全、留白充足、CTA 明確。

確認後再撰寫 HyperFrames HTML，執行 lint、preview、snapshot，最後才 render MP4。

```

七個實作原則

  • 每支影片先定義比例、長度、fps、平台與受眾。
  • 每個 scene 都要有明確 start、duration、track 與目的。
  • 每個可視元素都要確認是否需要 class="clip"。
  • 所有動畫 runtime 都要可 seek，不要依賴 setTimeout 或 real-time playback。
  • 先 lint，再 preview，再 snapshot，最後 render。
  • 每個場景都要有進場與退場，不要讓元素突然出現或消失。
  • 把 prompts、storyboard、frame.md、composition HTML、assets、render output 都保留下來，才能重現與迭代。

常見反模式與修法

反模式一：把 HyperFrames 當 React / Vue app

HyperFrames composition 是 HTML + data attributes + 可 seek runtime，不是一般 SPA。不要讓 agent 產生過度複雜的框架依賴，除非已確定 render pipeline 能穩定載入。

反模式二：動畫自己播放

如果動畫依賴瀏覽器時間自行播放，render seek 到某個 frame 時可能狀態不一致。修法是使用 GSAP paused timeline 或其他可 seek runtime adapter。

反模式三：時間軸太短

影片 data-duration 是 30 秒，但 GSAP timeline 只有 8 秒，後段可能停住或被截斷。修法是延長 timeline 到影片總長。

反模式四：忘記 class="clip"

需要依時間出現的元素若沒有 class="clip"，可能全程可見。修法是補上 clip class 並設定 data-start / data-duration。

反模式五：同軌 clip 重疊

同一 track 的 clip 若重疊，可能造成解析或視覺意義混亂。修法是分軌或調整時間。

反模式六：一開始就做長片

新手不要從 60–180 秒開始。先做 10–20 秒、3–5 scene 的短片，比較容易建立穩定 loop。

Local、Lambda、Cloud Render 怎麼選

Local Render

適合開發迭代。優點是啟動快、不需要雲端基礎設施，最適合 preview、snapshot、短片初稿。缺點是跨機器輸出可能因字體與環境不同而有差異。

Lambda Render

適合已有 AWS 基礎設施、需要分塊平行渲染或大量批次輸出的團隊。優點是可擴展，缺點是要管理 AWS、權限、儲存、佇列與成本。

Cloud Render

適合不想管理 Chrome、FFmpeg、容器或 AWS 的使用者。由 HeyGen / HyperFrames 雲端跑算繪，通常按 credit 或平台方案計費。適合把 render 從本機移出，但要注意成本、隱私、素材授權與上傳資料。

與 Remotion 的比較

HyperFrames 與 Remotion 都屬於 code-to-video / programmatic video 的脈絡，但下注不同。

| 面向 | HyperFrames | Remotion |

| --- | --- | --- |

| Authoring | HTML + CSS + JS + GSAP | React / TSX |

| Agent 友善度 | LLM 更容易直接產生 HTML 與 data attributes | 需要 React 元件、型別與 build guardrails |

| 設計來源 | DOM 可成為 visual editor / agent 的共同事實來源 | React component tree 是主要事實來源 |

| 成熟度 | 新興、agent-native、快速演化 | 生態較成熟、production use case 多 |

| 適合場景 | AI agent 生成短片、HTML-native motion、快速模板化 | React 團隊、型別安全、既有 component system |

判斷上不必把兩者當成單純替代品。若團隊已有成熟 React 設計系統，Remotion 仍很強；若目標是讓 agent 快速理解、修改、渲染影片，HyperFrames 的 HTML-first 方向很有吸引力。

官方套件與架構角色

PDF 提到 HyperFrames 生態可拆成幾個層次：

  • @hyperframes/core：型別、parser、generator、adapter、composition contract。
  • @hyperframes/engine：執行與 seek runtime，讓頁面能在指定時間點呈現正確狀態。
  • @hyperframes/producer：完整 HTML 到影片管線，含 encoding、音訊混合、Docker / render orchestration。
  • NLE / Studio 元件：如 timeline、preview、source editor、property panel、file tree、agent activity track 等，可用於建立 hosted authoring workflow。

這代表 HyperFrames 不只是 CLI；它也能成為更大影片編輯器、agent 工作台或自動化內容平台的底層 render core。

可直接落地的 AI 學習雷達流程

流程 A：知識庫文章轉短影音

  • 選一篇 AI 學習雷達文章。
  • 擷取一句話摘要、三個重點、兩個應用場景、一個 CTA。
  • 產生 15–25 秒 storyboard。
  • 寫 frame.md：比例、字級、色彩、鏡頭語言、字幕規則。
  • 用 /hyperframes 產生 HTML composition。
  • lint / preview / snapshot。
  • render MP4。
  • 把影片與原文 URL 一起發佈。

流程 B：課程宣傳短片

  • 輸入課程名稱、講師、日期、三個學習成果。
  • 用變數控制講師、時間、地點、CTA。
  • 套同一份品牌模板批次產出橫版與直版。
  • render 16:9、9:16、1:1 三種尺寸。
  • 存回課程資料夾與 KB 文章。

流程 C：產品 demo / feature launch

  • 用 capture 擷取產品頁、dashboard 或 design tokens。
  • 產出 before / after 或三段式功能解說。
  • 加入字幕、旁白、highlight box、cursor movement。
  • 用 snapshot 做視覺 QA。
  • render 成社群短片或 landing page hero video。

快速檢查清單

開始前：

  • Node.js 22+ 是否可用。
  • FFmpeg 是否可用。
  • 專案是否有固定字體或 fallback。
  • 目標比例與輸出平台是否明確。
  • 是否已載入 HyperFrames skills。

寫 composition 時：

  • root composition 是否有 id、width、height、duration、fps。
  • 每個 clip 是否有 start、duration、track。
  • 需要 timing 的元素是否有 class="clip"。
  • 動畫是否可 seek。
  • 音訊是否有 volume、media-start 與對齊策略。

輸出前：

  • lint 是否通過。
  • preview 是否能看完。
  • snapshot 是否涵蓋關鍵秒數。
  • 字幕是否手機可讀。
  • CTA 與原文連結是否正確。
  • render output 是否可播放。

對欒帥的採用建議

建議把 HyperFrames 放在「AI 學習雷達內容再製」而不是「一般影片生成」的位置：

  • 短期：挑 1 篇高價值 KB 文章，做 20 秒橫版 demo，驗證 agent loop。
  • 中期：建立 AI 學習雷達固定片頭、片尾、卡片版型與字幕規則。
  • 長期：把 weekly digest 自動轉成短影音草稿，讓 agent 每週產生 storyboard、composition 與 preview，再由人類審美把關。

最值得先做的不是追求大片，而是固化一條可重複管線：Markdown article → storyboard → frame.md → HyperFrames HTML → lint → preview → snapshot → render → 發佈。
