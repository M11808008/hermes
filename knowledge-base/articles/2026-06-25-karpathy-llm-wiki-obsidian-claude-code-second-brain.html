Karpathy LLM Wiki｜用 Obsidian 與 Claude Code 建立可累積的 AI 第二大腦

Rafael Estrela 的 X 貼文用葡萄牙文整理了 Andrej Karpathy 的 LLM Wiki 想法：不要只把 AI 當成寫程式工具，而是把 Claude Code 這類 coding agent 連到一個資料夾，讓它閱讀文章、逐字稿、PDF、筆記，整理成一套可查詢、可交叉連結、可持續更新的第二大腦。

Avid 後續也用英文 X 貼文把這件事包裝成「15 分鐘建立 AI second brain」：下載 Claude Desktop、Obsidian Desktop、建立 vault、放入 Markdown 檔，並用 Karpathy 的 prompt 讓 Claude Code 連到 vault。這則補充來源凸顯同一個趨勢：Claude 不只是搜尋引擎，而是可以疊在自己所有筆記與資源之上的 intelligence layer。

這與傳統 RAG 不同：不是每次查詢都把資料切 chunk 丟進向量庫，而是先讓 LLM 把知識「編譯」成維基頁面。之後查詢時，agent 先讀 index 與相關條目，再回答問題。

一句話摘要

LLM Wiki 的核心是：raw sources 保留原始資料、wiki pages 存放 AI 編輯後的知識條目、CLAUDE.md / AGENTS.md 定義維護規則，讓個人知識不再停留在一次性聊天，而能長期累積。

架構快照

| 層級 | 角色 | 說明 |

|---|---|---|

| raw / sources | 原始資料層 | 放文章、PDF、逐字稿、書摘、網頁、截圖說明；原則上不讓 LLM 改動 |

| wiki | 編譯知識層 | LLM 產生主題頁、人物頁、概念頁、摘要、索引、交叉連結 |

| schema | 治理規則層 | 用 CLAUDE.md / AGENTS.md 定義命名、引用、更新、矛盾標記、查詢流程 |

| agent | 維護者 | Claude Code、Codex、Hermes、Cursor 等可讀檔與改檔的 agent |

| Obsidian | 人類介面 | 讓使用者用圖譜、wikilink、搜尋與筆記介面瀏覽知識 |

X 貼文整理的操作流程

Rafael Estrela 貼文給出的流程大意是：

  • 安裝 Obsidian，建立一個 Vault。
  • 用 Claude Code 開啟這個 Vault 或資料夾。
  • 放入 Karpathy 的 LLM Wiki 設定檔／wiki 範式。
  • 讓 Claude 建立三個結構：原始資料資料夾、wiki 頁面資料夾、管理規則檔。
  • 把文章、PDF、逐字稿、筆記等資料放到原始資料夾。
  • 對 Claude 說「分析並整理這些內容」。
  • 之後即可針對已存資料提問，並讓系統維持連結與可搜尋性。

與既有 n8n 版 LLM Wiki 的差異

AI 學習雷達已收錄「n8n 版 LLM Wiki｜用 Google Docs / Sheets 實作 Karpathy 預編譯知識庫」。本篇的差異是：

| 面向 | n8n / Google Docs 版 | Obsidian / Claude Code 版 |

|---|---|---|

| 目標使用者 | 低程式門檻、自動化平台使用者 | 會用本機資料夾、Markdown、coding agent 的知識工作者 |

| 儲存格式 | Google Docs / Sheets | Markdown + Obsidian vault |

| Agent 角色 | n8n workflow orchestrates LLM | Claude Code / Codex / Hermes 直接讀寫 vault |

| 可攜性 | 綁 Google 文件與 n8n 節點 | 純文字 Markdown，可放 Git、Dropbox、iCloud、NAS |

| 治理方式 | 工作流節點與表格索引 | CLAUDE.md / AGENTS.md / wiki schema |

為什麼這對欒帥重要

  • **AI 學習雷達可以變成 raw source 層**：外部連結、文章、PDF、影片逐字稿都先進 raw。
  • **公開知識庫是 wiki 層**：整理過的 Markdown 文章就是可讀、可部署、可搜尋的編譯知識。
  • **Hermes / Codex / Claude Code 是維護 agent**：負責查重、補來源、更新 metadata、重建索引、部署驗證。
  • **Obsidian 可作為私有深度筆記層**：不公開的課程規劃、名單、講義草稿可留在本機 vault。
  • **它避免每次從零開始問 AI**：重複概念先編譯成頁面，後續 agent 讀既有脈絡即可。

導入建議

| 階段 | 做法 |

|---|---|

| 1. 建資料夾規則 | raw、articles/wiki、assets、private、CLAUDE.md / AGENTS.md |

| 2. 定義 metadata | title、source、date、tags、importance、summary、canonical-url |

| 3. 建查重規則 | 同主題優先更新 canonical article，不重複開新檔 |

| 4. 建驗證規則 | 外部連結、GitHub metadata、部署 200 OK、敏感資訊遮蔽 |

| 5. 建輸出規則 | 公開知識庫、私密 Obsidian、課程講義、社群貼文各自不同格式 |

注意事項

  • LLM Wiki 不是免審稿工具；agent 會整理知識，但引用、事實與矛盾仍需抽查。
  • raw sources 應保留原樣，避免 LLM 改壞原始證據。
  • 若含個資、報名名單、客戶資料，不應放到公開部署資料夾。
  • Obsidian vault 若同步到雲端，需確認權限與備份策略。
