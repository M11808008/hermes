Codex AGENTS.md 專案規則檔｜讓 coding agent 自動讀取團隊慣例

來源：伊森說 Threads 貼文提醒使用 ChatGPT Codex 時，應替專案製作 `AGENTS.md`，把重要專案規則寫進去，避免每次換聊天視窗都重新提醒。貼文延伸連到科技兔 YouTube 教學〈ChatGPT Codex 的 AGENTS.md 檔案是什麼？怎麼用？讓 Codex 永久記住專案規則〉。本文用 OpenAI 官方文件與 agents.md 開放格式交叉查證。

一句話摘要

`AGENTS.md` 可以理解成「給 coding agent 看的 README」：把安裝、測試、程式風格、安全邊界、PR/commit 慣例與專案特殊限制寫成可自動讀取的規則，讓 Codex 每次開始工作前先載入同一套上下文，減少重複提示與 agent 自行猜測。

官方查證重點

| 重點 | 查證結果 | 實務意義 |

|---|---|---|

| Codex 會讀 `AGENTS.md` | OpenAI 官方文件明確寫：Codex reads `AGENTS.md` files before doing any work。 | 專案規則可以常駐，不必每次開新任務重講。 |

| 可分全域與專案層級 | OpenAI docs 說 Codex 會建立 instruction chain：全域 `~/.codex/AGENTS.md` 或 `AGENTS.override.md`，再往專案目錄讀取規則。 | 個人偏好放全域；repo 特定規則放專案根目錄。 |

| 可用 override 覆蓋 | OpenAI docs 提到 `AGENTS.override.md`，可在全域或子目錄層級覆蓋。 | 適合付款、資安、部署、測試等高風險子系統設定更嚴格規則。 |

| agents.md 是跨 agent 開放格式 | agents.md 官方站把它描述為 simple, open format for guiding coding agents，並稱已被大量開源專案使用。 | 不只 Codex，未來也可被其他 coding agent 讀取。 |

| OpenAI 自己 repo 也使用 | openai/codex repo 有自己的 `AGENTS.md`，內含 Rust crate、fmt、lint、測試與禁止修改 sandbox 相關常數等規則。 | 最佳實例是把「人類 reviewer 會提醒的事」前置到 agent 規則中。 |

建議寫入 AGENTS.md 的內容

1. 專案啟動與測試指令

  • 安裝依賴：例如 `pnpm install`、`uv sync`、`bundle install`。
  • 開發伺服器：例如 `pnpm dev`。
  • 測試與 lint：例如 `pnpm test`、`pytest`、`ruff check`、`cargo test`。
  • 哪些測試一定要跑、哪些測試耗時只能背景跑或交給 CI。

2. 程式風格與架構慣例

  • 語言版本與 formatter。
  • 命名慣例、目錄分層、API route 規範。
  • 禁止或偏好的套件與 pattern。
  • 資料庫 migration、schema 修改、feature flag 規則。

3. 安全與權限邊界

  • 不可提交 secrets、API keys、token。
  • 不可直接改 production config。
  • 涉及刪除、付款、寄信、大量外部呼叫時要先停下來回報。
  • 使用者資料、個資或課程私密資料不可外流。

4. 驗收與交付格式

  • 完成後要回報改了哪些檔案、跑了哪些測試、哪些沒跑與原因。
  • PR/commit 訊息格式。
  • 文件更新與 changelog 規則。

欒帥可用模板

```md

AGENTS.md

Project overview

  • 這個專案的用途：...
  • 主要技術棧：...

Setup commands

  • Install deps: `...`
  • Run dev server: `...`
  • Run tests: `...`

Code style

  • ...

Safety rules

  • 不要提交 secrets 或修改 production credentials。
  • 會刪除資料、寄送訊息、收費或部署前，先停下來說明風險。

Verification

  • 修改程式後至少執行：`...`
  • 若測試無法執行，需說明原因與替代驗證。

```

實務提醒

  • `AGENTS.md` 不是萬能記憶；它是「專案預設規則」。臨時需求、當前任務目標、最新 bug context 仍要在 prompt 裡交代清楚。
  • 規則不要寫得太長，否則 agent 會被噪音干擾；把高頻、穩定、會重複提醒的內容留下即可。
  • 大型 monorepo 可在子目錄放更接近該服務的規則，避免所有 agent 任務都背整本規格書。
  • 規則變動後，要讓下一個 Codex session 重新啟動或確認已重新讀取。

Why it matters

對 AI coding workflow 來說，`AGENTS.md` 是低成本但高槓桿的 context engineering：它不能取代測試與 code review，但能顯著降低 agent 開局犯錯、反覆問同樣問題、忘記專案慣例的機率。越常用 Codex 的團隊，越應該把口頭提醒沉澱成 `AGENTS.md`。
