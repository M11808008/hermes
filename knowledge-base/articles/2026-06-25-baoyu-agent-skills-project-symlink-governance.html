宝玉 Agent Skills 管理法｜用專案內安裝與軟連結治理 Claude / Codex 技能庫

來源是宝玉在 X 的長文，回應 LinearUncle 提到同時使用多個 coding agent 時，`/.agents/skills`、`/.codex/skills`、`~/.claude/skills` 各自分裂、互相找不到技能的痛點。本文保留原貼的工程化做法，並對照 OpenAI Codex 與 Claude Skills 官方文件，整理成適合欒帥日後治理個人 agent skill library 的決策筆記。

一句話摘要

這則貼文值得收錄的重點不是「又多一個 skills 安裝技巧」，而是提出一個可長期維護的技能庫治理原則：技能原件集中管理、專案內按需啟用、用軟連結跨工具暴露入口，避免全域技能污染上下文，也讓更新與修 bug 可以回饋原始 repo。

核心做法

1. Skills 只裝在專案裡，不裝全域

宝玉的理由是上下文管理。Agent Skills 通常採用 progressive disclosure：一開始只把名稱、描述、路徑等摘要放進上下文，等 agent 判斷相關時才載入完整 `SKILL.md`。但全域技能太多時，光是摘要清單就會擠壓工作台；若誤觸發完整技能內容，也會浪費更大的上下文空間。

因此做法是：

  • 不把所有技能塞進全域位置。
  • 每個專案只連入真正會用到的 skills。
  • 專案上下文保持乾淨，降低錯誤觸發與 token 浪費。

這與 OpenAI Codex 文件提到的機制一致：Codex 會從 repository、user、admin、system 等位置讀取 skills；repository 位置會掃描 `.agents/skills`。OpenAI 也明確說 Codex 會先放入技能摘要，並在需要時載入完整 `SKILL.md`，因此大量安裝仍會產生上下文治理成本。

2. 技能原件集中在 GitHub 專案資料夾

贴文中的範例是把開源 skill 專案集中放在：

```text

~/GitHub/baoyu-skills

~/GitHub/baoyu-design

```

這裡存放的是「技能原件」，不是每個專案都複製一份。這個分層很重要：

| 層級 | 角色 | 好處 |

|---|---|---|

| `~/GitHub/<skill-repo>` | 技能原件 / upstream repo | 可以 `git pull` 更新，也能直接 commit 修正 |

| 專案內 `.agents/skills/<skill>` | 專案按需入口 | 只讓當前專案看到需要的技能 |

| `.claude/skills` | Claude Code 入口 | 讓 Claude Code 沿著入口找到同一批技能 |

3. 用軟連結而不是複製貼上

貼文範例：

```text

.agents/skills/baoyu-comic -> ~/GitHub/baoyu-skills/skills/baoyu-comic

.agents/skills/baoyu-design -> ~/GitHub/baoyu-design/skills/baoyu-design

.claude/skills -> .agents/skills

```

軟連結的價值是「一份原件，多個入口」。改原件，所有專案入口同步更新；在專案中請 agent 修 skill，實際修到的也是原本的 repo，後續可以直接提交 PR 或回饋開源。

對欒帥的實用採用建議

建議採用的目錄模型

可以把這則貼文的做法整理成三層：

```text

~/GitHub/agent-skills/ # 各種公開或自製技能原件

~/GitHub/<project>/.agents/skills/ # 每個專案真正啟用的技能集合

~/GitHub/<project>/.claude/skills # 指向 .agents/skills 的 Claude Code 入口

```

如果同時用 Codex、Claude Code、Hermes 或其他 coding agent，核心原則不是讓每個工具各自複製一份，而是建立一個「專案技能入口」，再依各工具的規則掛過去。

適合放專案內的技能

| 技能類型 | 是否適合專案內啟用 | 原因 |

|---|---|---|

| 該專案專用部署流程 | 高 | 路徑、環境變數、測試命令通常只對該 repo 有效 |

| 特定設計系統 / 品牌聲線 | 高 | 避免污染其他專案輸出風格 |

| 課程頁、簡報、知識庫固定格式 | 高 | 可重用但通常有專案脈絡 |

| 通用查詢 / 搜尋技能 | 中 | 若常用可全域，但摘要太多時仍建議精簡 |

| 一次性實驗技能 | 低 | 先放 sandbox，成熟後再納入正式技能庫 |

可直接交給 agent 的自然語言指令

宝玉強調不用記 `ln -s` 指令，直接交代 agent 即可。欒帥可以使用這種說法：

```text

請把 ~/GitHub/agent-skills/skills/<skill-name> 軟連結到目前專案的 .agents/skills/<skill-name>，如果 .agents/skills 不存在就先建立；再讓 .claude/skills 指向 .agents/skills。請不要複製資料夾，使用 symlink，並檢查連結是否有效。

```

如果要更穩妥，可以再加：

```text

建立前請先檢查是否已有同名 skill；若已存在，先回報它是真實資料夾還是 symlink，不要覆蓋。

```

風險與治理 caveats

| 風險 | 判讀 |

|---|---|

| 軟連結指到壞掉的路徑 | repo 搬移或刪除後，專案會看到 dangling symlink；需要定期檢查 |

| 不同工具的 skill 規格不完全一致 | `SKILL.md` 是共同核心，但 frontmatter、支援檔案、觸發規則與工具限制可能不同 |

| 修原件會影響多個專案 | 這是優點也是風險；重大修改前最好 git branch 或 commit checkpoint |

| 全域技能仍有用途 | 真正每天都用、低風險、短描述的技能可以全域；但不應把全域當垃圾桶 |

| 安全與資料邊界 | 從公開 repo 安裝 skills 前應讀 `SKILL.md`、scripts、templates，避免讓 agent 執行不明腳本 |

官方文件對照

  • OpenAI Codex Skills：Codex 支援 repository / user / admin / system 等不同 scope；repository 會掃描 `.agents/skills`，並以 progressive disclosure 管理上下文。
  • Claude Code Skills：Claude Code 支援 `.claude/skills/<skill>/SKILL.md`，技能可在相關時自動載入，也可用 `/skill-name` 直接呼叫。
  • Claude Agent Skills API 文件：Anthropic 強調 filesystem-based architecture 與 progressive disclosure，讓技能的 metadata、instructions、supporting files 分階段載入。

這些官方設計都支持宝玉的結論：skills 不是越多越好，而是要用 scope、目錄與入口來治理。

欒帥可落地的下一步

  • 盤點目前真正高頻使用的 Skills，分成全域、專案、實驗三類。
  • 對每個專案建立 `.agents/skills` 作為共同入口。
  • 只把該專案需要的技能用 symlink 連入。
  • 對 Claude Code 建 `.claude/skills -> .agents/skills`。
  • 對 Codex 則優先使用官方文件建議的 repository `.agents/skills` scope。
  • 每次更新公開技能 repo 前先 commit；每次修技能後，把 bug fix 回饋到原 repo。
