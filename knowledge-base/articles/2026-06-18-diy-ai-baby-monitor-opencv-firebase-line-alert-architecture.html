自製智慧寶寶攝影機架構｜OpenCV / YOLO、Firebase 與 LINE 告警的準爸爸工程專案

來源為 GenAI & Agentic AI 論壇中的工程筆記。作者從市售 Cubo Ai 類智慧寶寶攝影機出發，思考是否能用 Python 影像處理、OpenCV、AI 偵測、Firebase 與 LINE Messaging API，自製一套可客製化的寶寶攝影機系統。這篇的價值不是替代市售安全產品，而是把「AI 協助拆架構」的工程思路清楚寫出來。

一句話摘要

自製智慧寶寶攝影機的合理 MVP 是把「即時觀看」與「AI 偵測 / 告警」分成兩條線：影像線由攝影機 → 串流服務 → 手機觀看；資料線由 Python / OpenCV / YOLO 偵測事件 → Firebase 記錄狀態與截圖 → LINE 發出告警。

貼文中的架構

| 層級 | 技術選項 | 角色 |

|---|---|---|

| 攝影機來源 | USB Camera、IP Camera、Pi Camera | 取得即時影像 |

| 串流服務 | Flask MJPEG、RTSP、WebRTC、MediaMTX、go2rtc | 將畫面提供給手機或網頁觀看 |

| AI 偵測 | Python、OpenCV、YOLO | 判斷寶寶翻身、臉遮住、長時間不動、畫面異常等事件 |

| 中控資料層 | Firebase Auth、Firestore、Storage、Functions | 登入權限、事件紀錄、狀態同步、截圖 / 短片保存、觸發流程 |

| 告警 | LINE Messaging API | 異常時推播通知照護者 |

兩條線拆開是關鍵

| 線路 | 流程 | 重點 |

|---|---|---|

| 影像線 | 攝影機 → 串流服務 → 手機觀看 | 低延遲、穩定、不中斷 |

| 資料線 | Python AI 偵測 → Firebase → 手機 / LINE 告警 | 紀錄、同步、權限、事件通知 |

這個拆法很實用，因為 Firebase 不適合承載即時影像串流本體；它更適合做身份驗證、事件資料、截圖、狀態同步與觸發 Functions。即時影像則交給 MediaMTX、WebRTC、RTSP 或簡化版 Flask MJPEG。

建議 MVP 路線

| 階段 | 目標 | 驗證標準 |

|---|---|---|

| 1. 即時畫面 | 手機 / 網頁能看到攝影機畫面 | 同一 Wi-Fi 下穩定觀看 30 分鐘 |

| 2. Python 抓畫面 | OpenCV 能讀取影格 | 每秒固定抽 frame，不影響觀看串流 |

| 3. 基礎事件 | 先做畫面中斷、長時間無動靜、亮度異常 | 可降低誤報，先避開高難度姿態判斷 |

| 4. YOLO / 姿態偵測 | 偵測寶寶位置、遮臉風險、翻身方向 | 用離線影片測試 precision / recall |

| 5. Firebase 記錄 | 事件寫入 Firestore、截圖進 Storage | 可回看時間、事件類型、截圖 |

| 6. LINE 告警 | 異常事件推播 LINE | 告警內容含事件、時間、截圖連結 |

| 7. 安全與權限 | 登入、權限、內網 / HTTPS、資料保存策略 | 只有授權家人可看，影像不公開暴露 |

對欒帥的實用價值

  • **AI 工程教學案例**：很適合用來示範如何讓 AI 協助拆一個跨硬體、串流、後端、AI、通知的系統架構。
  • **IoT × AI Agent 課程素材**：可拆成多個可實作單元：MJPEG streaming、OpenCV frame capture、YOLO inference、Firebase event log、LINE push。
  • **產品思維案例**：市售產品勝在可靠、易用、支援與安全；自製產品勝在可客製、可研究、可調整模型與事件邏輯。
  • **架構設計練習**：這篇可作為「先拆線路，再定 MVP，再加 AI」的範例，避免一開始就追求完整智慧攝影機。

安全與倫理提醒

  • **不可把自製系統當成唯一照護保障**：嬰幼兒安全是高風險場景，自製 AI 可能漏報、誤報或當機。
  • **先做非關鍵輔助功能**：例如事件紀錄、畫面中斷通知、照護日誌，不要一開始就承諾遮臉 / 窒息風險偵測。
  • **影像隱私要優先**：寶寶影像、家庭畫面、截圖與通知連結都應有權限控管與保存期限。
  • **夜視、遮擋、光線與角度會大幅影響模型**：任何 AI 判斷都需要用實際環境影片測試，不應只用 demo 驗證。
  • **LINE 告警要設計節流**：避免一個事件連續推播造成通知疲勞；也要保留心跳檢查，避免系統壞了卻沒有告警。
