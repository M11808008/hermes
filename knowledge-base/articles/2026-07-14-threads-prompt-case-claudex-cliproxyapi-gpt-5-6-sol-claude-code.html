設計核心：不要改壞原本的 Claude Code

這份 prompt 最值得保存的不是單一命令，而是它對「安全改設定」的要求很完整：

  • 建立永久可用的新指令 `claudex`。
  • 原本 `claude` 指令與設定不得被覆蓋。
  • shell 設定檔修改前必須備份。
  • 若 alias / function 已存在，需更新而不是重複追加。
  • OAuth token、認證檔案與敏感資料不得輸出。
  • 完成後需驗證 proxy、port、alias、模型與普通 `claude` 是否仍未被污染。

這很適合作為「讓 Agent 修改本機開發環境」時的提示詞範本：不是只叫它安裝，而是要求它偵測、備份、最小影響、驗證與回報。

原文 prompt 的目標結構

Prompt Case 留言中的 prompt 可整理為以下任務規格：

### 目標

  • 建立永久可用的 `claudex` 指令。
  • 執行 `claudex` 時，Claude Code 透過本機 CLIProxyAPI 使用 `gpt-5.6-sol`。
  • 原本 `claude` 指令和設定不受影響。

### 主要步驟

  • 自動偵測作業系統、CPU 架構與 shell，需支援 macOS、Linux、Windows PowerShell。
  • 檢查 Claude Code、CLIProxyAPI 與作業系統套件管理工具是否已安裝。
  • 若尚未安裝 CLIProxyAPI，使用該作業系統適用的官方方式安裝。
  • 啟動 CLIProxyAPI，設定為本機服務並監聽 `http://127.0.0.1:8317`。
  • 執行 Codex OAuth 登入；需要使用者瀏覽器授權時，清楚等待使用者完成。
  • 設定本機認證值 `sk-dummy`。
  • 建立永久 `claudex` 指令；修改 shell 設定前先備份。
  • bash / zsh 使用 alias；PowerShell 使用 function，且執行完需恢復原環境變數。
  • 驗證 CLIProxyAPI 運行、port 可連、`claudex` 存在、模型為 `gpt-5.6-sol`，且普通 `claude` 未被改寫。

技術脈絡：為什麼這件事重要？

這篇貼文背後的趨勢是 AI coding stack 的拆層：

  • Claude Code：負責 agent loop、工具調度、檔案修改、互動式開發體驗。
  • CLIProxyAPI：負責把不同 CLI / OAuth / API 介面轉成相容端點。
  • GPT-5.6 Sol：作為實際推理與 coding 模型後端。
  • alias / function：把上述設定包成可重複啟動的本機指令。

換句話說，開發者開始不再把「coding agent 工具」與「模型供應商」綁死，而是把 harness、proxy、模型、shell 啟動器組合成自己的本機工作流。

風險與實務提醒

這類 proxy / OAuth / alias 工作流雖然方便，但實作時應特別注意：

  • 不要把 OAuth token、cookie、認證檔案或 access token 貼到聊天、GitHub 或公開文件。
  • 任何修改 `~/.zshrc`、`~/.bashrc`、PowerShell `$PROFILE` 的操作都應先備份。
  • 應建立獨立指令，例如 `claudex`，不要覆蓋 `claude`。
  • 本機 proxy port 只應綁定 `127.0.0.1`，避免暴露到區網或公網。
  • 需要留意 Claude Code、OpenAI / Codex、CLIProxyAPI 各自的服務條款與帳號風險。
  • 若只是實驗，建議先在可還原的 shell profile 或測試帳號中跑通。
