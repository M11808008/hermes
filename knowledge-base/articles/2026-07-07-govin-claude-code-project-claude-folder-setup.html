Govin.AI｜如何正確設定 Claude Code 專案：把 .claude/ 當作基礎建設

一句話摘要

Claude Code 的生產力不只來自模型，而是來自專案內的 `.claude/` 基礎建設：把規則、指令、技能、子代理與權限設定好，才不會每次都靠臨場 prompt 補救。

Threads 貼文重點

Govin.AI 分享「如何正確設定 Claude Code 專案」：多數開發者跳過設定就直接開始下指令，這是最大的錯誤。

貼文指出，一個完整的 Claude Code 專案核心在 `.claude/` 資料夾：

  • 先寫 `CLAUDE.md` 作為操作手冊。
  • 專案長大後，再把規則拆到 `rules/`。
  • 用 `commands/` 放重複性工作流程。
  • 用 `skills/` 做情境觸發自動化。
  • 用 `agents/` 放隔離子代理。
  • 用 `settings.json` 鎖好權限。

另有兩層 `.claude/`：一個跟著 repo 提交，另一個全域 `~/.claude/` 放個人偏好與跨專案自動記憶。貼文的核心觀點是：`.claude/` 不是雜物資料夾，而是 coding agent 的基礎建設。

建議的 Claude Code 專案結構

可把專案設計成：

```text

project/

CLAUDE.md

.claude/

settings.json

settings.local.json

commands/

test.md

review.md

deploy.md

skills/

database-migration.md

api-contract.md

agents/

security-reviewer.md

frontend-reviewer.md

rules/

backend.md

frontend.md

testing.md

```

實際路徑仍需依 Claude Code 版本與官方文件調整；但這個架構思路是正確的：把常態規則與工作流放進檔案，而不是每次在對話中重講。

各元件的角色

1. CLAUDE.md：專案操作手冊

`CLAUDE.md` 是 Claude Code 進入專案後最重要的上下文。建議包含：

  • 專案目的與產品背景
  • 架構概覽
  • 重要資料夾與檔案
  • 本地開發、測試、lint、build 指令
  • code style 與命名規範
  • 安全限制與禁止事項
  • PR / commit 規則
  • 常見陷阱與已知問題

重點是「具體可執行」，不要只寫「請寫好程式」。例如應寫：測試用 `pnpm test`、API schema 在 `packages/api/schema`、修改 migration 後需跑 `pnpm db:check`。

2. rules/：大型專案的規則拆分

當 `CLAUDE.md` 變得太長，可把規則拆到 `rules/` 或等價的模組化文件：

  • backend.md
  • frontend.md
  • database.md
  • testing.md
  • security.md
  • design-system.md

這能降低單檔混亂，也方便團隊分工維護。

3. commands/：重複流程變 slash command

`commands/` 適合放重複性流程，例如：

  • `/test`：跑測試、整理失敗原因、修復後重跑。
  • `/review`：做 PR 前檢查，包含安全、效能、測試覆蓋。
  • `/deploy`：跑 build、檢查環境、生成部署摘要。
  • `/changelog`：根據 git diff 生成變更紀錄。

判斷標準：同一件事做超過三次、每次 prompt 超過五行，就值得做成 command。

4. skills/：情境觸發的自動化知識

Skills 與 commands 不同：commands 偏手動呼叫，skills 偏「任務符合時自動套用的程序知識」。例如：

  • database migration skill
  • API contract update skill
  • design token update skill
  • test-driven bugfix skill

好的 skill 應包含觸發條件、步驟、驗證方法與常見陷阱。

5. agents/：隔離子代理

`agents/` 適合把任務拆給專門角色：

  • security-reviewer：只看資安風險。
  • performance-reviewer：找效能瓶頸。
  • frontend-reviewer：檢查 UI / accessibility。
  • test-writer：補測試。
  • database-expert：看 schema、migration、query。

子代理的價值在於隔離視角與工具權限，不讓所有事情都塞給同一個主 agent。

6. settings.json：權限與安全邊界

`settings.json` 應管理 Claude Code 能做什麼：

  • 允許哪些工具與 shell command。
  • 哪些操作需要詢問。
  • 哪些檔案或命令禁止碰。
  • hooks 是否在讀寫前後自動跑檢查。

這對正式專案很重要：如果沒有設定，agent 很容易在高自治狀態下碰到不該碰的憑證、環境檔或破壞性指令。

repo .claude/ vs 全域 ~/.claude/

貼文提到有兩個層級：

repo 內 `.claude/`

適合提交到版本控制，讓團隊共享：

  • 專案規則
  • 團隊共用 commands
  • 專案專用 agents
  • 安全權限預設
  • hooks 與檢查流程

全域 `~/.claude/`

適合個人使用：

  • 個人偏好
  • 跨專案 commands
  • 常用 agents
  • 個人 memory / settings
  • 不應提交到團隊 repo 的私有設定

這個分層非常重要：團隊規則要進 repo，個人習慣要留在 user 層，避免把個人偏好污染專案。

對 AI 學習雷達的價值

這篇值得收錄，因為它把 Claude Code 從「聊天式 coding 工具」提升到「可配置的工程系統」。真正的差別不只是會不會叫 Claude 改檔，而是有沒有讓它在每次任務開始時自動知道：

  • 這個專案是什麼。
  • 該怎麼跑測試。
  • 哪些檔案不能碰。
  • 該用哪些流程完成任務。
  • 哪些檢查必須通過。
  • 哪些任務應交給專門 subagent。

這也是從 vibe coding 走向 agentic software engineering 的關鍵。

可轉成 Luarn-Pin 工作流的清單

新專案初始化時，可用以下 checklist：

  • 建立 `CLAUDE.md`，寫清楚專案目的、架構、指令、規範。
  • 建立 `.claude/settings.json`，先限制危險命令與敏感檔案。
  • 把常做流程整理成 `.claude/commands/`。
  • 把高頻但需判斷的任務整理成 `.claude/skills/`。
  • 把 review / security / tests 拆成 `.claude/agents/`。
  • 專案規則進 repo；個人偏好放 `~/.claude/`。
  • 每次任務完成後，把新發現的專案規則回寫到 `CLAUDE.md` 或對應 rule。

後續可追蹤

  • Claude Code 官方對 skills、agents、settings、hooks 的語法是否持續變動。
  • `.claude/` 結構如何與 MCP、CI、GitHub Actions、pre-commit 結合。
  • 團隊共用的 `.claude/` 是否需要 code review 與版本控管。
  • 是否可建立一套「Claude Code 專案初始化模板」給教學或顧問案使用。
