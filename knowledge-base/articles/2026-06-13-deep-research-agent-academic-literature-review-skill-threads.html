Deep Research Agent｜Hermes/ECC 學術文獻回顧 Skill 與零幻覺檢核流程

來源為 Tommy / 𝖄𝖚𝖖𝖎 在 Threads 分享的開源專案 Deep Research Agent。貼文將它描述為一套能在 Hermes / ECC 框架中使用的文獻回顧 skill，目標是減少 AI 寫學術報告時捏造論文、引用低品質期刊或只靠摘要過度推論的問題。本文以 Threads 貼文與 GitHub repo / README 作為依據，整理其架構、安裝需求、可用情境與導入限制。

一句話摘要

Deep Research Agent 是一個偏「學術研究生產線」的 agent skill：用 Scopus / OpenAlex / 搜尋 MCP 先做文獻發現，再要求全文驗證、DOI 可用性檢查、Q1-Q2 期刊篩選、Remi peer review、Word 文件編譯、Obsidian 與 NotebookLM 入庫；它很適合重度文獻回顧與研究草稿，但需要 Scopus key、MCP、Python/Node 環境與人工審核，不能當成完全免責的論文自動產生器。

7 階段流程拆解

| 階段 | README 描述 | 欒帥使用時的判斷 |

|---|---|---|

| Phase 0 / 0.5 | 先制定 search plan，等待使用者批准，再開背景程序抓取高影響力 papers | 適合保留人工確認點，避免 agent 一開始就大量跑 API / 花 token |

| Phase 1 Discovery | 鎖定 Q1-Q2，必要時標 Q3，禁止 Q4 與 MDPI | 適合高標準學術回顧；但不同領域對 MDPI 的接受度需人工判斷 |

| Phase 2 Deep Extraction | 彙整 metadata、全文與 key findings | 重點是全文而非只讀 abstract，可降低摘要誘發幻覺 |

| Phase 3 Structural Drafting | 用證據支持 claims，生成 APA 7 引文草稿 | 應要求每個主張回指來源段落或 DOI |

| Phase 4 / 4.5 Anti-Hallucination | 刪除 AI 用語，ping DOI；404 引文移除 | DOI 可解析不等於內容支持主張，仍需人工閱讀驗證 |

| Phase 5 Remi Review | 內建 peer-review loop，去除 AI fluff，提升學術語氣 | 可作為風格與邏輯 review，但不能取代真實指導教授或同儕審查 |

| Phase 6 Compilation | 用 Python 產生圖表並編譯 Word .docx | 對報告輸出很實用；需確認圖表資料來源與重現性 |

| Phase 7 Knowledge Graph | 寫入 Obsidian，並把引用文獻推到 NotebookLM | 適合建立個人研究知識庫，但會牽涉 Google 帳號與資料隱私 |

實作需求與摩擦點

| 類別 | README 要求 | 風險 / 注意 |

|---|---|---|

| Agent runner | Hermes Agent、ECC 或相容 Claude Code runner | 不同 runner 的 skill 格式、工具權限與背景程序支援可能不同 |

| Node.js | Node.js 18+、npx | MCP server 多半依賴 Node 生態 |

| Python | Python 3.10+ | 需安裝 python-docx、PyMuPDF、requests、matplotlib、seaborn、pandas |

| Scopus MCP | 需要 Elsevier Scopus API key | Scopus access 可能受機構訂閱與 API quota 限制 |

| NotebookLM MCP | 需 npx notebooklm-mcp-server auth | 會把文獻來源上傳到 Google NotebookLM，需確認資料授權與隱私 |

| Exa Search MCP | README 標示 Highly Recommended | 可能涉及額外 API key / 付費 |

| Obsidian | 預設找特定 vault path | README 目前提到 Windows 路徑 D:\\Tommy 與 Obsidian fallback，macOS 需調整 |

與 Consensus AI 的差異判讀

Consensus 偏向「產品化的論文搜尋與摘要介面」，Deep Research Agent 則偏向「可改造的 agent pipeline / skill」。後者的優勢是可自訂來源、期刊規則、審稿風格、輸出格式與知識庫連動；缺點是環境設定、API key、MCP、錯誤處理與驗證責任都落在使用者端。若只是快速查一個題目，Consensus 類產品摩擦較低；若要建立可審核、可重跑、可連進 Obsidian/NotebookLM 的研究工作流，這個 repo 比較有改造價值。

欒帥可用的導入策略

  • 先不要直接跑完整寫作流程，先用一個明確且小範圍的研究題目測試 Phase 0-2。
  • 驗證 Scopus / OpenAlex 的搜尋結果是否符合領域期待，尤其注意是否過度排除某些合法期刊。
  • 將 DOI 檢查升級成「DOI 可解析 + claim-to-source 對照」，不要只看 200/404。
  • 把輸出拆成三份：文獻清單、證據表、草稿；避免 agent 一步到位生成不可審核的長文。
  • 若要接 NotebookLM，先確認文獻 PDF/全文是否可合法上傳到個人 NotebookLM。
  • macOS 使用者需先改掉 README 中 Windows 專用輸出路徑與 Obsidian vault 路徑。

適合 / 不適合場景

| 適合 | 不適合 |

|---|---|

| 博論/碩論前期文獻盤點 | 需要立即交付且完全不能檢查來源的任務 |

| 研究計畫書背景章節 | 法律、醫療或政策決策的最終依據 |

| 跨資料庫文獻初篩 | 沒有 Scopus / MCP / API key 的零設定使用者 |

| 建立 Obsidian 研究知識庫 | 只能接受 SaaS UI、不想維護本機 agent 的使用者 |

| 需要 APA docx 與圖表輸出 | 需要中文正式論文全文輸出的場景；README 目前強制 English output |

判讀限制

Threads 貼文提到「1,400+ 篇大數據漏斗、兩名 AI 探員各 700+ 篇」，而 README 的架構說明中也提到先抓大量 abstracts、再深讀高相關全文，並在 Phase 0.5 抓取 exactly 30 high-impact papers。這些數字可能會隨 skill 設定與查詢主題改變；導入時應以 repo 實際 SKILL.md / README 版本與執行 log 為準，不應把社群貼文數字視為固定保證。

此外，DOI 可解析、期刊分區、排除 MDPI、AI peer review 都只能降低風險，不能保證結論正確。真正學術使用仍要保留人工閱讀、方法學評估、引用倫理與領域專家審核。
