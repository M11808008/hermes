一句話摘要

Hybride PageIndex RAG 是一套地端文件問答系統：上傳 PDF 後用 MinerU 解析表格、公式、圖說與 OCR，建立 PageIndex 式章節樹與頁碼範圍，再由 Agno Agent 以 Qdrant dense + BM25 + RRF 混合檢索命中章節，回 PostgreSQL 取頁面原文作答，並以串流聊天、引用頁碼、會話記憶與思考過程降低企業私有文件 RAG 的幻覺與外洩風險。

與一般 RAG 的差異

一般 RAG 常把 PDF 切成固定長度 chunks，再用向量相似度取 Top-K；這在 SOP、規格書、論文或跨頁表格中容易遇到三個問題：

  • 章節邊界被切斷，回答缺上下文。
  • 關鍵字、料號、錯誤碼不一定能靠純向量命中。
  • 命中 chunk 後不一定知道原文頁碼與脈絡。

Hybride PageIndex RAG 的重點是把文件先整理成「章節樹 + 頁碼範圍」，查詢時先定位到可能相關的章節，再回頁面原文作答。它不是只追求「搜到相似文字」，而是把檢索、頁碼定位、原文取證與回答可答性判斷串成 Agentic RAG 流程。

部署與測試路線

README 給出的啟動方式偏向 Windows / PowerShell 與本地服務組合：

  • 用 uv 安裝 Python 依賴，Python 版本鎖定 >=3.12,<3.13。
  • 啟動 FastAPI 後端，範例使用 127.0.0.1:8200。
  • 啟動 React / Vite 前端，範例使用 127.0.0.1:5173。
  • 選用 PostgreSQL 127.0.0.1:5433 與 Qdrant 6333。
  • 若要完整 Agent 對話能力，需開啟 USE DATABASE、USE QDRANT、USE AGNO，並配置 LM Studio / LiteLLM / embedding endpoint。
  • 若要 MinerU VLM / hybrid backend，需下載 MinerU2.5-Pro 模型。
  • 驗證包含 pytest、ruff、frontend test / build，以及 live e2e scripts。

對實作的判讀

這個 repo 的價值在於把「企業私有文件 RAG」的幾個關鍵環節都串起來：

  • 文件解析不是只抽純文字，而是把表格、公式、圖說與 OCR 納入頁面 Markdown。
  • 檢索不是純向量，而是 dense + BM25 + RRF，對料號、錯誤碼、規格值更友善。
  • 回答不是直接拿 Top-K chunk 生文，而是讓 Agent 先找章節，再取原文，再判斷是否足以回答。
  • 介面不是命令列 demo，而是有 React 聊天、歷史對話、串流回答與解析測試台。

對欒帥來說，這很適合納入「地端 AI / RAG / 企業知識庫」案例庫，未來可拆成課程中的一個完整範例：從 PDF ingestion 到 Agent trace，再到引用來源與隱私治理。
