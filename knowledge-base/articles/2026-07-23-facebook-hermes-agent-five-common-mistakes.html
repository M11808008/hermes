這篇很值得保存，因為它抓到 Hermes 新手最容易忽略的成本與品質來源：auxiliary models、SOUL.md、記憶與工具集設定；但其中有幾個指令/設定需要用官方文件與實際 CLI 行為校正，特別是 `hermes skills disable` 不是目前可用的 skills 子命令，應改用 `hermes skills config` 互動設定，或依需求用 `hermes skills uninstall` 移除 hub skill。

五個建議逐點查核

1. 不要讓主力模型包山包海：大方向成立

官方設定文件明確說明，Hermes 會用 auxiliary models 處理 image analysis、web page summarization、browser screenshot analysis、session-title generation、context compression 等側任務。官方也提醒：預設 `auxiliary.*.provider: "auto"` 會用主 chat model；如果主模型是昂貴 reasoning model，輔助任務會增加成本。

可採用的正確設定方向：

```yaml

auxiliary:

vision:

provider: openrouter

model: google/gemini-2.5-flash

web_extract:

provider: openrouter

model: google/gemini-2.5-flash

session_search:

provider: openrouter

model: google/gemini-2.5-flash

compression:

provider: openrouter

model: google/gemini-2.5-flash

```

或更穩妥：執行 `hermes model`，選 `Configure auxiliary models` 逐項設定。

貼文說「省下 30% 到 50% token」可作為經驗值，但不是官方保證；實際節省取決於使用場景、工具頻率、是否大量 vision/web_extract/compression、主模型價格與 context 長度。

2. SOUL.md 不能空：成立，但要區分用途

官方 SOUL.md 指南說：`SOUL.md` 是 Hermes instance 的 primary identity，會定義助理如何說話、避免什麼、預設人格與溝通風格。若 SOUL.md missing、empty 或無法讀取，Hermes 會 fallback 到 built-in default identity。

因此，貼文提醒「空 SOUL.md 會讓助理缺乏個性與規則」方向成立。更精準的建議是：

  • SOUL.md 放「助理人格、語氣、預設行為、工作原則」。
  • 具體專案規則放 AGENTS.md / CLAUDE.md / .cursorrules。
  • 使用者長期偏好與事實放 memory / user profile。
  • 不要把一時任務需求或易過期細節塞進 SOUL.md。

3. `/goal` 要有驗收標準：成立

`/goal` 是 persistent goals 機制，會讓 Hermes 持續推進直到目標判定完成或達到回合上限。貼文提醒「模糊目標容易讓 agent 亂跑或停不準」是正確的。

好的 goal 應包含：

  • 技術棧與範圍。
  • 明確交付物。
  • 驗收條件。
  • 可用工具/部署目標。
  • 停止條件，例如 URL 回傳 200、測試通過、檔案生成完成。

貼文提到 `max_turns: 10` 作為安全上限也合理；官方設定有 goal / agent 相關 max_turns，用來避免 persistent goal 無限延伸。

4. 定期檢查記憶檔：成立，但不是「定期刪除」

官方記憶文件的核心是：記憶應保存可長期重複利用的偏好、專案慣例、環境資訊，避免保存暫時任務進度、易過期細節或錯誤推論。貼文提醒檢查 `USER.md` / `MEMORY.md` 的方向成立。

但圖卡上「定期刪除舊記憶」容易被誤解。比較好的說法是：

  • 定期審核，而不是盲目刪除。
  • 刪掉錯誤、過期、一次性或太細碎的記憶。
  • 補上會降低未來溝通成本的穩定偏好。
  • 程序性知識應放 skill，不應放 memory。

5. 精簡工具與技能：方向成立；指令需校正

Hermes 的 toolsets 可透過 `hermes tools` 互動設定，或 `hermes tools enable NAME` / `hermes tools disable NAME` 管理。工具越多，工具 schema 與選擇空間通常越大；對小任務可用較精簡 toolsets，有助於降低上下文與誤選工具風險。

但貼文中的 skills 指令需要校正：目前 `hermes skills --help` 顯示沒有 `disable` 子命令。正確做法是：

```bash

hermes skills list

hermes skills config

```

用 `hermes skills config` 互動啟用/停用個別 skills；若是 hub-installed skill 且確定不需要，才用：

```bash

hermes skills uninstall <skill-name>

```

工具則可使用：

```bash

hermes tools list

hermes tools disable <toolset>

hermes tools enable <toolset>

```

另外，貼文提到的：

```yaml

tools:

tool_search:

enabled: auto

```

我在目前本機 Hermes v0.14.0 與官方文件中沒有找到這個作為通用工具自動載入設定的明確配置項。更保守的可查核說法是：用 `hermes tools` / `hermes tools enable|disable` 管理 toolsets，或在單次執行時用 `hermes chat --toolsets web,terminal,skills` 明確縮小可用工具。

實務版 10 分鐘清理清單

  • 跑 `hermes model`，進入 `Configure auxiliary models`，把 vision、web_extract、session_search、compression 改到便宜且穩定的模型。
  • 打開 `~/.hermes/SOUL.md`，補上助理人格、語氣、工作原則與禁忌。
  • 檢查 `~/.hermes/memories/` 或 Dashboard / Profile / Memory，刪除錯誤與過期記憶。
  • 跑 `hermes tools list`，停用目前平台不需要的 toolsets。
  • 跑 `hermes skills config`，停用很少使用或容易干擾當前平台的 skills。
  • 未來下 `/goal` 時都加上可驗證完成條件。

保存原因

這篇貼文適合作為 Hermes 新手 onboarding 與成本優化 checklist：它抓到成本、人格、目標、記憶、工具五個真正影響體驗的核心槓桿。需要補上的只是指令精準度與官方語義邊界：auxiliary models 可以省成本但不是保證比例；SOUL.md 是人格不是萬用設定檔；memory 要審核不是亂刪；skills 停用應走 `hermes skills config`，不是不存在的 `hermes skills disable`。
