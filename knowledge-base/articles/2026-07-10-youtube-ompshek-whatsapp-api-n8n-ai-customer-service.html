影片主題

影片圍繞三個實務問題：

  • 使用 WhatsApp API 做 AI 自動回覆是否需要付費。
  • 中小企如何申請與設定 Meta WhatsApp Business Platform / Cloud API。
  • 如何把 WhatsApp API 訊息接到 n8n，後續串 AI Agent、MCP 或 RAG，用公司資料回覆客戶。

影片描述特別強調：AI 自動回覆本身不是 Meta 另收的「AI 費」，重點在 WhatsApp Business Platform 的訊息計費模型。若客戶先發訊息，企業通常可在 24 小時 customer service window 內免費回覆 service messages；若企業主動觸達或超出窗口，就會進入 template messages 與類別計費。

官方文件驗證：收費邏輯

Meta Developers 的 WhatsApp pricing 文件與 Brave Search 摘要交叉驗證後，可整理成以下判讀：

  • 使用者主動傳訊息給企業後，會開啟 customer service window。
  • 在 24 小時 customer service window 內，企業可以用 service messages 回覆，通常不向 Meta 支付 service-message 費用。
  • 企業若要主動發送訊息，或在 24 小時窗口外重新聯絡客戶，通常需要使用 template messages。
  • Template messages 依用途分類，例如 marketing、utility、authentication；類別不同，費率與審核重點也不同。
  • Meta 的免費入口窗口與費率政策會變動，實作前應以 Meta 官方 pricing 頁面與所屬國家/地區費率表為準。

因此，「AI 回覆是否收費」要拆成兩層：

  • AI / n8n / LLM 成本：由你使用的 AI 模型、API 或自架系統決定。
  • WhatsApp 訊息成本：由 Meta WhatsApp Business Platform 的 service window、template 類別與地區費率決定。

官方文件驗證：Template Message

Meta template fundamentals 文件指出，templates 是 WhatsApp Business Account 的資產，用於企業發起或超出服務窗口的訊息情境。實務上可把它想成「事先審核過的標準訊息」。

對 AI 客服流程來說，這有兩個影響：

  • 客戶已經開啟對話時，AI Agent 可以在服務窗口內自然回覆，不一定每句都要 template。
  • 若要做行銷提醒、未付款提醒、身份驗證碼、訂單通知等主動訊息，就要設計對應 template，且注意分類、內容合規與費率。

n8n 串接重點

n8n 官方 WhatsApp Business Cloud credentials / node 文件顯示，串接通常需要：

  • Access Token。
  • WhatsApp Business Account ID。
  • Phone Number ID 或 Cloud API 相關帳號資訊。
  • Meta webhook 回調 URL 與驗證 token。
  • 可公開訪問的 n8n webhook endpoint；本機開發時常需 ngrok / tunnel，正式環境則應用 HTTPS 網域。

影片的 n8n 示範位於 14:28 之後，重點是把 Meta 的 webhook 驗證與訊息事件接到 n8n，後續即可串：

  • AI Agent 節點處理自然語言回覆。
  • 公司 FAQ / 文件 / 商品資料作為 RAG 知識庫。
  • MCP 或其他 API 工具查詢訂單、庫存、預約、CRM。
  • 回覆節點再把結果送回 WhatsApp。

Access Token 與權限風險

Meta Access Tokens 文件與相關官方資料顯示，長效或永久 token 通常會牽涉 Business / System User / App 權限設定。常見權限包含：

  • `whatsapp_business_messaging`：發送與接收 WhatsApp 訊息。
  • `whatsapp_business_management`：管理 WhatsApp Business Account 相關資源。

實務建議：

  • 不要把 token 寫死在前端或公開 repo。
  • 放在 n8n credentials、環境變數或 secret manager。
  • 只給必要權限，並記錄 token 由哪個 Business / System User 產生。
  • 團隊離職、權限調整或疑似外洩時要能快速輪替。
  • n8n workflow 匯出分享前，確認 credentials 沒有一起外洩。

對中小企的導入價值

這類 WhatsApp + n8n + AI Agent 架構的價值，在於把高頻客服轉成可追蹤、可自動化的流程：

  • 售前：回答商品、方案、課程、價格與常見問題。
  • 售中：查詢訂單、預約、付款狀態或交付資訊。
  • 售後：提供操作指引、保固、退換貨與客服分流。
  • 內部：把 WhatsApp 對話事件同步到 CRM、Google Sheets、Notion 或 Airtable。

若再加上 RAG，就能讓 AI 根據公司文件回覆，而不是只靠模型記憶。這特別適合資料分散但問題重複度高的團隊。

導入前檢查表

帳號與電話

  • 準備 Facebook / Meta 帳號與 Business Portfolio。
  • 使用可接收驗證的新電話號碼；避免拿既有私人 WhatsApp 號碼直接測試。
  • 確認該號碼未被其他 WhatsApp / WhatsApp Business App 綁定，或了解遷移限制。

訊息政策

  • 分清楚 service messages 與 template messages。
  • 知道 24 小時 customer service window 何時開始、何時結束。
  • 主動訊息要先設計 template，避免被 Meta 拒審或歸到錯誤類別。
  • 定期回查官方 pricing 頁，因 Meta 計費政策會更新。

技術串接

  • n8n endpoint 必須可被 Meta webhook 連到。
  • 正式環境要使用 HTTPS 與穩定網域，不要長期依賴臨時 tunnel。
  • AI Agent 要有明確 fallback：不知道時轉人工，不要亂編價格或承諾。
  • RAG 文件需版本管理，避免 AI 用過期條款回覆客戶。

安全與治理

  • Token 與 webhook secret 不公開。
  • 對付款、退款、合約、醫療、法律等高風險問題保留人工確認。
  • 設定回覆紀錄與稽核，方便追查錯誤訊息。
  • 對客戶明確揭露可能由 AI 協助回覆。

可複用工作流設計

一個穩健的 WhatsApp AI 客服 n8n workflow 可拆成：

  • Webhook 接收 Meta WhatsApp event。
  • 驗證來源與訊息類型。
  • 取出使用者 phone / message / timestamp。
  • 查 CRM 或資料庫確認客戶狀態。
  • 用 RAG 檢索公司 FAQ / 商品 / 政策文件。
  • AI Agent 產生回覆草稿。
  • 用規則檢查敏感字、價格、承諾與信心分數。
  • 低風險自動回覆；高風險轉人工審核。
  • 寫回對話紀錄與後續任務。

這比「收到訊息就丟給 ChatGPT 回覆」更安全，也更適合商業場景。
