MQTT 與 AIoT 整合運用｜AI 不只要會聊天，還要會上班：Prompt、Context、Harness 三層工程

一句話摘要

AI 要「上班」不能只靠神 prompt；需要 Prompt、Context、Harness 三層一起設計，才能從聊天工具變成可執行、可監控、可治理的企業 AI agent。

貼文將三個層次整理為：

  • Prompt Engineering：把問題問清楚。
  • Context Engineering：給 AI 正確背景資料。
  • Harness Engineering：讓 AI 在安全、可控、可監管的系統中執行任務。

貼文的核心句是：單次對話看 Prompt，企業知識看 Context，真正能上班則需要 Harness。

三層工程怎麼理解

1. Prompt Engineering：任務說明層

Prompt 是使用者與模型互動的第一層，重點是把需求、限制、輸出格式、角色與判準說清楚。它適合處理一次性任務，例如改寫文案、摘要、翻譯、簡單問答。

但 prompt 本身通常不夠處理長流程任務，因為它無法保證資料正確、工具可用、權限安全、錯誤可回復。

2. Context Engineering：資訊架構層

Context Engineering 關注的是「模型在回答或行動前拿到什麼背景」。它包含：

  • 文件與知識庫檢索。
  • RAG 與資料切片。
  • 長期記憶與短期工作記憶。
  • 使用者偏好、任務狀態、工具輸出。
  • 讓模型知道哪些資料可信、哪些過期、哪些不可外流。

在企業場景裡，Context Engineering 決定 AI 能否理解組織知識，而不是只靠模型內建知識猜答案。

3. Harness Engineering：執行環境層

Harness Engineering 是讓 AI 真正能工作的外部系統設計。它不只是提示詞，而是包住模型的一整套「工作環境」，包括：

  • 工具調用與權限邊界。
  • 沙箱、審批、回滾與錯誤恢復。
  • 任務拆解、排程、hooks、subagents。
  • 日誌、監控、可觀測性與成本控管。
  • 評測、驗證、人工覆核與風險治理。

換句話說，Prompt 告訴 AI 要做什麼，Context 給 AI 做事所需的資料，Harness 則決定 AI 能不能安全、穩定、可追蹤地把事做完。

這些資料共同指向同一個趨勢：AI 工程正在從「寫好 prompt」進化到「設計 AI 的資料環境與執行環境」。尤其是 agent 要進入企業工作流時，權限、工具、監控、回復、評估會比單一 prompt 更重要。

為什麼值得 AI 學習雷達收錄

這則貼文適合保存為 AI agent 教學中的核心概念卡，原因有四：

  • 它用非常短的句子區分三種常被混用的工程能力。
  • 它把 AI 的使用場景從聊天轉成「上班」，貼近企業導入語境。
  • 它能連到 Hermes、Claude Code、Codex、MCP、RAG、workflow automation 等實作。
  • 它適合放進課程中作為「從 prompt 到 agent system」的轉場框架。

對欒帥的應用

  • 可用於 AI 課程開場：不要只教 prompt，要教 AI 如何接資料、接工具、接流程。
  • 可整理成企業導入 AI 的成熟度模型：Prompt → Context → Harness。
  • 可對應 Hermes Agent 的能力：記憶與知識庫屬於 Context，工具調用、排程、權限、驗證與部署屬於 Harness。
  • 可用於 AIoT 場景：MQTT 資料流、設備狀態、告警與自動化控制，不能只靠 prompt，必須有明確 context 與 harness 才能安全上線。

注意事項

Harness Engineering 不是一個單一標準工具，而是一組工程實務集合。不同團隊可能用不同名詞描述同樣問題，例如 agent orchestration、runtime、workflow engine、guardrails、observability、evals。教學時應把重點放在「讓 AI 可控地執行任務」而不是名詞本身。
