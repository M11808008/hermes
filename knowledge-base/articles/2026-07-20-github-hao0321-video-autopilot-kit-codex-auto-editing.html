一句話摘要

`video-autopilot-kit` 是一套開源短影音 / YouTube 自動化框架：它不是單純的一鍵剪片 SaaS，而是把創作者工作流拆成可配置問卷、Python/ffmpeg pipeline、CapCut 草稿/GUI 自動化、字幕與 b-roll 對位、交付前 QA gates；適合搭配 Codex / Claude 這類 agent，把「素材 → 初剪 → 字幕 → 補畫面 → QA」變成可迭代的創作系統。

兩條 first-class path

Path 1 — Programmatic（建議欒帥優先看）

  • Win / Mac / Linux 都可走。
  • 需求是 Python 3.9+ 與 `ffmpeg` / `ffprobe`。
  • 不需要 CapCut、不需要 Computer Use。
  • 模組包含 `longform_maker`、`silent_vlog_maker`、字幕、b-roll、screen cleanup、premium motion fx，以及 `delivery_qa` 等交付前檢查。
  • 對欒帥的 macOS 環境與課程示範更穩，因為可重現、可版本控制、可在 agent loop 裡跑。

Path 2 — CapCut-assisted

  • 作者本人主用，但 README 誠實標示 Windows-first、版本敏感。
  • 用 CapCut Desktop 草稿 JSON 直改 + AI assistant / Computer Use 操作 CapCut 視窗套模板與匯出。
  • TROUBLESHOOTING 提醒：剪映 CN 6.0+ 草稿加密不可直改；CapCut 國際版 6.x–9.x 多數仍是明文，但必須先用 `detect_draft_format()` 驗證。
  • Mac 上 CapCut GUI 自動化沒有等效機制，建議走 programmatic path。

值得學起來的設計模式

  • 問卷化 onboarding：`SETUP.md` 先問品牌、niche、製作設定，把工具變成「你的系統」，不是套別人的私有數據。
  • 素材命名即索引：b-roll 用 `coffee-pour.mp4`、`sunset-beach.mp4` 這類內容檔名，就能和 captions 自動對位。
  • 補素材策略：素材不足時用動態字卡 / motion graphics 補 gap，而不是讓影片卡住。
  • QA gates：交付前檢查頻閃、死空檔、字幕同步、螢幕錄影外框、b-roll 比例，比單純「剪完可播放」更接近可交付工作流。
  • Agent-friendly repo：把 README、SETUP、examples、troubleshooting 都寫清楚，讓 Claude / Codex 能接手安裝與改造。

對 AI 學習雷達的意義

這是「AI 內容生產後段自動化」的重要案例。近期 HeyGen / Hyperframes / Remotion 偏向影片生成或程式化渲染，`video-autopilot-kit` 則更接近創作者日常：素材、字幕、b-roll、CapCut、ffmpeg、QA 與平台發布前品質控制。它讓 AI agent 從「寫腳本」往「維護剪輯 pipeline」移動。

Caveats

  • 已有早期歸檔記錄；本篇是針對 Simon Joe / 駱君昊社群貼文與 2026-07-20 GitHub 狀態的更新整理。
  • 貼文說「剪完直接可以用」應理解為 demo / 特定素材條件下的工作流；實務仍需人工審片、授權檢查與品牌品質 gate。
  • 長影片自動剪輯可能耗時很久；公開留言已有 Codex 跑 80 分鐘的案例。
  • CapCut-assisted 路徑高度依賴版本與平台，Mac 使用者不宜把 GUI 自動化當主路線。
  • 商業使用仍要自行處理素材、音樂、字型、肖像與平台政策。
