shooding Threads｜Superpowers 與 Agent Skills：2026 的重點從模型強弱轉向流程品質

一句話摘要

這則貼文抓到一個重要趨勢：AI agent 的勝負不只看模型，而是看你有沒有把「好工程師的做事流程」寫成可重複執行的 skills。

Threads 貼文重點

shooding 的貼文主張：

  • GitHub 上近期最紅的東西，居然是大家的 `.claude/skills` 資料夾。
  • `superpowers` 已經約 25 萬顆星。
  • 它不是模型，而是一套「逼 agent 好好做事」的方法。
  • 流程包含：先問清楚需求、拆規格給使用者簽核、走 TDD，再讓 subagent 分工與互相 review。
  • 有時能讓 agent 自己跑一兩小時仍不偏離方向。
  • 2026 的重點可能已從「哪個模型最強」轉為「你給 agent 的流程有多好」。

為什麼這件事重要

Agent skills 的爆紅，反映的是一個產品與工程現實：

  • 模型越強，越需要流程約束，否則只是更快地亂做。
  • 長任務不只需要 prompt，還需要階段、退出條件、驗收點與回滾策略。
  • TDD、code review、subagent 分工，其實是把人類工程團隊的管理模式移植到 agent。
  • `.claude/skills`、`SKILL.md` 這類文件，把經驗變成可重用的行為規格。

換句話說，skill 不是「讓模型多知道一點」，而是「讓模型用某種工作方式行動」。

可複用方法論

從貼文可整理出一個 agent skill 流程模板：

  • 需求釐清：agent 先問清楚目標、限制、驗收標準。
  • 規格拆解：把需求轉成可審核的小任務。
  • 使用者簽核：在動手前先確認方向。
  • TDD：先寫測試或驗收條件，再實作。
  • Subagent 分工：不同任務可平行處理，但要避免檔案衝突。
  • Review gate：先檢查是否符合規格，再檢查品質、安全與維護性。
  • 收斂與交付：彙整結果、跑測試、列出已知限制。

對 Hermes / Codex 工作流的啟發

這則貼文也可以反向提醒 Hermes 與 Codex 使用者：

  • 只把 skill 當知識庫不夠，skill 應該寫出「何時使用、步驟、驗證、坑、退出條件」。
  • 若 agent 常常違反流程，問題可能不是模型不懂，而是技能觸發條件、強制驗收點、工具權限與上下文載入沒有設計好。
  • 長任務最好拆成可驗收階段，不要只給一句「幫我完成整個功能」。
  • Subagent 有用，但必須搭配 review 與衝突管理，不然只是把混亂平行化。

對 AI 學習雷達的價值

這則貼文值得保存，因為它把 AI agent 的下一個學習重點說清楚：

  • Prompt engineering → Skill engineering
  • Model selection → Workflow design
  • One-shot output → Iterative verified work
  • Single agent → Multi-agent orchestration
  • 靠聰明 → 靠制度

未來真正有價值的 AI 能力，可能是把自己的專業 SOP 寫成 agent 能穩定遵守的技能包。

注意事項

  • GitHub stars 是注意力指標，不等於適合直接導入生產流程。
  • 外部 skill pack 應先檢查權限、安裝腳本、可讀性、維護狀態與是否會執行危險命令。
  • TDD 與 review 能降低偏航，但不能保證安全；仍需人類檢查關鍵變更。
  • `.claude/skills` 概念可移植到 Hermes、Codex、Cursor 等工具，但每個平台的載入機制與觸發規則不同。
